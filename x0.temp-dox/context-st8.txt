THE CONCEPT
=============
A menu whose CONTENTS depend on whatever is currently focused (or, when
opened with no implicit target, whatever the user picks from a list of
open windows) -- generalizing the chrome "&" icon's "one action, scoped to
whatever's focused" pattern into a real per-project, PDL-declared ACTION
LIST, modeled directly on `!.gem-flashlite--yolo/fuzz-op-r&d/fo-menu-sys.md`'s
Fuzz-Op dynamic menu system (see CONTENT MODEL below) -- not a small fixed
hardcoded action set.

First real payload once built: an emoji picker, offered only when a
text-editing field is the resolved focus target (now that emoji rendering
actually works end-to-end -- see the earlier emoji fix this session).


THREE ENTRY POINTS -- each resolves its target differently
==============================================================
This is the piece I had wrong initially -- there isn't one uniform
"resolve active_window() and go" path. There are three distinct ways in,
and only two of them have an implicit target at all:

1. HOTKEY -- Ctrl+Q (locked, see HOTKEY section below for the full
   verification trail). Fires from wherever the user currently is. Target
   resolves immediately via `active_window()`, exactly like window-geom's
   own chrome-icon click: non-NULL -> that window's declared METHODs;
   NULL -> desktop's declared METHODs. No picker step -- the hotkey IS the
   implicit target.

2. NEW CHROME BUTTON -- glyph ':' (locked), 5th icon alongside o/-/x/&,
   added specifically "for accessibility": an explicit, mouse/nav-
   selectable alternative to the hotkey for anyone who can't or doesn't
   want to use it. Same resolution as the hotkey (the window whose chrome
   row you clicked it on), same target-snapshot pattern as '&'/
   open_window_geom_for_project(). Fitting it in is not "just add one
   array entry" -- see CHROME ICON COUNT note under STEPS.

3. LAUNCHER CLICK (opening it cold, like any top-level app -- e.g. from
   the normal project launcher list, exactly how "settings" itself is
   opened) -- THIS is why it's "settings-like" and not just a bigger
   window-geom. There is no implicit target when entered this way: no
   specific window was clicked on, no hotkey fired from inside anything.
   So the FIRST screen it shows is a picker: every currently OPEN project
   ("procs" showing in the taskbar) pulled from desktop_ui_state.txt's
   live desktop_registry_<N>_* fields, plus a "Desktop" entry for
   desktop-scoped actions. The user picks one, and THEN the menu shows
   that target's declared METHODs -- same second screen the hotkey/chrome-
   button paths jump straight to, just reached via an extra selection step
   instead of an implicit resolve.

Entry points 1 and 2 share one resolver function (`active_window()`,
snapshot-at-trigger-time, same as window-geom). Entry point 3 is a
genuinely different first screen (a project picker), not a variant of the
same resolver -- build it as its own explicit step, not a hack bolted onto
active_window() returning some fake "nothing picked yet" sentinel.


HOW IT WILL KNOW THE "FOCUSED" WINDOW (entry points 1 & 2 only)
====================================================================
Reuses `active_window()` (wraith-alpha_manager.c:1165) directly -- the SAME
resolver every other focus-aware function in this file already uses
(resolve_frame_width/height, write_semantic_projection_files,
open_window_geom_for_project's own caller at chrome-icon-click time). It
returns either a real `Window *` or NULL. NULL is not a hypothetical edge
case to newly invent handling for -- it is already a first-class,
well-exercised state in this codebase: build_desktop_shell_markup(NULL) and
write_semantic_projection_files() both already have real "no window
focused" branches (desktop banner + taskbar only, no chrome/panel).

Concretely: `Window *target = active_window();` at trigger time.
  - target != NULL  -> menu resolves that window's project_id and looks up
    ITS declared METHOD entries (see CONTENT MODEL).
  - target == NULL  -> menu resolves the desktop's own declared METHODs
    (a "virtual" target -- see CONTENT MODEL for where these live).

This is a SNAPSHOT at trigger time, not a live binding -- same as
window-geom's own wg_target.txt (written once, in open_window_geom_for_project(),
never updated again while that window-geom instance stays open). If focus
changes after the context menu is already open, the menu does NOT
retarget. Matches existing precedent; not treated as a bug to fix here.

Entry point 3 (launcher click) skips this resolver entirely for its first
screen -- see THREE ENTRY POINTS above.


CONTENT MODEL -- PDL METHOD entries, fo-menu-sys.md pattern
================================================================
Confirmed with the user: NOT a small fixed built-in action list. Each
Wraith project gets its own new `METHOD | <name> | <op-path>` section in
its project.pdl, mirroring fuzz-op's `piece.pdl` exactly:

    METHOD | resize   | projects/wraith-alpha/wraith-projects/settings/window-geom/...
    METHOD | close     | <op path>
    METHOD | emoji     | <op path>              (only listed if focus is a text field --
                                                   filtering logic still TBD, see STILL OPEN)

Lifecycle, ported directly from fo-menu-sys.md section 3 ("The Lifecycle
of an Action") to this codebase's own terms:
  1. Projection: context-menu's manager resolves the target (window's
     project_id, or the desktop's own pseudo-project_id) and calls
     `pdl_reader.+x <target> list_methods` to turn its METHOD section into
     markup, writes gui_state.txt.
  2. Render: chtpm_parser substitutes `${target_methods}`, renders
     `<button>` elements per fo-menu-sys.md pitfall #26 (never raw
     `[2] Feed` text).
  3. Input: user presses a digit or clicks a method button.
  4. Injection: `KEY:N` -> raw key `'0'+N` (fo-menu-sys.md pitfall #80 --
     inject the ASCII digit byte, not the raw integer N).
  5. Resolution: context-menu's manager calls
     `pdl_reader.+x <target> get_method <name>` to get the handler Op path.
  6. Execution: manager executes that handler Op.
  7. Sync: manager refreshes gui_state.txt, calls trigger_render(), pulses
     the marker file -- same as every other project in this codebase
     already does, not a new mechanism.

THE HOTKEY -- LOCKED: Ctrl+Q. Chrome-button glyph LOCKED: ':'
================================================================
Two candidates ruled out before landing here, both worth keeping on
record so they don't get re-proposed blind later:
  - Bare Ctrl+Alt (or bare Shift, bare Alt) -- ruled out. Neither renderer
    can detect a modifier held BY ITSELF: GLUT only reports modifier state
    DURING an actual keyboard/mouse event (glutGetModifiers() is only
    meaningful inside an input callback, not pollable on its own), and a
    raw terminal emits nothing at all for a lone modifier keypress. Any
    usable trigger has to be a real chord (modifier + a real key).
  - Ctrl+; -- ruled out. Terminals encode Ctrl+<key> by masking the byte
    with 0x1F (the same `letter & 0x1f` trick already used for Ctrl+C in
    this codebase). ';' is 0x3B; masked with 0x1F that's 0x1B -- ESC.
    This is the exact same well-known aliasing that makes Ctrl+[ equal to
    ESC in every terminal. ESC is one of the most heavily-relied-on bytes
    in this codebase already (exits map_control mode, AND is the mandatory
    lead byte of every arrow-key/SGR-mouse escape sequence the raw reader
    parses) -- Ctrl+; would either do nothing, get eaten mid-escape-
    sequence-parse, or fire the "leave map_control" behavior by accident.
    Not usable at any letter/punctuation whose `& 0x1F` lands on 27 (also
    rules out Ctrl+[ itself, for the same reason).

Ctrl+Q chosen and verified clean:
  - `'q' & 0x1F` = 0x11 = 17 decimal. Not 27 (ESC), not 13/10 (Enter), not
    8/127 (Backspace), nowhere near WASD's plain-ASCII range (87/119 'W'/'w'
    etc) or the arrow codes (1000-1003).
  - The one real historical risk for Ctrl+Q specifically -- terminal
    XON/XOFF software flow control silently swallowing it before any
    program ever sees the byte -- was checked directly, not assumed away:
    pieces/keyboard/plugins/keyboard_input.c's raw-mode setup explicitly
    disables IXON (`raw.c_iflag &= ~(BRKINT | ICRNL | INPCK | ISTRIP |
    IXON);`, line 103). With IXON off, Ctrl+Q/Ctrl+S are NOT intercepted
    by the tty driver -- byte 17 arrives to this program like any other
    Ctrl+letter combo.
  - Grepped wraith-alpha_manager.c, text_edit_key.c, and wraith_gl.c for
    any existing use of raw key value 17 -- none found. Unclaimed.

Chrome-button glyph ':' -- no technical constraint here at all; icon
glyphs are just a printable character rendered as a button label
(g_chrome_icons[] entries are 'o'/'-'/'x'/'&', all plain printable ASCII,
no collision check needed beyond "not already one of those four"). ':' is
fine. The glyph and the hotkey don't have to match mechanically -- they
just both happen to be picked for the same feature here.

2do-jul12.txt originally claimed Ctrl+<letter> "already works end-to-end
today via the exact mechanism Ctrl+C's shutdown uses in both renderers."
That framing was checked directly against the actual Ctrl+C code in both
renderers and found to overstate things -- the BYTE MECHANISM is
confirmed correct, but "end-to-end" is not, for the reason below (still
relevant: this is what step 2 under STEPS actually has to build).

What's actually confirmed:
- ASCII: pieces/keyboard/plugins/keyboard_input.c puts the terminal in raw
  mode with ISIG explicitly disabled (`raw.c_lflag &= ~(ECHO|ICANON|IEXTEN|ISIG)`,
  line 106) -- meaning Ctrl+<any letter> arrives as a plain byte (1-26),
  NOT as a SIGINT/signal, for every letter, not just C. main()'s loop
  computes Ctrl+C generically as `(('c') & 0x1f)` (line ~237) -- i.e. the
  exact standard control-code formula, confirmed already in use here, and
  directly reusable for Ctrl+Q (`('q' & 0x1f)` = 17, see above).
- GL: wraith_gl.c's keyboard() callback receives Ctrl+C as literal byte
  value 3 via GLUT's own translation (line 505, `if (key == 3)`) -- same
  confirmation, GLUT already hands Ctrl+<letter> combos to this callback
  as their control-code byte, no extra plumbing needed to RECEIVE it.
- Collision safety reconfirmed independently: text_edit_key.c's WASD
  camera keys are matched as literal lowercase/uppercase ASCII ('w'=119,
  'W'=87 etc, text_edit_key.c:131-152) -- completely disjoint from Ctrl+Q's
  byte 17.

CORRECTION (2026-07-13, found while actually implementing this): the
paragraph above (kept for the record, since it was a real intermediate
belief, not deleted to hide a wrong guess) claimed the ASCII/GL
"forwarding path" was new work needed, reasoning from the fact that both
Ctrl+C handlers intercept-and-consume their byte instead of forwarding it.
That reasoning was WRONG on the actual consequence: rereading both loop
bodies line by line (keyboard_input.c's `while(1)` loop, wraith_gl.c's

bytes (keyboard_input.c: just 'c'&0x1f=3; wraith_gl.c: 3, 8, 10, 13, 127).
Byte 17 (Ctrl+Q) is not in either set, so it ALREADY falls through to the
normal forward call in both files (`writeCommand(c)` /
`append_keyboard_event((int)key)`) with ZERO changes needed to either
reader. Confirmed live: wrote `KEY_PRESSED: 17` directly into
pieces/keyboard/history.txt while the manager was running and watched
route_input(17) fire correctly in alpha_manager.log, both with a window
focused and with none -- see DECIDED below for the actual live-test
transcript. The real (and only) new code needed was downstream, inside
route_input() itself, which had nothing that gave byte 17 any meaning at
all before this pass -- that's what got built, not a reader-level
forwarding fix.

DECIDED, 2026-07-13 (both of the below were STILL OPEN as of the previous

  - RESET QUESTION -- DECIDED: no special context-menu-specific focus or
    persistence logic AT ALL. Resize/close/minimize triggered from the

DESKTOP: CHANGE BACKGROUND -- planned, not built (added 2026-07-13)
========================================================================
Two sub-features per direct user instruction: (a) a set of built-in
"auto" patterns to pick from, (b) let the user load their own image as
the background. Checked the ACTUAL current background rendering before
scoping this -- it is NOT a simple menu-wiring task like resize/close;
there is currently ZERO override mechanism for the desktop background at
all, so this is genuinely new rendering infrastructure.

CURRENT STATE (confirmed by reading the code, not assumed):
  - wraith_rgb_daemon.c's parse_frame_meta() hardcodes
    `bg[0]=15; bg[1]=23; bg[2]=32;` (a solid navy) and NEVER reads any
    "FRAME | bg_..." key from current_frame.meta.pdl -- there isn't even
    a parse branch for one. clear_buffer(buffer, bg[0], bg[1], bg[2])
    fills the ENTIRE raster with this single solid color before any
    object draws on top. This is the one and only place "background"
    exists today. No pattern support, no image support, no per-project
    override -- confirmed via full-file grep, not inferred.
  - ASCII has no equivalent concept at all -- its "background" is just
    whatever the terminal itself shows outside drawn characters.

WHAT WOULD NEED TO EXIST (proposed, nothing below is built):
  1. A background CONFIG file, likely
     projects/user/pieces/profiles/guest/desktop/background.txt (same
     guest-profile tree as CREATE FOLDER above -- natural fit, both are
     "things about the user's desktop") declaring mode (solid/pattern/
     image) and a value (color triple / pattern name / image path).
  2. wraith-alpha_manager.c's meta.pdl emission needs new FRAME | bg_mode
     and FRAME | bg_value fields (reading the config file above), the
     same "manager reads config, publishes into meta.pdl, daemon reads
     meta.pdl" shape every other cross-binary value in this doc already
     uses (cell_width_px, frame_width, etc.) -- not a new pattern, the
     SAME one.
  3. wraith_rgb_daemon.c's background fill needs to branch on bg_mode:
     - solid: unchanged, today's clear_buffer() call.
     - pattern (the "auto patterns" case): propose PROCEDURAL generation
       (checkerboard, gradient, dots -- computed directly in C, no asset
       files needed) rather than pre-rendered pattern assets, since
       "auto" implies built-in/zero-setup. A small named-pattern enum +
       one generator function per pattern is enough; this is genuinely
       simple compared to (4) below.
     - image: load and blit a user-supplied image to fill WIDTH x HEIGHT
       before objects draw. REUSE stb_image.h -- already vendored in
       this exact repo at pieces/system/emoji_extract/lib/ for the emoji
       rendering work done earlier this session. Don't add a second copy
       or a different image library; this is exactly the kind of
       duplication the REUSE RULE exists to prevent. Decode once per
       background change (not per frame -- cache the decoded/resized
       pixel buffer, only re-decode when bg_value's image path changes),
       then tile or stretch-to-fit (which of the two -- not decided) into
       the WIDTH x HEIGHT buffer before clear_buffer()'s normal solid-fill
       call is skipped in favor of this blit.
  4. ASCII CANNOT show a pattern or image -- this is a genuine, permanent
     exception to this session's own "GL and ASCII must mirror" rule
     (unlike x/y/width/height geometry, which now correctly mirrors both
     ways). Explicitly flag this as an ACKNOWLEDGED, PERMANENT asymmetry
     when building it -- don't try to force ASCII parity, and don't let a
     future "renderers must agree" live-test treat this one specific gap
     as a bug. Decide (not yet decided) what ASCII shows instead: keep
     its plain default appearance regardless of GL's background, or show
     some crude text indicator (e.g. a single repeated fill character) --
     leaning toward the former (simplest, and ASCII users likely don't
     expect image support from a text terminal anyway) but not decided.
  5. "Load image" needs a way for the user to POINT AT a file -- the
     natural answer, given CREATE FOLDER above, is to browse the SAME
     guest/desktop/ tree (so a user drops an image file into their
     desktop folder via some external means -- exact mechanism for
     getting a file INTO that tree at all is itself unaddressed, e.g. no
     upload/import mechanism exists in this codebase -- then picks it via
     a folder-browsing UI). If a browsing UI gets built for CREATE FOLDER
     first, change-background's image picker should reuse that SAME UI,
     not build a second one -- noted as a design synergy in CREATE
     FOLDER's own section above, repeating here since it's the same
     point from the other direction.

TWO REAL BUGS FOUND AND FIXED WHILE LIVE-TESTING (both confirmed live with
before/after evidence, not theoretical):
  1. derive_repo_root() (context-menu's ops file, copied from settings')
     only recognized an ABSOLUTE project_dir. wraith-alpha_manager.c
     actually passes a RELATIVE one on a project's very FIRST invocation
     (fired by the project's own manager at launch time, hardcoded relative
     PROJECT_DIR macro -- same pattern settings_manager.c uses) and only
     switches to an ABSOLUTE one on SUBSEQUENT KEY:/PROJECT_ACTION:
     dispatches (run_active_project_input_op() -> project_dir_for_window(),
     g_project_root-based). settings' ops never needed repo_root on that
     first call, so this pre-existing inconsistency never surfaced there --
     context-menu's ops DOES need it immediately (to invoke pdl_reader.+x),
     so it silently computed a wrong path and pdl_reader.+x failed quietly
     (stderr to /dev/null), showing "no methods declared" even though
     piececraft-wraith genuinely had METHOD lines. Fixed by recognizing
     both forms.
  2. Stale key replay: context-menu's own session/history.txt is
     append-only and read via "just take the last KEY_PRESSED line," with
     no "already consumed" tracking -- the EXACT footgun settings' own ops
     file documents as a previously-fixed real bug (2fix-july6.txt bug 1).
     Confirmed live: after testing resize once, simply opening context-menu
     AGAIN later (targeting a different window) silently re-ran resize
     again on open, because the old "KEY_PRESSED: 1" line was still the
     last line in history.txt. Fixed by truncating context-menu's own
     session/history.txt every time open_context_menu_for_active_window()
     opens it fresh -- a brand-new open has no meaningful history yet by
     definition.
  3. close_active_window() (used by the chrome 'x' icon) could NOT be
     reused as-is for the context-menu close action: it always operates on

     active), not the original target. Added find_window_by_project_id()
     instead, and route the close DESKTOP_ACTION through that -- same
     underlying close mechanics (WSTATE_CLOSED + compact_registry()), just
     targeting the right window. Not a bug found live, but a real design
     gap caught by careful reading before it would have shipped broken.

Resumed after the checklist above was handed off. User reported close
"staying open, not killed" -- investigated and found close's underlying
mechanics actually correct (confirmed via registry state), but found and
fixed a REAL separate bug while checking: multi-instance ambiguity (see
below). Then built the emoji picker + right-click on top, per direct
user request, and found FIVE MORE real bugs live-testing that. Full list,
all confirmed fixed with before/after evidence:

  1. Multi-instance close ambiguity: find_window_by_project_id() matched
     the FIRST open window with a given project_id -- with two instances
     of the same project open, close could hit the wrong one (would look
     exactly like "the window I'm looking at won't close"). Added
     find_window_by_id() (unique per instance) as the primary resolution
     path; cm_target.txt is now 2 lines (project_id + unique id) instead
     of 1. Verified live: opened two piececraft-wraith instances, closed
     one, confirmed the OTHER stayed open untouched.
  2. Interact-mode emoji filtering built and verified: cm_target.txt
     gained a 3rd line (interact_mode, computed via
     read_project_map_control_for_dir() on the RESOLVED target, checked
     inside open_context_menu_for_active_window() itself so all entry
     points behave consistently, not just right-click). context-menu's
     ops filters METHOD names by an "emoji_" prefix convention: only
     emoji_ methods shown when interact_mode=1, only non-emoji_ methods
     otherwise. Verified live: right-click on wrai-text-editor while in
     INTERACT showed exactly the 10 emoji_ methods, none of resize/close.
  3. text_edit_key.c extended with a new "insert <utf8_text>" mode
     (pieces/system/file_ops/) -- the existing apply_key() only ever
     wrote ONE byte per keystroke, structurally incapable of inserting a
     4-byte emoji. Reused the file/tool per Bible section 11 rather than
     adding a new one. Verified standalone (CLI test) before wiring it
     into anything else.
  4. Markup-line-length bug: publish_state() built ALL button tags as one
     giant concatenated string with embedded <br/> MARKUP tags but no
     real '\n' between them, then wrote it as one fprintf("%s\n", ...)
     call. wraith-alpha_manager.c's project_probe_body_lines() reads
     wraith_body.txt through a 256-BYTE fgets() buffer per physical
     line -- 10 buttons concatenated exceeded that, so the line got split
     mid-tag across multiple reads, and the non-'<'-starting continuation
     chunks got treated as plain text -- reported live as literal
     "<br/>"/"onClick=" text visible in GL. Fixed by giving each button
     its own real '\n' (matches window-geom's own write_wraith_body()
     pattern, which does this too, just never explained why).
  5. Emoji labels showed the METHOD NAME ("emoji_grin") instead of the
     actual emoji glyph -- reported live. Added resolve_display_label(),
     which resolves each method's real handler via get_target_method_handler()
     and extracts the emoji from the "...insert_emoji:<emoji>" handler
     string for display, falling back to the bare method name for
     non-emoji methods (resize/close).
  6. Off-by-one in the new insert_emoji DESKTOP_ACTION prefix match:
     strncmp()/pointer-arithmetic both used 47 for a prefix that's
     actually 46 bytes (`printf '...' | wc -c` confirmed) -- the branch
     silently never matched, so the emoji dispatch queued correctly but
     was never actually picked up by route_command(). No error, no
     crash, just silent no-op -- exactly the kind of bug that "looks
     like it should work" on code review alone. Fixed the literal.
  7. Missing update_state()/trigger_render() on BOTH the Ctrl+Q and
     right-click paths to open_context_menu_for_active_window() -- every
     OTHER path that opens a window (chrome-icon dispatch, taskbar
     activation) calls both right after; these two didn't. Symptom: a
     newly-opened window's geometry/content wouldn't actually reach
     current_frame.objects.pdl until some UNRELATED later action
     happened to trigger a render itself -- confirmed live via a width
     check taken immediately after a right-click showing the OLD active
     window's width, self-correcting only after a later keypress. This
     one is exactly why the width fix (item 8 below) looked broken on
     first test and turned out not to be.
  8. context-menu's own project.pdl given WINDOW | window_width | 48 (half
     of the 96 default) specifically to prove geometry modification works
     end-to-end for a NEW project, not just the ones tested during Phase
     A. Confirmed live, isolated from bug #7's staleness: w=48 in
     current_frame.objects.pdl immediately after either a direct launch
     or a right-click-triggered open.
  9. id_prefix inconsistency: open_context_menu_for_active_window() used
     the literal "context-menu" (hyphen) for launch_window_instance()'s
     id_prefix, while the normal launcher path computes
     sanitize_token("context-menu") = "context_menu" (underscore) for the
     exact same project -- two different window->id styles for what's
     otherwise the identical project. Harmless for correctness (id is
     just an internal lookup key, used consistently) but confusing in
     logs/registries. Fixed to match the launcher's own convention.
 10. BUILD SYSTEM GAP, the deepest one: pieces/system/file_ops/ (text_edit_key.c,
     text_editor_view.c, dir_browse.c, file_copy.c) was NEVER covered by
     compile_all.sh at all -- confirmed via grep, zero references, unlike
