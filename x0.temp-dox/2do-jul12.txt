  confirmed by grep, only the one load call site touches it. A saved
  height change currently has zero visual effect. Real, confirmed gap.

  window-geom/ops/src/wraith_project_input.c` (~l.263-289, key==13

  `project.pdl` using the pipe-delimited `SECTION | KEY | VALUE` format
  `read_pdl_value()` expects. Format looks compatible on inspection,

Design is largely settled from discussion earlier today — this phase
is about BUILDING it, not re-deciding the open questions below, except
where explicitly still marked open.

**Concept:** ONE key (or right-click), working identically across
every Wraith screen (any embedded project, or the desktop itself),
that opens a context menu whose CONTENTS depend on whatever is
currently focused — mirroring how the chrome "&" resize/window-geom
icon already scopes itself to whichever window is focused
(`open_window_geom_for_project(window->project_id)`). An emoji picker
(now that emoji rendering actually works) becomes the first real thing
this menu offers when a text-editing field is focused — not a
special-cased binding of its own.

**DECIDED — implement as designed, don't re-litigate:**

1. Hotkey: **Ctrl+\<letter\>**, not a bare modifier. Investigated in
   detail: bare Shift/Alt/Ctrl+Alt are literally undetectable as a
   standalone press in either the ASCII terminal reader
   (`pieces/keyboard/plugins/keyboard_input.c`) or GLUT
   (`wraith_gl.c`) — modifiers only ever exist as a byte-change or
   flag ALONGSIDE a real key, never alone. Shift+letter is additionally
   ruled out for an editor specifically (indistinguishable from typing
   a capital letter in the ASCII path). Ctrl+letter already works
   end-to-end today via the exact mechanism Ctrl+C's shutdown uses in
   both renderers, and produces bytes 1-26 which can never collide with
   normal typing (32-126) or the arrow codes (1000-1003). Exact letter
   still open — pick one not meaningful in INTERACT-mode's WASD/
   backspace/enter set or the desktop's digit/arrow nav (candidates:
   `G`, `/`).
2. Right-click ALSO opens it. Confirmed cheap: the mouse button number
   is already captured end-to-end in both paths (GL's
   `mouse(int button,...)`, ASCII's SGR mouse parsing in
   `keyboard_input.c`) and both reach `wraith-alpha_manager.c`'s
   `handle_mouse()` (~l.4543, comment already documents SGR codes:
   0=left/1=middle/2=right) — but nothing branches on button value
   today, every button does the identical action. Needs one new
   `if (btn == 2)` (ASCII) / `if (button == GLUT_RIGHT_BUTTON)` (GL)
   branch each side, not new plumbing.
3. Architecture: build it as **a real launched project window**, like
   window-geom itself (`launch_window_instance(...)`, matching
   `open_window_geom_for_project()`'s own pattern) — NOT a floating
   overlay. This codebase's navigation model is page-based
   (`PROJECT_PAGE:`), not z-ordered overlays, so a real window is the
   natural fit, not a fight against the grain.
4. Menu content source: `projects/wraith-alpha/session/
   desktop_ui_state.txt` is ALREADY a live, continuously-updated
   registry of every open window (rewritten by `update_state()` on
   every single state change, not a snapshot) with
   `desktop_registry_<N>_id/title/project_id/instance_no/state/open/
   minimized/visible` per slot — confirmed live content this session.
   The new context-menu project's own ops can read this file directly
   for zero new export work to list "Projects" (open windows) plus a
   "Desktop" entry for desktop-level actions — same idea as
   window-geom's own `wg_target.txt` "which thing am I acting on"
   pattern, reused rather than reinvented.

**STILL OPEN — decide these while implementing, don't guess silently:**

- How the menu renders in ASCII vs GL specifically — likely its own
  small page/overlay state (similar to file_browser's pattern).
- How "what's focused" resolves for a DESKTOP-level (no window
  focused) invocation vs a WINDOW-level one — likely mirrors
  `open_window_geom_for_project()`'s own `active_window()` null-check
  branching, reused not reinvented.
- Whether picker selection inserts an emoji via `apply_editor_key()`'s
  existing key-based insert path, or needs a new insert-by-codepoint
  Op (would belong in `pieces/system/file_ops/` per Bible section 11
  if genuinely reusable beyond this one picker — decide based on
  whether any OTHER planned feature would also want "insert an
  arbitrary codepoint at cursor," not just this one).
- Whether the SAME Ctrl+\<letter\> is the ONE universal binding for
  every context (desktop, any project window) with only the MENU
  CONTENTS varying by focus, or whether the key itself should also
  vary — default to one key, context-sensitive contents (matches the
  user's own "similar to how resize button works" framing) unless a
  concrete reason emerges not to.
- Whether right-click opens the exact SAME menu as the hotkey
  (context-sensitive to focus, not click position) or a click-position
  -specific variant — default to "same menu, same focus-resolution
  logic, one function" per this session's own repeatedly-learned
  lesson about not letting two paths independently compute the same
  thing (see bugs #8, #12, #13 — all the SAME root shape, three times
  in one day).

STEPS, in order:

- Bible section 11 REUSE RULE: check `pieces/system/` for an existing
  shared Op before writing new manager/ops logic from scratch. Also —
  new lesson from today's own loose end above — check for an existing
  FUNCTION of the same purpose already in the file you're editing, not
  just other files, before adding a new one.
- Verify with real evidence, not guesses: this session's real bugs were
  all found by reading two functions side by side and comparing their
  actual criteria/formats, or by reading actual receipt/log data, not
  by re-theorizing from a screenshot or assuming code that "looks
  right" behaves right live.
- Distrust comments that assert two things are "in sync" or "mirrored"
  without a shared helper enforcing it. Prefer one function driving
  both consumers over two independently-maintained copies of the same
  criterion — proven THREE times today (bugs #8, #12, #13) to be the
  single most common recurring bug shape in this codebase specifically.
- One-shot event files (queues) must be cleared by their reader
  immediately after consuming, or they silently re-apply forever.
- `wraith_gl.c` is the one shared GL process for every Wraith project —
  a bug or fix there is repo-wide, not project-specific.
