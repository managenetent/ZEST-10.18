🎯 WHY EMOJI FIRST? (The Strategic Detour)
═══════════════════════════════════════════════════════════════════════════════

Counterintuitively, we're **pausing full window-geom verification** to add emoji
rendering + context menus before diving into overlapping-window debugging. Here's
the reasoning:

  1️⃣  EMOJI RENDERING IS A FOUNDATION
      • Rich text UI (buttons with emoji labels, dialog icons, status glyphs)
      • Requires: FreeType integration, Unicode handling, font caching
      • Blocks: Any feature that wants to display rich text
      • Smart to build now while infrastructure is fresh

  2️⃣  CONTEXT MENUS UNLOCK TESTING
      • Right-click context menu = perfect testing tool for embedded windows
      • Allows per-window interactions without full UI build-out
      • Can open "Edit Window Geometry" from context menu
      • Gives immediate feedback: does resize work? Do windows render correctly?
      • Also tests focus tracking (which window got right-clicked?)

  3️⃣  BOTH SHARE SAME INFRASTRUCTURE
      • Per-project rendering (already have this: manager + ops pattern)
      • Focus/active-window tracking (needed for context menus anyway)
      • Marker-file notifications (same as current state_changed.txt pattern)
      • Once context menus work, we get overlapping-window handling "for free"

  4️⃣  SEQUENCING: EMOJI → CONTEXT MENUS → FOCUS TESTING → WINDOW-GEOM VERIFICATION
      • Emoji: Rich text foundation
      • Context menus: Testing tool + focus-tracking verification
      • Focus/overlap: Proven working via context menu interactions
      • Window-geom: Now testable via context menu, with focus/overlap already verified

TL;DR: We're building the **testing infrastructure** (context menus + focus) while
adding rich-text capability (emoji). This makes the window-geom verification phase
much more reliable (we'll know focus works, overlap works, click-handling works).
