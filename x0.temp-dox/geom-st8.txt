HOW TO USE IT (user-facing)
==============================
1. Open any project window normally (launcher list or taskbar). It becomes
   the ACTIVE window (highlighted title bar).
2. On that window's chrome row (the small icon row next to its title),
   click/nav-select the '&' icon ("chrome_geom", the last chrome icon).
   This is a per-window action, not global -- it targets whichever window
   was active at the moment you clicked it. wraith-alpha_manager.c's
   open_window_geom_for_project() snapshots that window's project_id into
   window-geom's own session/wg_target.txt right then; it does NOT keep
   tracking focus afterward, so switching windows after opening window-geom
   does not retarget it -- you'd need to close and reopen it via '&' again
   on the new target.
3. This opens a NEW window titled "WINDOW GEOM" showing four nudge rows and
   an Apply button:
     [-] X   [+] X
     [-] Y   [+] Y
     [-] W   [+] W
     [-] H   [+] H
     Apply Changes
   Each button is a normal nav-selectable/clickable control (arrow keys +
   Enter, or mouse/GL click) -- same interaction model as everywhere else in
   Wraith, nothing special. Each press nudges an in-memory working value by
   1 (session/edit_state.txt), it does NOT save automatically.
4. Click/select "Apply Changes" to actually persist the four working values
   into the TARGET project's own project.pdl WINDOW section (window_x/
   window_y/window_width/window_height). Nothing changes on screen for the
   target window until this step.
5. Close window-geom and reopen (or just switch to) the target window --
   its title bar, content panel, and (as of this pass) overall frame size
   will reflect the new geometry in both ASCII and GL. No restart of the
   whole manager needed; geometry is (re)loaded fresh every time that
   project's window is (re)launched (launch_window_instance() ->
   load_window_geometry_from_pdl()), not on a timer.

Caveats to tell a user testing this by hand:
- height only visibly does anything in GL right now (see below) -- ASCII
  will not visibly grow/shrink, only its border width will if you change W.
- values aren't clamped to anything sensible on the low end beyond >=0;
  setting width/height very small will visually break the layout (chrome
  icons etc. are still emitted at their old fixed offsets -- see "REMAINING
  KNOWN GAPS" below).
- there's no "reset to default" button; to undo, nav back into window-geom
  targeting the same project and nudge the values back, or hand-edit that
  project's project.pdl WINDOW section directly, or delete the WINDOW
  section from project.pdl entirely to fall back to defaults (0/0/0/0 reads
  as "unset" and each resolver falls back to its historical default).


WHAT WAS FIXED THIS PASS
==========================

1. Stale comments (3 locations) -- all rewritten to describe current reality:
   - wraith-alpha_manager.c ~381-388 (load_window_geometry_from_pdl header)
   - wraith-alpha_manager.c ~1318-1325 (launch_window_instance call site,
     formerly "KPI A1 read-path-only, nothing renders until KPI A2")
   - window-geom_manager.c 23-38 (formerly claimed "READ-ONLY"; Apply/KEY:13
     has written real values since before this pass, comment just never
     caught up)

2. resolve_frame_height() added (wraith-alpha_manager.c, right after
   resolve_frame_width()) -- same precedence/default-preserving structure:
   window->height if >0, else layout_declared_height if >0, else historical
   default 30 (matches wraith_root's long-standing hardcoded h=30, and equals
   desktop_body_top(2) + old desktop_body_height(23) + footer_band_height(4)
   + 1 bottom-border row, so the default reproduces prior behavior exactly).

3. write_semantic_projection_files() (GL/semantic path) now derives ALL of
   its frame-size numbers from resolve_frame_width()/resolve_frame_height()
   instead of hardcoded literals:
     - wraith_root OBJECT: w=<frame_width> h=<frame_height> (was w=96 h=30)
     - desktop_banner OBJECT: w=<frame_width> (was w=96)
     - window_chrome_row / content panel OBJECTs: w=<frame_width-2>
       (was w=94)
     - content panel h=<desktop_body_height-1>, where desktop_body_height
       is now computed as frame_height - desktop_body_top(2) -
       footer_band_height(4) - frame_border_rows(1), instead of the old
       literal 23 (default frame_height=30 reproduces 23 exactly, unchanged)
     - meta.pdl's FRAME | cols/rows/viewport_width_px/viewport_height_px
       now match frame_width/frame_height (10px/18px cell size unchanged)
       instead of hardcoded 96/30/960/540 -- this closes a metadata
       inconsistency that would otherwise have appeared the moment any
       project actually saved a non-default width/height.
   audit.txt and desktop_state.pdl's own separate cols=96/rows=30 literals
   were deliberately NOT touched -- confirmed (again) that nothing reads
   those two particular fields, they're debug-output-only, and touching them
   would have required restructuring variable scope for no functional gain.
   Flagged here as a known, still-stale, but harmless loose end.

4. ASCII path (build_desktop_shell_markup): width was already wired via
   resolve_frame_width() pre-existing; left as-is. Height was deliberately
   NOT given an ASCII equivalent -- there is no existing vertical-extent
   mechanism in ASCII's rendering (no per-row loop analogous to
   append_frame_border()'s per-column dash loop; content just grows to
   however many lines the project body produces, bounded only by an
   unrelated pre-existing max_lines=40 cap). Inventing a fake truncation/
   padding loop just to claim height "works" in ASCII would have been
   dishonest. This is a real, acknowledged asymmetry between the two
   renderers for height specifically (width already had a milder version of
   this same asymmetry -- ASCII's width was always cosmetic-border-only, GL
   now also has real width effect on top of that).
