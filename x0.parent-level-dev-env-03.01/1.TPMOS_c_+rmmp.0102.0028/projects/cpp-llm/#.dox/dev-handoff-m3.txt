### T2: Manager Input Loop (`manager/cpp-llm_manager.c`)
- **Requirement:** Implement the "Live Submission" pattern.
- **Logic:**

    - Detect `ENTER` keypress by monitoring `pieces/keyboard/history.txt`.
    - On `ENTER`, transition `iqabel` to `THINKING` state.
    - Pass the buffer to `cpp-llm_bridge`.
- **Safety:** Ensure `usleep(100000)` throttling is maintained during the wait for AI response.

### T3: Dynamic Response Area (`layouts/cpp-llm.chtpm`)

- **Logic:**
    - The Manager must wrap `cpp-llm_bridge` output into ASCII borders.
    - Populate `${cpp-llm_response_area}` in `state.txt` with formatted lines.
    - Handle overflow (scrolling or truncation at 20 lines).

## 3. iqabel Agent Integration (Phase 3)

### T4: FSM Engine
- Implement the state machine in `iqabel/piece.pdl`:
    - `IDLE`: Waiting for user or ledger event.
    - `THINKING`: Querying AI Bridge.
    - `ACTING`: Parsing JSON from AI and executing a PAL script via `prisc`.
    - `LEARNING`: Writing result to `pieces/memories/` based on `ledger.txt` success.

### T5: JSON Command Parser
- **Requirement:** Extract executable intent from AI prose.
- **Logic:** Look for ` ```json ` blocks in the bridge output.
- **Actions:** Map JSON `action` fields to TPMOS Ops (e.g., `move_piece`, `create_item`).

## 4. Engineering Standards & Constraints
- **Pathing:** NEVER use absolute paths. Resolve root via `argv[1]` (passed by `run.sh`).
- **Persistence:** All AI responses must be logged to `pieces/debug/cpp-llm_history.log`.
- **CPU Safety:** Do not block the manager on the AI query; run the bridge as a child process and poll for completion to keep the UI responsive.
