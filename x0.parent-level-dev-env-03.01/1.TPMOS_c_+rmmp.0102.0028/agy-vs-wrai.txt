## BUG #1 (CRITICAL — this is why typing didn't work): missing
## bare-numeric-key fallback in process_history()

**agy (correct):** its main loop, reading pieces/apps/player_app/history.txt
(the file its layouts' `<interact src="...">` declares), handles THREE
line shapes:
  - "COMMAND: <x>" -> handle_command(x)
  - "KEY_PRESSED: <n>" -> process_key(n)
  - anything else -> `int key = atoi(line); if (key != 0 || line[0]=='0')
    process_key(key);` <- THE FALLBACK

That fallback exists because `chtpm_parser.c`'s `inject_raw_key()`
(called for every keystroke while an INTERACT-mode element is active —
see its own PRIORITY 1 branch, `if (interact_history_path[0] != '\0')`)
writes ONLY the bare numeric key code — `fprintf(fp, "%d\n", code);` —
no "KEY_PRESSED: " prefix at all. Every character you type while editing
in agy arrives in its history file as a bare number, and only that
fallback branch ever sees it.

**wrai (bug):** `process_history()` only checks for `"COMMAND: "` and
`"KEY_PRESSED: "` substrings — no bare-number fallback. Since
`editor.chtpm` declares `<interact src="projects/wraith-alpha/wraith-projects/wrai-text-editor/session/history.txt" />`,
every keystroke typed while INTERACT is active lands in THIS project's
own `session/history.txt` as a bare number too (identical mechanism to
agy) — and gets silently ignored, matching zero branches. This is the
exact, full explanation for "not yet accepting input" from last night's
handoff: the key genuinely never reaches `text_edit_key.+x` at all.

**Fix:** add the same fallback agy already has, calling
`text_edit_key.+x` for editor.chtpm's page instead of `process_key()`.
Applied below.

## BUG #2 (CRITICAL — real data-loss risk): directory browser always
## treats a file click as "load now", even in Save-As mode

**agy (correct):** `update_gui_state()`'s file-listing loop:
```c
if (browser_mode == 0) snprintf(action, ..., "SET_LOAD_FILE:%s", target_file_path);
else                    snprintf(action, ..., "SET_PATH:%s", target_file_path);
```
In LOAD mode, clicking a file loads it immediately. In SAVE-AS mode,
clicking an EXISTING file only fills in the path field (`SET_PATH:`) —
it does NOT load/overwrite the working buffer. This matters: while
picking a save target, browsing past existing files must not destroy
whatever you're currently editing.

**wrai (bug):** `build_directory_browser_markup()` always emits
`PROJECT_ACTION:SET_LOAD_FILE:%s` regardless of `browser_mode`. In
Save-As mode, clicking any existing file in the listing immediately
overwrites the current in-progress document with that file's contents
— the opposite of what Save-As is for, and a real way to silently lose
unsaved work.

**Fix:** branch on `st->browser_mode` exactly like agy does, emitting
`PROJECT_ACTION:SET_PATH:%s` in save mode. Requires also adding a
`SET_PATH:` case to `handle_command()` (wrai never had one — see bug #3
sibling note below, this part is not optional, it's required for the
fix to do anything). Applied below.

## BUG #4 (THE actual root cause of "not accepting input" /
## "arrow key isn't moving the cursor") — write_state() hardcoded
## is_map_control=0 on every write

**This is the exact same bug class already found and fixed for
piececraft-wraith earlier the SAME session** (that one hardcoded `=1`
instead of `=0` — see j11.wraith-foundation-fix-fut.txt's bug 1 entry).
Writing this project's ops from scratch, the identical mistake got
reintroduced in the opposite direction.

`is_map_control` is a SHARED field in `session/state.txt`:
`wraith-alpha_manager.c`'s `set_project_map_control_in_dir()` (fired by
this project's own EDIT TEXT (INTERACT) button, or by ESC) is the real
owner and writes 0 or 1 there. `write_state()` — called at the end of
EVERY invocation of this project's own ops, including the very
invocation that processes the INTERACT command itself (`run_active_
project_input_op()` is called synchronously from inside that same
`route_command()` branch) — was hardcoding `is_map_control=0`
unconditionally. So the sequence was: press Enter on INTERACT ->
`is_map_control` set to 1 -> this project's ops runs synchronously as
part of that SAME action -> `write_state()` immediately clobbers it
back to 0 -> by the time the very next keystroke arrives,
`is_map_control` reads 0 again -> `wraith-alpha_manager.c`'s
`route_input()` never takes its map-control branch -> the key never
gets forwarded to this project as `KEY_PRESSED:` at all. INTERACT mode
was functionally dying after exactly one command, every single time —
which is why arrows didn't move the cursor and typing did nothing, even
after bug #1's fix (there was nothing left for that fix to catch,
because the key was never being forwarded in the first place).

Confirmed live via `alpha_manager.log`/`session/history.txt`: only
`COMMAND: INTERACT` and `COMMAND: SAVE_AS` ever appeared — zero
`KEY_PRESSED:` lines for any of the arrow presses or typed characters
that were attempted in between.

**Fix:** `write_state()` now reads the CURRENT `is_map_control` value
out of the existing `session/state.txt` first (new
`read_current_is_map_control()` helper) and preserves it, instead of
hardcoding the literal — the exact same fix shape already applied to
piececraft-wraith's `save_state()`. Rebuilt clean.

## BUG #5 (ASCII/GL mismatch reported live via screenshot) — stale
## Phase-1 scene.objects.pdl still being rendered in GL

Confirmed live: a screenshot of the GL window showed old Phase-1
test-harness content ("Status: Ready with emoji rendering enabled",
"Font: Noto Color Emoji...", "Phase 1: Emoji Rendering Infrastructure

other actively-maintained Wraith project follows), so this file can
never go stale like this again. Also manually cleared the currently-
stale file on disk immediately rather than waiting for the next
keypress to self-heal it.

## BUG #6 (matches agy's own UX, wasn't wired up) — Enter on the file
## path field didn't trigger Save/Load

**agy:** pressing Enter while focused on the file-path cli_io field
calls `read_file_path_input()` then dispatches `SET_LOAD_ACTION`/
`SET_SAVE_ACTION` depending on mode (`process_key()`'s `key==10||13`
branch, checked against `get_active_gui_index()==2`).

**wrai (bug):** chtpm_parser's own generic cli_io Enter-handling saves
the typed value to `gui_state.txt` (correct, automatic) and then
injects a raw `13` into `session/history.txt` as a bare number — but
`process_history()`'s bare-number fallback only ever called
`apply_editor_key()`, and only for the "editor" page. On the
"file_browser" page, that bare `13` matched nothing and was silently
dropped — the path got saved into `gui_state.txt` correctly (visible on
screen), but nothing ever acted on it. This is why "file save doesn't
seem to save" and why there was never a "Saved to X" confirmation
message (agy's own notification-on-success mechanism was never broken
— it just never got a chance to run, because `handle_command(st,
"BROWSER_ACTION")` was never called).

**Fix:** factored the editor-key and file-browser-Enter handling into
one `dispatch_raw_key()` helper, called uniformly from both the
`KEY_PRESSED:`-prefixed and bare-number branches of
`process_history()`. On the `file_browser` page, Enter (`10`/`13`) now
calls `handle_command(st, "BROWSER_ACTION")` — the same combined
load/save dispatch the LOAD/SAVE button already used, so pressing Enter
in the path field now does the same thing clicking that button does,
matching agy's own Enter-triggers-the-action behavior. Safe even if
Enter came from the search field instead: `BROWSER_ACTION` reads
`file_path_input` itself and reports "Enter a file path first." if it's
empty, mirroring agy's own graceful no-op for an unrelated Enter.

## BUG #7 (shared infrastructure, not project-specific — explains why
## bug #6's fix alone didn't work): Enter-while-typing-in-a-cli_io had
## NO working path to any embedded Wraith project

Confirmed via `debug.txt`: `chtpm_parser.c`'s own raw-key relay
(`inject_raw_key(13)`, fired by its generic cli_io Enter-handling)
writes to `projects/wraith-alpha/session/history.txt` — the DESKTOP
SHELL's own history file (matching the OUTER layout's `<interact>`
declaration) — not the embedded project's own `session/history.txt`.
`chtpm_parser` only recognizes an `<interact>` tag from the layout it's
directly parsing (`current_layout`); a project's own `<interact
src="...">` tag, buried inside markup that only gets SUBSTITUTED into
the outer shell (not separately parsed), is never actually applied.

Separately, `wraith-alpha_manager.c`'s `route_input()` — which DOES
watch the correct per-project history file — has its own Enter-dispatch
branch deliberately gated behind `!g_active_gui_is_typing` (added
earlier the same session specifically so digit/arrow keys can't fight
cli_io typing). That gate caught Enter too, so while typing, Enter fell
all the way to the generic "any other key" branch, which only
re-renders and never forwards anything anywhere. Net effect: the typed
path value was saved into `gui_state.txt` correctly (visible on
screen — this part already worked, from an earlier fix the same
session), but literally nothing downstream ever acted on it, no matter
what the receiving project's own ops tried to do with it — bug #6's
fix had no event to catch.

**Why agy-text-editor was never affected:** it's reached via
`chtpm_parser`'s OWN `current_layout` directly (a standalone project,
not embedded in Wraith), so ITS `<interact>` tag IS the one
`chtpm_parser` recognizes, and its own main loop's bare-number fallback
(bug #1's reference case) correctly catches the raw `13` written to the
file agy actually watches. This bug could only ever manifest for a
Wraith-embedded project — wrai-text-editor is the first one built that
actually needed cli_io Enter-to-submit, which is why it was never
caught before now.

**Fix (in `wraith-alpha_manager.c`, not this project — this is why it
belongs in this doc's "shared infrastructure" category, not a
project-specific bug list):** added a narrow, Enter-only exception to
`route_input()` — when `g_active_gui_is_typing` is true AND the key is
`10`/`13` specifically (never digits/arrows, so the earlier fix that
gate exists for is untouched), forward it to the active project via the
same `append_project_history()` + `run_active_project_input_op()` shape
the `is_map_control` branch already uses. This makes it arrive as a
real `"KEY_PRESSED: 13"` line in the ACTIVE PROJECT's own
`session/history.txt` — exactly the shape `dispatch_raw_key()` (bug
#6's fix) already knows how to handle, and exactly the shape ANY other
Wraith project's own ops can already parse without needing its own
special-case code. Rebuilt `wraith-alpha_manager.c` clean.

## Fourth pass (code-level review against PITFALLS_ACTIVE_2026-03-18.txt,

**Bug #8 (shared infrastructure, `wraith-alpha_manager.c` — not
project-specific, but only exposed by a project whose embedded body has
plain non-interactive `<text label=...>` lines, which wrai-text-editor
is the first to have): nav-slot RESERVATION and nav-slot ASSIGNMENT used
two different criteria, silently drifting apart.**

`count_embedded_body_nav_slots()` (used by `count_project_nav_controls()`
→ both `recompute_nav_bounds()` and `dispatch_menu_index()`) counted ANY
`session/wraith_body.txt` line with a non-empty `label=` attribute as
consuming a nav slot. But `emit_embedded_line_objects()` — the function
that actually HANDS OUT real nav numbers during frame rendering — only
assigns one when `action[0] || has_target_id` is true. Plain `<text
label="..."/>` lines (no `onClick`/`href`, no `target_id`) get counted
into the reservation but never actually receive a nav number.

wrai-text-editor's `wraith_body.txt` has several of exactly these: the
"WRAI TEXT EDITOR" header, the "DIR: ..." line, and — until this pass —
the two blank `<text label="  " />` prefix chunks in front of the
search/path `<cli_io>` fields on the file_browser page. Each one
inflated the reserved count without ever being assigned a real number.

`dispatch_menu_index()` doesn't read nav numbers off the actual rendered
objects for a click — it re-derives `launcher_start`/`taskbar_start` by
ARITHMETIC from this same inflated count
(`launcher_start = CHROME_CONTENT_START + extra_project_slots`, where
`extra_project_slots` comes straight from `count_project_nav_controls()`).
Since the real per-frame numbering (via `emit_embedded_line_objects()`'s
shared `next_nav` counter) runs out sooner than the inflated formula
assumes, the computed segment boundaries drift away from where nav
numbers are actually assigned — exactly the "two independent
computations of the same number must never diverge" invariant
`WRAITH_RGB_ARCHITECTURE.md` calls out. A real, legitimately-assigned nav
number belonging to a project control (file_path_input) can fall on the
wrong side of `dispatch_menu_index()`'s miscalculated boundary and get
interpreted as a taskbar-slot index instead — this is the "keys input...
bleeding thru" the user saw as a jump to a completely different
project's window.

Note the comment that was originally on `count_embedded_body_nav_slots()`
claimed it "mirrors `emit_embedded_line_objects()`'s own chunk-detection
criterion exactly" — true when it was written (2fix.txt, 2026-07-05,
before target_id existed), but the 2026-07-11 target_id generalization
narrowed `emit_embedded_line_objects()`'s real criterion without anyone
updating this counter to match. A stale comment asserting two functions
are in sync is exactly the kind of thing static review catches that a
screenshot can't — the screenshot just shows "wrong window," not why.

**Fix:** made `count_embedded_body_nav_slots()` extract the same
`onClick`/`href`/`target_id` attributes `emit_embedded_line_objects()`
does and use the identical `label[0] && (action[0] || has_target_id)`
condition, so the two can no longer drift apart. Rebuilt
`wraith-alpha_manager.c` clean (only pre-existing, unrelated
`-Wformat-truncation` warnings on path buffers, present before this
change too).

**Bug #9 (wrai-text-editor-specific, `wraith_project_input.c`): GL's
file_browser body wrote blank placeholder text instead of the
descriptive prefixes ASCII uses.** `file_browser.chtpm` (ASCII) labels
the two cli_io fields with `"║ SEARCH: "` / `"║ FILE:   "` prefix text
before each field. The GL dual-write in
`write_gui_state_and_body()` wrote `<text label="  " />` (two blank
spaces) in both spots instead — so in GL, once you'd typed anything, the
field showed only the raw typed value with zero context for what it
was, matching the user's exact complaint. Fixed to write `"SEARCH: "` /
`"FILE: "` instead, matching ASCII. (These same two tags were also two
of the plain-label chunks inflating bug #8's reservation count — fixing
the label text doesn't fix #8 by itself since the tags are still
non-interactive `<text>`, but #8's fix now correctly excludes them from
the count either way.) Rebuilt both `wraith-alpha_manager.c` and
`wraith_project_input.c` clean, and copied the latter's binary from
`ops/src/+x/` to the runtime `ops/+x/` path per
`compile_wraith_project_tree()`'s convention.

**Bug #10 (wrai-text-editor-specific, `wraith_project_input.c`'s
`main()`): `session/state_changed.txt`'s queued `PROJECT_PAGE:`
transition is a one-shot event but was never cleared after being
read, so it silently re-applied on every future invocation for the
rest of the window's life.** Proven from the frame dump itself:
`st->response` was still `"Enter Save-As path."` in the file_menu
frame — unchanged from the file_browser frame — proving
`BROWSER_ACTION` never actually ran (it would have overwritten
`response`). `main()`'s queued-transition block runs BEFORE
`process_history()`, so a stale, never-cleared `PROJECT_PAGE:file_menu`
entry (queued at some earlier point in the window's life, e.g. an
earlier "FILE MENU" click) clobbered `state.active_page` back to
`"file_menu"` before `dispatch_raw_key()`'s `active_page ==
"file_browser"` check for the Enter key even ran — so the Enter-forward
(bug #6/#7's fix) never fired at all. Same bug class as bug #4's
`is_map_control` clobber, different field: a value meant to be
consumed-once and cleared was instead being treated as a durable,
persistent setting. Fixed by truncating `state_changed.txt` immediately
after reading its last queued line. Rebuilt `wraith_project_input.c`
clean, copied to the runtime `ops/+x/` path.

**GL arrow keys "sometimes register twice and skip" — user confirmed
this is NOT project-specific ("its other wraith gl projects 2"), and
static review confirms why: `projects/wraith-alpha/ops/wraith_gl.c` is
the ONLY GL renderer/input file in the entire repo — every embedded
Wraith project's window runs through this one shared GLUT process, so
an input bug here affects all of them identically.** Root cause: no
`glutIgnoreKeyRepeat()` call anywhere in `main()`'s setup. Without it,
X11's auto-repeat mechanism generates synthetic `KeyRelease`+`KeyPress`
pairs for held keys (and, at certain timing boundaries, even for a
quick tap), and freeglut delivers each as a distinct call to
`keyboard()`/`special_keyboard()` — neither of which has any
repeat-suppression or dedup logic of its own (confirmed by reading both
functions: `append_keyboard_event()` unconditionally appends on every
call). ASCII's input path reads keys from the terminal directly, never
touching GLUT/X11 callbacks at all, which is exactly why the user has
"never had this problem in ascii." Fixed by adding
`glutIgnoreKeyRepeat(1)` — GLUT's own standard API for filtering
OS-generated auto-repeat events so only genuine distinct key-down
transitions reach the callbacks — right after the other
`glutXFunc()` registrations in `main()`. Rebuilt `wraith_gl.c` clean
with the GL toolchain (`-lglut -lGLU -lGL -lm`). Needs a fresh live
retest to confirm both the double-register and the skip symptom are
gone — if skip persists independently after this, that's evidence of a
genuinely separate cause and should be treated as a fresh, unconfirmed
report rather than assumed fixed.

## Seventh pass (2026-07-12): GL confirmed fully working, then two more
## shared-infrastructure bugs found — ASCII input entirely dead for
## embedded projects, and a cosmetic GL indicator bug. Both root-caused
## and fixed, not just logged this time.

User confirmed GL's INTERACT/movement now works reliably (the sixth
pass's diagnostic logging never actually got exercised — the underlying
mechanism was fine all along, most likely a genuine one-off timing
fluke, not a bug). Two NEW things surfaced immediately after:
"now its working in gl but not mother-ascii terminal" and, in GL
specifically, "the '>' doesn't go to '^' when enter pressed *but
xelector input still works" (explicitly noted as cosmetic-only).

**Bug #11 (shared infrastructure, `pieces/chtpm/plugins/chtpm_parser.c`
— affects every wraith-alpha-embedded project's ASCII input, not just
wrai-text-editor): `inject_raw_key()` wrote raw key codes in a format
its own reader could never recognize.** Traced the full ASCII INTERACT
path: `chtpm_parser.c`'s own "Active mode" handler for an
`onClick="INTERACT"` element (~l.2749) calls `inject_raw_key(eff)` for
every keystroke while engaged. `resolve_project_history_path()`
(~l.392, with its own 2026-07-11 comment explaining this deliberately)
correctly routes any `wraith-alpha/...`-prefixed project_id — which
includes EVERY embedded project, wrai-text-editor included — to the
SAME shared `projects/wraith-alpha/session/history.txt` file
`wraith-alpha_manager.c`'s `process_history_file()` actually polls.
That routing is correct and intentional. But `inject_raw_key()` wrote
each key as a bare number (`fprintf(fp, "%d\n", code)`), while
`process_history_file()`'s parser requires the literal `"KEY_PRESSED: "`
substring to recognize a key line at all (matching the exact format
GL's `wraith_gl.c::append_keyboard_event()` uses when writing to the
SEPARATE `pieces/keyboard/history.txt` file it polls) — a bare number
matches none of its `strstr()` checks and is silently dropped. This is
why GL worked (correct format, correct file) and ASCII didn't
(correct file, wrong format) for the exact same underlying action.

Fix: `inject_raw_key()` now checks whether the resolved destination is
specifically `projects/wraith-alpha/session/history.txt`
(`history_target_needs_key_pressed_prefix()`, new helper) and, only in
that case, writes with the `"KEY_PRESSED: "` prefix — every OTHER
destination `inject_raw_key()` can write to (a project's own directly-
declared `<interact>` target, or the `pieces/apps/player_app/history.txt`
fallback) keeps the original bare-number format unchanged, since
agy-text-editor's and other legacy consumers' own main loops
intentionally expect that format (confirmed via wrai-text-editor's own
`process_history()` comment referencing "agy-text-editor's own main
loop has always had this exact fallback"). `inject_command()` needed no
change — it already wrote with the correct `"COMMAND: "` prefix.
Rebuilt `chtpm_parser.c` clean (0 errors, 3 pre-existing unrelated
`-Wformat-truncation` warnings in different functions).

**Bug #12 (shared infrastructure, `wraith-alpha_manager.c`, GL-only,
purely cosmetic — confirmed by the user's own report that input kept
working regardless): the nav-selector glyph for an `onClick="INTERACT"`
embedded-body element never checked `is_map_control` at all.**
`emit_embedded_line_objects()`'s glyph decision (the function
responsible for wrai-text-editor's "EDIT TEXT (INTERACT)" button, a
`role=project_body_embedded` object) only ever checked `has_target_id
&& g_active_gui_is_typing` — correct for cli_io fields, but a plain
`onClick="INTERACT"` button has no `target_id`, so this condition could
never be true for it, meaning the glyph could only ever show `">"` or
blank, never `"^"`, regardless of whether INTERACT was actually
engaged. A SEPARATE function (`append_project_scene_objects()`, for
`scene.objects.pdl`-sourced controls) already had the correct check
(`is_map_control && action=="INTERACT"`) — but `emit_embedded_line_objects()`
is a different code path (for `wraith_body.txt`-sourced embedded lines)
that never shared or duplicated that logic at all.

Fix: extracted `append_project_scene_objects()`'s own inline
`is_map_control`-from-`state.txt` read into a new shared helper,
`read_project_map_control_for_dir()` (project_dir-scoped, NOT the same
as the existing global `read_project_map_control()`, which always
resolves the desktop's globally-focused project — a function iterating
embedded content for a specific window must scope this per-window, or
every open INTERACT-capable window would show "^" simultaneously
whenever ANY one of them was actually engaged). Both functions now call
the same helper. `emit_embedded_line_objects()`'s glyph decision now
also shows `"^"` when the selected element's `action=="INTERACT"` and
that window's own project is genuinely in map-control mode. Rebuilt
clean (0 errors, warning count went DOWN by one — removed a duplicate
`snprintf` in the process of extracting the shared helper). Full
`compile_all.sh` sweep confirmed 0 failures repo-wide after both fixes.

**Bug #13 (shared infrastructure, `pieces/chtpm/plugins/chtpm_parser.c`
— affects ASCII rendering for every wraith-alpha-embedded project's
INTERACT indicator, not just wrai-text-editor): `render_element()`'s
`is_active`/`"^"` glyph decision (~l.2207) was `idx == active_index`
only — ASCII's own fully independent, locally-tracked "is this element
active" state, which `active_index` (~l.2603) is ONLY EVER set to by
THIS file's own key-handling code (specifically: Enter pressed on a
locally-focused `onClick="INTERACT"` element). It has zero connection
to `is_map_control` (the actual authoritative shared state
`wraith-alpha_manager.c` owns) — so ANY activation that happens via a
path this file doesn't itself directly witness (a GL mouse click, or
even a keyboard Enter that arrived via a different route than this
file's own normal-mode dispatch) correctly engages the real
`is_map_control` flag (bug #11's fix made that forwarding work) but
never touches THIS file's `active_index`, so ASCII's glyph silently
falls out of sync — identical root shape to bug #12, independently
present in the OTHER renderer because GL and ASCII maintain two
completely separate "is this active" trackers instead of both reading
one shared source of truth.

Fix: added `current_project_is_map_control_active()` (new, mirrors
`wraith-alpha_manager.c`'s `read_project_map_control_for_dir()` — same
file, same key, no shared C function possible across two separate
binaries, but the exact same logic) — resolves the currently-focused
project via `get_var("project_id")` (this file only ever renders one
focused project's embedded body in detail at a time, matching the
desktop's own "Focus: `<window>`" invariant, so a single global lookup is
correctly scoped here, unlike bug #12's GL-side fix which genuinely
needed per-window scoping since GL emits objects for every open window
in one pass) and reads that project's own `session/state.txt`.
`render_element()`'s `is_active` now also becomes true when
`el->onClick == "INTERACT"` and this returns true, independent of
`active_index`. Rebuilt clean (0 errors, 3 pre-existing unrelated
warnings, same as before this change). Full `compile_all.sh` sweep
confirmed 0 failures repo-wide.

**Standing lesson, now proven three times in one day (bugs #8, #12,
#13):** whenever this codebase has TWO renderers (or two functions
within one renderer) independently deciding the same visual fact from
their own local state instead of one shared source of truth, they WILL
drift. Any future "X works in GL but not ASCII" (or vice versa) report
should start by asking exactly this question — grep both files for
independent local trackers of the same concept — before assuming the
underlying functional mechanism (which is usually what actually gets
fixed first, correctly, and which visual indicators then fail to keep
up with) is broken.
