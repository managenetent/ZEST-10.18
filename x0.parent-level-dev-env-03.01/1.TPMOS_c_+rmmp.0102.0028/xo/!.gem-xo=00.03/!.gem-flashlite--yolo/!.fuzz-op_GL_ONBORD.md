# Fuzz-Op-GL Architecture: Sovereign & Genetic Modular Standards

**Date:** April 24, 2026  
**Status:** A24 Genetic Alignment Finalized  

---

## 1. The Core Philosophy: Host vs. Manager

Fuzz-Op-GL follows the **"Sovereign Piece"** pattern, which decouples State (Logic) from View (Rendering).

### A. The Host (The TV) - `gl_desktop.c`
- **Role:** Dumb renderer. It maintains a **Local Cache** of the scene state.
- **Authority:** Only owns the **Free Mode Camera** (for smooth 60fps flight).
- **Communication:** Reads `session/state.txt` (via `gltpm_parser.c`) and writes user inputs to `session/history.txt`.
- **Sync:** In Free Mode, it injects `COMMAND: CAMERA_SET` back to the Manager to keep the "Source of Truth" updated.

### B. The Manager (The Studio) - `fuzz-op-gl_manager.c`
- **Role:** Sovereign State Master.
- **Authority:** Owns all **Entity Positions**, **Map Selection**, and **Locked Camera Modes** (1st/3rd Person).
- **Communication:** Reads `session/history.txt` to process inputs and writes the updated frame state to `session/state.txt`.

---

## 2. Rendering: The "Genetic" Reconstruction

Entities in Fuzz-Op-GL are not hardcoded primitives. They are reconstructed from **Artifacts**.

### A. Artifact Definitions (`artifact.txt`)
- Located in: `projects/fuzz-op-gl/pieces/<id>/artifact.txt`
- Format: 8x8x8 Voxel Bitmask (8 Z-levels, each with 8 hex-encoded rows).
- Purpose: Defines the physical 3D shape of the symbol.

### B. Sovereign Styles (`state.txt`)
- Colors are defined in the piece's state file (e.g., `face_top_rgb=0,255,0`).
- This allows generic rendering: the Host reads the bitmask and the face colors, and reconstructs the entity in 3D without knowing *what* it is.

---

## 3. Data Pipeline & Input Routing

1.  **Input:** Host captures key -> Injects to `history.txt`.
2.  **Logic:** Manager reads `history.txt` -> Updates piece state files (via `piece_manager`) -> Updates `session/state.txt`.
3.  **Pulse:** Manager writes `session/frame_changed.txt`.
4.  **Parse:** Host detects pulse -> Reloads `state.txt` via `gltpm_parser.c` -> Updates its local cache.
5.  **Draw:** Host renders the 3D scene.

## 4. Nuances & Common Pitfalls

### ⚠️ Pitfall: `strtok` Corruption (FIXED)
- **Problem:** Recursive parsing (Map -> Tile -> Artifact) previously used `strtok`, which corrupted the global pointer and caused map rows to disappear.
- **Solution:** **ALWAYS** use `strtok_r` for all string splitting in the Parser.

### ⚠️ Pitfall: Sovereign Drift
- **Problem:** If the Host updates its local `cam_pos` but the Manager doesn't know, the next state pulse will "snap" the camera back to the Manager's stale coordinates.
- **Solution:** The Host must sync back using `COMMAND: CAMERA_SET` in `history.txt`.

### ⚠️ Nuance: Coordinate Systems
- **ASCII Map:** Center is typically `(10, 5)` in a 20x10 grid.
- **GL World:** Map `(0,0)` is usually rendered at GL `(0,0,0)`.
- **Z-Verticality:** Max Z-level is now **10** (previously 2).

## 5. Quick-Start for New Agents

1.  **To add a new entity:**
    - Create `projects/fuzz-op-gl/pieces/<new_id>/`.
    - Add `artifact.txt` (shape) and `state.txt` (position/colors).
    - Register the symbol in `assets/tiles/registry.txt`.
    - Place the symbol in `maps/map_01_zX.txt`.

2.  **To modify rendering:**
    - Edit `pieces/apps/gl_os/plugins/gl_desktop.c`.
    - Recompile using `sh button.sh c`.

3.  **To modify logic:**
    - Edit `projects/fuzz-op-gl/manager/fuzz-op-gl_manager.c`.
    - Recompile using `sh button.sh c`.
