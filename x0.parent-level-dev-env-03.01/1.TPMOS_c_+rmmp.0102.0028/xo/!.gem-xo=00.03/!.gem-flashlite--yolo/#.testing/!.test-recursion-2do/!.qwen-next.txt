# 🤖 Qwen-Code Local Integration: Feasibility & Architectural Blueprint
**Status:** Architectural Planning / Pre-Implementation
**Date:** March 26, 2026
**Philosophy:** "The Bot Writes Its Own Ops"

---

## 1. Executive Summary

This document outlines the plan to integrate a **local Qwen-Code instance** into the TPM-AI FSM ecosystem. The goal is to enable autonomous bot pieces to not only execute PAL scripts but also **generate new ops**, **troubleshoot failures**, and **author new features** recursively.

Currently, the `qwen` command is available system-wide (`/usr/local/bin/qwen`). This blueprint describes how to embed this capability into the testing bot architecture defined in `testing-2dolist.txt` and `user-sim-xm26.txt`.

```c
// Function signatures for ai_bridge.c

int ai_debug_ledger(const char *ledger_excerpt, char *diagnosis, size_t size);
int ai_validate_c_code(const char *code, char *errors, size_t size);
```

## 4. Feasibility Analysis

### 4.1 Technical Feasibility: **HIGH**

**Why it works:**
1.  **File-Based IPC:** TPM already uses files for all communication. AI reads/writes files—perfect match.
2.  **Ops Are Standalone:** `+x` binaries are just executables. AI can write C, compile, and drop them in.
3.  **Prisc Is Text-Based:** PAL scripts are plaintext. AI can generate/modify them easily.
4.  **Qwen CLI Exists:** `/usr/local/bin/qwen` is already installed and functional.

### 4.2 Implementation Complexity: **MEDIUM**

| Challenge | Mitigation |
|-----------|------------|
| **C Code Compilation** | Use `compile_all.sh` as a post-generation step |
| **Prompt Engineering** | Store templates in `codegen_bot/prompts/` |
| **Safety/Auditability** | All AI output goes to `generated/` first, requires human approval |
| **Context Window** | Feed only relevant `ledger.txt` excerpts + `state.txt` |

### 4.3 Risk Assessment

| Risk | Probability | Impact | Mitigation |
|------|-------------|--------|------------|
| AI generates broken C | Medium | Low | Code goes to `generated/` pending compile check |
| Infinite recursion loop | Low | Medium | FSM state machine has `retry_count` limits |
| Prompt injection via ledger | Low | Low | Ledger is internal; sanitize if external input added |
| Slow iteration (LLM latency) | High | Low | Cache common patterns; use AI for scaffolding only |

## 6. Example: AI-Generated Op Workflow

### 6.1 Trigger
Bot needs to inject text input (not just keycodes). No such op exists.

### 6.2 AI Query (via `ai_bridge.c`)
```
PROMPT:
"You are a TPM-compliant C developer. Write a new Op for the Piecemark-IT OS.

Requirement: Inject text string into a piece's input buffer.
Standards:
- Location: pieces/apps/bot_tester/ops/+x/inject_text.+x
- Use fork()/exec() pattern (CPU-safe)
- Append to pieces/keyboard/history.txt with timestamp
- Log to master_ledger.txt

Existing inject_key.+x pattern (for reference):
[... code snippet ...]

Output: Complete .c file ready to compile."
```

### 6.3 AI Output → `generated/ops/inject_text.c`
```c
/* AI-GENERATED Op: inject_text
 * Generated: 2026-03-26T14:32:00Z
 * Status: PENDING_REVIEW
 */

#include <stdio.h>
#include <stdlib.h>
#include <string.h>
#include <time.h>
#include <sys/stat.h>

#define MAX_PATH 4096

int main(int argc, char **argv) {
    if (argc < 2) {
        fprintf(stderr, "Usage: inject_text.+x <text_string>\n");
        return 1;
    }

    // ... [rest of generated code]
}
```

### 6.4 Human Review → Approve
- Move `generated/ops/inject_text.c` → `pieces/apps/bot_tester/ops/src/`
- Run `./compile_all.sh`
- New op available at `pieces/apps/bot_tester/ops/+x/inject_text.+x`

### 6.5 Bot Uses New Op
```asm
// nav_bot/scripts/signup.asm
call pieces/apps/bot_tester/ops/+x/inject_text.+x "bot_baby"
sleep 500000
call pieces/apps/bot_tester/ops/+x/inject_text.+x "KISS_123"
```

## 7. Prompt Templates (Stored in `codegen_bot/prompts/`)

### 7.1 `op_template.txt`
```
You are a TPM-compliant C developer for the Piecemark-IT OS.

Write a new Op (standalone C binary) with these requirements:
- Op Name: {{OP_NAME}}
- Purpose: {{OP_PURPOSE}}
- Input Args: {{OP_ARGS}}

TPM Standards:
1. Use fork()/exec()/waitpid() pattern (no system() unless interactive)
2. Signal handling: SIGINT handler with _exit()
3. File paths: Resolve via PRISC_PROJECT_ROOT env or default paths
4. Logging: Append to master_ledger.txt on success/failure
5. Determinism: No random() unless seeded from state.txt

Reference existing ops in: pieces/apps/playrm/ops/src/

Output: Complete .c file. No markdown, no explanations.
```

### 7.2 `debug_template.txt`
```
You are a TPM debugging assistant.

Analyze this ledger excerpt from a failed bot run:
{{LEDGER_EXCERPT}}

Bot State at Failure:
{{STATE_TXT}}

Task:
1. Identify the root cause
2. Propose a fix (PAL script patch or op modification)
3. If code change needed, output the corrected code block

Format:
ROOT_CAUSE: <one sentence>
FIX: <actionable step>
CODE: <if applicable>
```

### 7.3 `script_template.txt`
```
You are a PAL scripting expert for the Piecemark-IT OS.

Goal: {{GOAL_DESCRIPTION}}

Available Ops:
- inject_key.+x <code>
- inject_text.+x <string>
- assert_state.+x <piece_path> <key> <expected>
- wait_for_frame.+x <string>
- send_command.+x <command>

Constraints:
- Use sleep 2000000 (2s) between major actions
- Always log success/failure to master_ledger

Output: Complete .asm file. No markdown.
```

## 8. Safety & Auditability

### 8.1 Human-in-the-Loop Gates
| AI Action | Auto-Execute | Requires Approval |
|-----------|--------------|-------------------|
| Read ledger/state | ✅ | ❌ |
| Generate PAL script | ✅ (to `generated/`) | ❌ |
| Generate C code | ❌ | ✅ (before compile) |
| Execute op | ✅ | ❌ |
| Modify existing files | ❌ | ✅ (all changes) |
| Spawn new bot piece | ❌ | ✅ |

### 8.2 Audit Trail
All AI actions logged to `master_ledger.txt`:
```
[TIMESTAMP] AI_ACTION: script_generated | bot=nav_bot | output=generated/scripts/nav_v2.asm
[TIMESTAMP] AI_ACTION: code_generated | op=inject_text | status=PENDING_REVIEW
[TIMESTAMP] AI_ACTION: debug_complete | root_cause=timeout | fix=sleep_increased
```

### 8.3 Containment
- AI runs in user context (no root)
- File writes restricted to `generated/` subtree
- Compilation requires explicit `compile_all.sh` invocation
- No network access (local Qwen only)

## 9. Testing Strategy

### 9.1 Unit Tests (AI Bridge)
```c
// Test: ai_query returns valid output
test_ai_query_basic() {
    char output[4096];
    int result = ai_query("What is 2+2?", output, sizeof(output));
    assert(result == 0);
    assert(strstr(output, "4") != NULL);
}

// Test: ai_generate_op produces compilable C
test_ai_generate_op_compiles() {
    int result = ai_generate_op("inject_text", "generated/test_op.c");
    assert(result == 0);
    assert(system("gcc -o /tmp/test_op generated/test_op.c") == 0);
}
```

### 9.2 Integration Tests (Full Loop)
1.  **Script Gen Test:** "Navigate to User menu" → AI generates `.asm` → Execute via `prisc+x` → Verify `current_layout.txt`
2.  **Debug Test:** Break `nav_to_user.asm` → Run → AI analyzes failure → AI patches → Re-run → Success
3.  **Op Gen Test:** Request new op → AI writes C → Human approves → Compile → Bot uses new op
