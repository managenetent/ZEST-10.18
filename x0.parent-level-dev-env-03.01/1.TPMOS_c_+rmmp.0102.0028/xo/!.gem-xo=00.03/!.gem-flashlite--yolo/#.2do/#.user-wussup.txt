
     1 # Fuzz-Op Rendering & Navigation Glitches - Current State & Theory
       (Handoff)
     2
     3 ## Summary of Problem
     4
     5 The Fuzz-Op project is not loading correctly. It continues to display the
       generic `playrm.chtpm` layout instead of its intended `fuzz-op.chtpm`
       layout. Consequently, Fuzz-Op specific pet statistics (name, hunger,
       etc.) and dynamic action methods are missing from the UI. Additionally,
       the Escape key functionality for exiting map activation is not working as
       expected in Fuzz-Op, though it functions correctly in other projects like
       Op-Ed.
     6
     7 ## Observed Behavior
     8
     9 **Current Fuzz-Op UI:**
    10 *   **Layout:** `LAYOUT ID: playrm.chtpm` is displayed.
    11 *   **Content:** Pet Name, Status, Turn, Time, Hunger, Happiness, Energy,
       Level are all blank or use generic placeholders.
    12 *   **Game Map:** Displays a generic map.
    13 *   **Methods:** Action buttons (scan, feed, etc.) are missing.
    14 *   **ESC Key:** Does not exit map activation state.
    15
    16 **Expected Fuzz-Op UI:**
    17 *   **Layout:** `LAYOUT ID: fuzz-op.chtpm` should be displayed.
    18 *   **Content:** Fuzz-Op specific pet details (e.g., "Pet Name:
       Fuzzball", Hunger: 50/100) should be visible.
    19 *   **Methods:** Dynamic action buttons (e.g., `[2] scan`, `[3] feed`)
       should be populated and clickable.
    20 *   **ESC Key:** Should function to exit map activation and return to the
       previous menu/selector.

    28 ## Theorized Root Causes
    29
    30 Despite previous interventions, the core issues persist, suggesting a
       deeper problem in state management or UI component interaction for
       Fuzz-Op:
    31
    32 ### 1. Persistent Incorrect Layout Loading (`playrm.chtpm`)
    33
    34 *   **Problem:** `playrm.chtpm` continues to load even though
       `pieces/display/layout_changed.txt` correctly points to
       `projects/fuzz-op/layouts/fuzz-op.chtpm`, and `chtpm_parser.c` is updated
       to recognize `fuzz-op` as modern.
    35 *   **Hypotheses:**
    36     *   **State Incompleteness Triggering Fallback:** The parser might
       have a fallback mechanism that defaults to `playrm.chtpm` if critical
       Fuzz-Op specific state (`active_target_id`, pet stats) is missing or not
       correctly retrieved *after* `load_vars()` runs. Even if
       `is_modern_layout` is true, the subsequent state loading might fail,
       leading to a generic layout.
    37     *   **`active_target_id` Not Persisted/Used:** The initialization of
       `active_target_id` in `fuzz-op_manager.c` might not be persisting
       correctly, or `get_state_fast()` calls within the manager are failing to
       retrieve Fuzz-Op entity data. If `active_target_id` remains "xlector" or
       unset for Fuzz-Op, the manager cannot populate the necessary state.
    38     *   **`resolve_project_op.c` Interaction:** While
       `resolve_project_op.c` seems to correctly identify the layout path, there
       might be an interaction issue where this path is not correctly passed or
       used by the parser, or it's being overridden.
    39
    40 ### 2. Missing Fuzz-Op Content (Pet Stats & Methods)
    41
    42 *   **Problem:** Pet-specific data (`pet_name`, `hunger`, etc.) and
       action methods (`scan`, `feed`) are not displayed.
    43 *   **Hypotheses:**
    44     *   **`active_target_id` Issue:** This is the most probable cause. If
       `active_target_id` is not correctly set to a Fuzz-Op entity (like
       `"pet_01"`) and persisted, `get_state_fast(active_target_id, ...)` will
       return "unknown" or default values for pet stats.
    45     *   **Method Generation Failure:** The `${piece_methods}` placeholder
       relies on `active_target_id` and `get_piece_methods_op`. If
       `active_target_id` is incorrect, the manager cannot fetch or pass the
       correct methods to the parser.
    46     *   **Manager State Write Failure:** `fuzz-op_manager.c`'s
       `write_gui_state()` might not be correctly fetching or writing the
       Fuzz-Op specific state variables (even if `active_target_id` were
       correct).
    47
    48 ### 3. Broken ESC Key Functionality
    49
    50 *   **Problem:** The Escape key does not exit map activation state in
       Fuzz-Op.
    51 *   **Hypothesis:** This is likely a symptom of the underlying state and
       focus management issues. If the parser or manager doesn't correctly
       signal Fuzz-Op's active state (`active_index`, `active_target_id`), the
       ESC key handler (which relies on this context) may not know how to revert
       to the previous state.
    52
    53 ## Architectural Context (`fuzz-op-r&d/fo-menu-sys.md` & Core Principles)
    54
    55 *   **Dynamic Menus:** The system relies on a dynamic menu system where
       the UI (`.chtpm` layout) contains placeholders like `${piece_methods}`.
    56 *   **PDL Driven:** Piece capabilities (`METHOD` entries in `piece.pdl`)
       are the source of truth for actions.
    57 *   **Manager as Dispatcher:** The manager (`fuzz-op_manager.c`) is
       responsible for setting the `active_target_id`, fetching methods via PDL,
       and executing actions.
    58 *   **State Synchronization:** All relevant state (`project_id`,
       `active_target_id`, pet stats, methods) must be correctly written to
       state files (`gui_state.txt`, `player_app/manager/state.txt`) and
       persisted.
    59 *   **Dumb Theater:** The CHTPM parser (`chtpm_parser.c`) acts as a "Dumb
       Theater," rendering what the state files provide. It should not contain
       app-specific logic.
