       1 **Subject: Adapting Dynamic Method Population from fuzz-op PDL to MP3 List for mp3-store**
       2
       3 **Findings on fuzz-op's PDL Method Population:**
       4
       5 In fuzz-op and the broader TPMOS ecosystem, the `piece.pdl` file acts as the "DNA" for a piece, defining its state,
         metadata, and available actions or "methods". The `METHOD` entries within the PDL map specific events or user
         interactions to handler paths, which are typically executable Op binaries (`.+x`) or Prisc scripts (`.asm`). For
         example, a `METHOD` entry might link the "feed" event to an Op that handles feeding a pet.
       6
       7 The module code (e.g., `playrm_module.c`) uses a function like `get_method()` to dynamically query the `piece.pdl`
         for the appropriate handler based on a given event. This mechanism allows the UI (via CHTPM layouts, potentially
         using `${piece_methods}`) to present these available actions to the user. When an action is selected, the module
         retrieves the handler path using `get_method()` and executes the associated Op or script.
       8
       9 **Adapting for MP3 Store:**
      10
      11 The principle of dynamic population can be applied to the `mp3-store` project. Instead of reading methods from
         `piece.pdl`, the `mp3-store` will use a list of MP3 files.
      12
      13 1.  **Data Source:** The primary data source will be a list of MP3 files. This can be populated by:
      14     *   Scanning a designated directory (e.g., `projects/mp3-store/mp3s/`).
      15     *   Reading from a configuration file (e.g., `mp3_list.txt`).
      16
      17 2.  **"Methods" as MP3 Entries:** Each MP3 file represents an item for user interaction. Key information includes:
      18     *   Filename (e.g., `song_01.mp3`) for playback.
      19     *   Displayable Title (e.g., "Song Title") for UI presentation.
      20     *   Optional Metadata (Artist, Album) for richer display.
      21
      22 3.  **UI Presentation:** The `mp3-store` manager module will:
      23     *   Scan the MP3 source to get the list of tracks.
      24     *   Dynamically generate CHTPM UI elements (e.g., `<button>`) for each MP3, using the MP3 title as the button
         label.
      25     *   The `onClick` attribute of these buttons will invoke a common Op (e.g., `play_mp3.+x`), passing the
         specific MP3 file path as an argument.
      26
      27 4.  **Execution Logic:**
      28     *   A single Op (`play_mp3.+x`) will handle playback.
      29     *   When a user selects an MP3, the manager will construct and execute the command to call `play_mp3.+x` with
         the MP3 file path. Example: `./projects/mp3-store/ops/+x/play_mp3.+x "mp3s/song_01.mp3"`.
      30
      31 **Key Adaptation Points:**
      32 - Data source shifts from PDL to filesystem scan/manifest file.
      33 - "Methods" are MP3 titles; handlers are calls to a generic `play_mp3` Op with MP3 path arguments.
      34
      35 This approach maintains the TPM principle of data-driven, dynamic interaction, allowing the `mp3-store` to adapt
         its available options based on external data.
