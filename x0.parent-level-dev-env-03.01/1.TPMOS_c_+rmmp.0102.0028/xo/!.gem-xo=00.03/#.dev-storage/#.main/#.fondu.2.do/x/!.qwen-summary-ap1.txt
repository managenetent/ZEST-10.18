## PART 2: FONDU SPECIFICATION CLARIFIED

### 2.1 Vision Refinement

**Original Vision:** "Melt the boundary between Project Source and Installed App"

**Clarified Vision:** Fondu is a **build-and-deploy pipeline** that transforms editable Project source code into production-ready, non-editable App installations.

**Key Insight:** Fondu should be **Option A first** (simple copy/compile tool), then evolve toward **Option B** (full package manager) once the pattern is proven.

### 2.2 Lifecycle Stages (Clarified)

```
┌─────────────────────────────────────────────────────────────────────────────┐
│ PROJECT LIFECYCLE                                                           │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [ACTIVE]                    [ARCHIVED]              [INSTALLED]            │
│  projects/<id>/       →      projects/trunk/   →     pieces/apps/installed/ │
│  - Editable source             - Source only           - Non-editable       │
│  - In dev builds               - Excluded from         - Binaries + assets  │
│  - Daily iteration               standard compile      - Production-ready   │
│  - Developer-facing              - Read-only ref       - User-facing        │
│                                                                             │
│  Fondu Installer triggers:                                                  │
│  1. Developer marks project "ready for production"                          │
│  2. Fondu pulls from projects/                                              │
│  3. Compiles binaries                                                       │
│  4. Deploys to pieces/apps/installed/                                       │
│  5. Registers in app_list.txt                                               │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 2.3 When to Use: APP vs. PROJECT vs. FONDU-INSTALLED

```
┌─────────────────────────────────────────────────────────────────────────────┐
│ DECISION TREE: What should I build?                                         │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│ Q1: Is this SYSTEM INFRASTRUCTURE (parser, renderer, orchestrator)?        │
│     └─> Place in: pieces/system/ or pieces/chtpm/                          │
│                                                                             │
│ Q2: Is this a DEVELOPMENT PROJECT (game, simulation, user content)?        │
│     ├─> YES: Create in projects/<project_id>/                              │
│     │         - Use project.pdl with entry_layout                          │
│     │         - Implement <project>_manager.c                              │
│     │         - Launch via loader menu                                     │
│     │         - Can later deploy via Fondu                                 │
│     └─> NO: Continue...                                                    │
│                                                                             │
│ Q3: Is this a SYSTEM TOOL (editor, manual, utility)?                       │
│     ├─> YES: Create in pieces/apps/<app_id>/                               │
│     │         - Use <app_id>.pdl with layout_path                          │
│     │         - Implement <app>_module.c                                   │
│     │         - Launch via button href                                     │
│     │         - Always system-managed, never trunked                       │
│     └─> NO: Reconsider design                                              │
│                                                                             │
│ Q4: Should this project be deployable as a "production app"?               │
│     └─> YES: Use Fondu Installer after development complete                │
│     └─> NO: Keep as projects/ only (dev/internal use)                      │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 2.4 Fondu Installer: Phased Implementation

#### PHASE 1: Minimal Viable Fondu (Recommended Starting Point)

**Scope:** Simple build-and-copy tool with basic registry updates.

```
fondu.c Responsibilities:
├── 1. Source Acquisition
│   ├── Validate source directory (projects/<id>/)
│   ├── Copy directory structure to temp build dir
│   └── Detect build script (build.sh, Makefile, or use template)
│
├── 2. Compilation
│   ├── Execute build script in temp dir
│   ├── Compile all .c files to binaries
│   └── Place in +x/ subdirectories
│
├── 3. Deployment
│   ├── Create pieces/apps/installed/<app_id>/
│   ├── Move compiled binaries from temp dir
│   ├── Copy required assets (.chtpm layouts, .pdl files)
│   └── Set non-editable permissions (chmod 555)
│
└── 4. Registry
    ├── Append app_id to pieces/os/app_list.txt
    └── Add route to pieces/os/project_routes.kvp
```

**Phase 1 Limitations (Intentional):**
- No remote trunk support (local `projects/` only)
- No dependency resolution (flat builds)
- No versioning (latest only)
- No rollback capability

#### PHASE 2: Enhanced Fondu (Future)

**Scope:** Add package management features once Phase 1 is proven.

- Remote trunk mounting (external project sources)
- Version tracking and rollback
- Dependency resolution via `.pdl` manifests
- Uninstall capability
- App store UI integration

## PART 4: TECHNICAL NOTES

### 4.1 Fondu.c Pseudocode

```c
/*
 * fondu.c - TPMOS App Installer (Phase 1)
 * Usage: ./fondu <project_id>
 */

#include <stdio.h>
#include <stdlib.h>
#include <string.h>
#include <unistd.h>
#include <sys/stat.h>
#include <sys/wait.h>

#define SOURCE_BASE "projects/"
#define INSTALL_BASE "pieces/apps/installed/"
#define BUILD_DIR "/tmp/fondu_build/"

int main(int argc, char *argv[]) {
    if (argc != 2) {
        fprintf(stderr, "Usage: %s <project_id>\n", argv[0]);
        return 1;
    }

    char *project_id = argv[1];
    char source_path[512];
    char install_path[512];
    char build_path[512];

    // 1. Validate source exists
    snprintf(source_path, sizeof(source_path), "%s%s", SOURCE_BASE, project_id);
    if (access(source_path, F_OK) != 0) {
        fprintf(stderr, "Error: Project '%s' not found in %s\n", project_id, source_path);
        return 1;
    }

    // 2. Create build directory
    snprintf(build_path, sizeof(build_path), "%s%s", BUILD_DIR, project_id);
    mkdir_recursive(build_path);

    // 3. Copy source to build directory
    copy_directory(source_path, build_path);

    // 4. Detect and run build script
    if (file_exists("%s/Makefile", build_path)) {
        run_command("make -C %s", build_path);
    } else if (file_exists("%s/build.sh", build_path)) {
        run_command("%s/build.sh", build_path);
    } else {
        // Use standard TPMOS compile template
        compile_all_c_files(build_path);
    }

    // 5. Create installation directory
    snprintf(install_path, sizeof(install_path), "%s%s", INSTALL_BASE, project_id);
    mkdir_recursive(install_path);

    // 6. Deploy binaries and assets
    deploy_binaries(build_path, install_path);
    deploy_assets(build_path, install_path);

    // 7. Set non-editable permissions
    chmod_recursive(install_path, 0555);

    // 8. Update registry
    append_to_file("pieces/os/app_list.txt", project_id);
    add_route("pieces/os/project_routes.kvp", project_id, install_path);

    printf("✓ Installed: %s\n", install_path);
    return 0;
}
```

## PART 5: OPEN QUESTIONS

### 5.1 Decisions Needed

1. **Should Fondu support uninstall?**
   - Phase 1: No (manual removal only)
   - Phase 2: Add `./fondu --uninstall <app_id>`

2. **Should installed apps be editable?**
   - Recommendation: No (chmod 555, read-only)
   - Exception: `pieces/` subdirectory for runtime state

3. **Should Fondu validate dependencies?**
   - Phase 1: No (assume self-contained)
   - Phase 2: Parse `.pdl` for dependency manifests

4. **Where should Fondu binary live?**
   - Option A: `#.tools/fondu.+x` (system tool)
   - Option B: `pieces/system/fondu/+x/fondu.+x` (core system)
   - **Recommendation: Option A** (not part of runtime)

### 5.2 Future Considerations

- **Remote Trunk:** Mount external `projects/` from network/Git
- **Versioning:** Track app versions, support rollback
- **App Store UI:** Browser interface for installing/uninstalling
- **Dependency Graph:** Resolve and install dependent projects
- **Sandboxing:** Run installed apps in restricted environment
