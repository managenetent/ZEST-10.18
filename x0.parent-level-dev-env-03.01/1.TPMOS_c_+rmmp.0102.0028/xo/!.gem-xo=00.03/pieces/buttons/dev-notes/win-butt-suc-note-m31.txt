### Issue 1: MSYS2 Path Hardcoded
**File:** `#.buttons/windows/run.ps1`
**Line:** `& C:\msys64\usr\bin\bash.exe -lc "..."`

**Problem:** Assumes MSYS2 is installed at `C:\msys64`. If user installed
elsewhere (e.g., `D:\msys64` or `C:\Program Files\msys64`), this fails.

**Fix:**
```powershell
# Option A: Detect MSYS2 installation
$MSYS2_PATHS = @(
    "C:\msys64\usr\bin\bash.exe",
    "C:\Program Files\msys64\usr\bin\bash.exe",
    "D:\msys64\usr\bin\bash.exe"
)
$BASH = $MSYS2_PATHS | Where-Object { Test-Path $_ } | Select-Object -First 1
if (-not $BASH) {
    Write-Error "MSYS2 bash.exe not found. Please install MSYS2 or update path in run.ps1"
    exit 1
}
& $BASH -lc "..."

# Option B: Use environment variable
$BASH = $env:MSYS2_BASH ?? "C:\msys64\usr\bin\bash.exe"
```

**File:** `pieces/chtpm/plugins/orchestrator.c`

**Fix Direction:**

- Ensure writeCommand() writes identical format on both platforms
- Verify chtpm_parser reads history correctly on Windows

### Issue 4: Frame History Shows Only 1 Frame (UNRESOLVED)
**File:** `pieces/chtpm/plugins/orchestrator.c` - render_thread_func()

**Symptom:** Output shows single frame, not scrolling history.

**Expected (per #.docs/#.piece.mark.c-look_0.txt):**
```
--- FRAME UPDATE at 2026-03-04 06:26:17 ---
[frame 1]

--- FRAME UPDATE at 2026-03-04 06:26:19 ---
[frame 2]
```

**Problem:** Windows render thread clears screen every update instead of
appending frames when history mode is "on".

**Fix Direction:**
- Modify render_thread_func() to check `pieces/display/state.txt`
- If history mode is "on", don't clear screen - just print new frame
- Add separator lines between frames

---

### Issue 5: Ctrl+C Doesn't Kill GL Window (UNRESOLVED)
**File:** `pieces/chtpm/plugins/orchestrator.c` - handle_sigint()

**Symptom:** Pressing Ctrl+C kills orchestrator but GL window remains open.

**Possible Causes:**
1. handle_sigint() cleanup not working on Windows
2. GL renderer not registered with proc_manager
3. `system("powershell.exe -File ./kill_all.ps1")` not executing properly
4. GL renderer ignoring SIGTERM

**Fix Direction:**
```c
// In handle_sigint(), add explicit GL renderer kill
#ifdef _WIN32
    system("taskkill /F /IM gl_renderer.+x 2>nul");
    system("powershell.exe -File ./kill_all.ps1 >$null 2>&1");
#else
    system("pkill -9 -f gl_renderer");
    system("./kill_all.sh >/dev/null 2>&1");
#endif
```

## QUICK FIX GUIDE
================================================================================

### If Windows Buttons Don't Work At All

**Step 1: Verify File Structure**
```powershell
Test-Path "button.ps1"
Test-Path "#.buttons\windows\compile.ps1"
Test-Path "#.buttons\windows\legacy\compile_all.ps1"
```

**Step 2: Test Legacy Scripts Directly**
```powershell
& "#.buttons\windows\legacy\compile_all.ps1"
& "#.buttons\windows\legacy\kill_all.ps1"
```

**Step 3: Check PowerShell Errors**
```powershell
$ErrorActionPreference = "Continue"
.\button.ps1 compile 2>&1 | Out-String
```

### If Keyboard Input Not Working

**Step 1: Check keyboard history**
```powershell
Get-Content pieces/keyboard/history.txt -Wait
# Press arrow keys - should see entries like:
# [2026-03-31 14:00:00] KEY_PRESSED: 1002
```

**Step 2: Check if parser is running**
```powershell
Get-Process | Where-Object {$_.Path -like "*chtpm_parser*"}
```

**Step 3: Check frame pulse file**
```powershell
Get-Content pieces/display/renderer_pulse.txt
# Should update when menu selection changes
```

**Step 4: Enable history mode**
```powershell
echo "on" > pieces/display/state.txt
```

### If Compilation Fails

**Symptom:** GCC not found, or wrong flags

**Fix:**
```powershell
# Ensure MSYS2 MinGW is in PATH
$env:PATH = "C:\msys64\mingw64\bin;" + $env:PATH

# Or call bash to compile (uses MSYS2 environment)
& C:\msys64\usr\bin\bash.exe -lc "cd '$PROJECT_ROOT' && ./#.buttons/linux/legacy/compile_all.sh"
```

### If GL Window Won't Close

**Manual cleanup:**
```powershell
taskkill /F /IM gl_renderer.+x
taskkill /F /IM orchestrator.+x
# Or use button kill command
.\button.ps1 kill
```
