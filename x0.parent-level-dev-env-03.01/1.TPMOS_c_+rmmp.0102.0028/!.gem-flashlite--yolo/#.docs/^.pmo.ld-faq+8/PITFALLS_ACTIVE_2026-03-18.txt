PMO/TPM Pitfalls Playbook (Active)
Date: 2026-04-03
Purpose: Preserve high-value implementation pitfalls so future agents do not repeat known regressions.

1) WORLD/MAP PATHING DRIFT
- Canonical path is: projects/<project>/pieces/world_<id>/map_<id>/<piece_id>/
- Do not require an extra /pieces/ under map dirs.
- Keep fallback/root lookup only as migration bridge.
- New writes should be canonical-first.

2) HARDCODED PIECE IDS
- Never hardcode scan/load assumptions around fixed ids.
- Pieces can be dynamically created/removed.
- Selection and rendering must be directory/data driven.

2b) HARDCODED LOGIC DRIFT (ARCHITECTURAL RISK)
- Do not hardcode behavior where modular contracts already exist (piece/module/op boundaries).
- Hardcoded shortcuts increase token cost, increase agent confusion, and block partial upgrades.
- Keep editor-player recursion contract-driven so one side can evolve without breaking the other.

2c) NO SHARED .h HEADERS (CONVENTION)
- TPMOS convention: executable modules (.c → .+x) and data files (.pdl, .kvp, .txt) only.
- Do NOT create shared .h header files for path constants, type registries, or helper functions.
- Each .c file is self-contained: defines its own #define macros and static helpers at the top.
- For data-driven values (type→icon mappings, etc.), use external .pdl files as the canonical reference, with local lookup functions in each .c file that needs them.
- Canonical data file: pieces/os/type_registry.pdl

2d) NO SCATTERED HARDCODED SYSTEM PATHS (MEDIUM SEVERITY FIX)
- Never scatter paths like "pieces/master_ledger/plugins/+x/piece_manager.+x" across 7+ .c files.
- Each .c file defines its OWN local constant and helper:
    #define TPM_PIECE_MANAGER_CMD "'./pieces/master_ledger/plugins/+x/piece_manager.+x'"
    /* or for project_root-based calls: */
    #define TPM_PIECE_MANAGER_PATH "pieces/master_ledger/plugins/+x/piece_manager.+x"
    static char* pm_path_for(const char *project_root) { ... }
- Same pattern for ops: #define TPM_OPS_ROOT "pieces/apps/playrm/ops/+x" + local helper.
- If a system path moves, update the macro in each affected .c file. This is intentional: each file compiles independently with no header dependencies.

2e) NO HARDCODED TYPE/Glyph MAPPINGS IN C
- Piece type → icon mappings (npc→&, zombie→Z, chest→T, xlector→X) live in:
  pieces/os/type_registry.pdl
- Each .c file that needs them has its own local lookup function (infer_icon_from_type, icon_from_type).
- Adding a new type: update type_registry.pdl AND the local lookup in each relevant .c file.
- Do NOT create shared type registry headers.

2f) NO HARDCODED PROJECT DEFAULTS
- Ops should default to "template" (system default project), not "fuzzpet_v2" or other legacy names.
- Resolution order: PRISC env vars → location_kvp → player_app manager state → "template".
- Hardcoded project defaults cause silent failures when loading unregistered projects.

2g) DATA-DRIVEN DISPATCH OVER TYPE CHECKING
- Replace `strcmp(type, "npc")` with `strcmp(behavior, "aggressive")` — read behavior from piece state.txt.
- Replace `strcmp(action, "feed")` response key "fed" with the action name directly ("feed").
- The rule: check what the piece DOES (its behavior/traits), not what it IS (its type name).

3) PDL FILE-NAME MISMATCH
- Runtime may encounter both piece.pdl and <piece_id>.pdl conventions.
- Resolver must support both (with deterministic precedence).
- If parser/reader disagree on naming, methods silently fail (nav/hotkeys appear dead).

4) SELECTOR STATE LOSS (0,0 BUG CLASS)
- State rewrite loops can accidentally drop pos_x/pos_y/pos_z lines.
- Any state-rewrite code must preserve unrelated keys.
- Add default guards for selector keys when missing.

5) LOADER PATH REGRESSIONS
- Project loader can fail from wrong layout path mapping.
- Always verify loader target paths exist on disk after changes.

5b) CHTPM MODULE HARD-CODING LEAK
- In CHTPM modules (not parser), avoid baking project-specific behavior into infrastructure code.
- `chtpm_player`, `orchestrator`, and render helpers should stay generic plumbing.
- If project-specific rules appear there, agents copy them and architecture drifts away from PMO modular recursion.

5c) TEMPLATE DEFAULT TRAP (`project.pdl` / entry_layout / player_piece)
- Do not hardcode `entry_layout=game.chtpm` or fixed `player_piece` assumptions in generic templates.
- Missing fields must resolve via scan/registry rules (app/module/layout discovery), not guessed defaults.
- Wrong default entry layout can break visibility/movement and misroute editor vs op-ed flows.
- Rule: detect available layouts/modules from project/app context and write resolved values to explicit metadata.

6) FALLBACK ABUSE
- Temporary fallback is useful, but if used as primary path it masks migration bugs.
- Rule: canonical path first; fallback only if canonical is absent.
- Document each fallback location and planned removal criteria.
- If `pieces/apps/playrm/ops/src/render_map.c` regresses and op-ed shows a blank map, roll back to the last known-safe MAP-ER copy of `render_map.c` before chasing parser or project metadata.
- Keep a known-good source snapshot for `render_map.c` in the workspace so future agents can restore it quickly when `pieces/apps/player_app/view.txt` turns into an all-dots fallback.

7) MAP MEMBERSHIP CONFUSION
- `map_id` in state is compatibility metadata, not sole source of truth.
- Directory containment (world/map path) is authoritative for membership.

8) EVENT FLOW FRAGILITY (STEP 1 SCOPE)
- No-code pal/op-event create/bind must persist to disk, not UI-only memory.
- Save/load parity must include event bindings.
- Local play must boot from saved artifacts without manual edits.

9) PATH CONSTRUCTION WARNINGS
- Watch truncation in snprintf path building.
- Use bounded writes and explicit null-termination where fixed buffers remain.
- Re-run compile_all after path changes.

10) PROCESS/CPU SAFETY
- Avoid runaway loops and orphaned subprocesses.
- Prefer controlled fork/exec/wait patterns for critical calls.
- Keep focus-aware throttling and bounded polling.

11) MIGRATION DEDUP RISK
- During transition, same logical piece may appear in root + world/map.
- Renderer should prefer canonical nested entries and dedup deterministically.

12) CHANGE HYGIENE
- When touching core pathing, verify:
  a) loader launch
  b) save/load roundtrip
  c) local play boot
  d) event trigger path
  e) compile_all warnings in touched components
  f) entry_layout/player_piece resolution (no stale hardcoded defaults)

13) NO-GIT BACKUP MANDATE (MANUAL SNAPSHOTS REQUIRED)
- This workspace may not have usable git history; do not assume rollback exists.
- Before editing high-risk files, create manual backups in-place (same directory) with clear suffixes.
- Recommended suffix pattern:
  - `<filename>.bak.<YYYYMMDD-HHMM>`
  - or folder snapshot: `<dir>.bak.<YYYYMMDD-HHMM>/`
- Minimum backup-required areas:
  1) path resolution / loader / parser files
  2) state rewrite loops
  3) event persistence files
  4) major docs being consolidated/rewritten
- After successful validation, backups can be retained or pruned intentionally; never silently overwrite originals without a backup.

14) DUAL INJECTION BUG (INPUT PARITY, NOT DUPLICATION)
- For one UI action, do not inject both keyboard code and joystick code.
- UI/menu clicks should inject one normalized action key path.
- Joystick codes should come from actual joystick input path only.

15) PULSE SKIP TRAP (CONDITIONAL WAIT BUG)
- Never skip module pulse while waiting for marker files (`view_changed.txt`, etc.).
- Keep pulse/update heartbeat unconditional; only gate composition/render timing.
- If pulse is skipped, history is never consumed and controls appear dead.

16) MARKER FILE DISCIPLINE
- Initialize marker files at startup before watch loops (`view_changed.txt`, `frame_changed.txt`).
- For change detection, prefer marker size/append delta over mtime-only checks.
- Daemons that mutate visible state must append frame marker or UI appears frozen.

17) ONE WRITER RULE (FRAME OWNERSHIP)
- `current_frame.txt` must have a single owner/writer.
- Parser/composer writes the frame; renderer prints only.
- Dual writers cause flicker/corruption and nondeterministic behavior.

17b) STARTUP FOCUS / FRAME FLASH RACE
- If a new session briefly shows an old frame or loses the default `>` selector, stale `pieces/display/current_frame.txt` and/or `pieces/display/active_gui_index.txt` are usually the cause.
- Startup should clear both files before the first render path runs.
- Parser startup should ignore restored focus from a prior session unless the current layout explicitly reestablishes it.
- Symptom pattern: first frame renders without `>` on the expected default control, then the selector appears after the next refresh.

18) STATE/PDL INTEGRITY RULE
- Never overwrite whole piece definitions when setting state.
- State writes must preserve METHOD/RESPONSE/other sections.
- Use section-aware updates; avoid destructive rewrites.

19) ACTION RESPONSE TRACKING
- Responses should reflect the last action/route, not only threshold-derived state.
- If response routing is state-only, user feedback becomes misleading (appears random).

20) ROOT/PATH DRIFT + MEMORY LIFETIME RISKS
- Always resolve `project_root` dynamically from `pieces/locations/location_kvp` (runtime instance-aware).
- Do not rely on relative-path assumptions tied to cwd.
- Never free path buffers before I/O operations complete (use-after-free can create corrupted filenames/filesystem artifacts).

21) DIRECT READ PERFORMANCE TRAP
- High-frequency loops should avoid repeated heavyweight subprocess reads (`popen` storms).
- Use direct mirror/state file reads in hot paths; reserve manager/ops calls for writes/mutations.

22) INPUT/MODULE NORMALIZATION RULE
- Module logic should consume normalized input actions, not device-specific branching everywhere.
- Keep keyboard/joystick mapping in input/parser layers so app modules stay device-agnostic.

23) TRIM/SANITIZE TRAP (MIRROR READS)
- Mirror/state reads often include trailing spaces/newlines; always trim before compare/parse.
- Untrimmed keys/values cause false mismatches and silent branch failures.

24) HYBRID EXPANSION RULE (LARGE DATA IN LAYOUTS)
- Do not force large map payloads into brittle early-stage tag attributes.
- Expand heavy variables at final render/composition stage to avoid truncation/corruption risk.

25) RESPONSE SINK RULE (BACKGROUND OUTPUT)
- Background/plugin stdout is not a reliable user-visible response channel.
- Persist meaningful responses to response files/state and include them in composed frame output.

26) BUTTON/NAV PATTERN CONFUSION (CRITICAL FOR NEW APPS)
- DO NOT manually write nav indicators like "[>] 1. [Login]" or "[1] Login" in layout files.
- The CHTPM parser AUTOMATICALLY adds [>] (focused) and [ ] (unfocused) when rendering <button> elements.
- WRONG: <text label="║  [1] Login ║" />  <- Parser won't recognize as navigable
- CORRECT: <text label="║  " /><button label="Login" onClick="KEY:1" /><text label=" ║" />
- Module does NOT need to track gui_focus_index - parser manages this internally.
- Module does NOT write gui_focus to state - parser reads active_gui_index.txt for sync.
- Follow os.chtpm and fuzzpet_launcher.chtpm as canonical button patterns.
- If nav appears broken ([ ] missing, focus not moving), check:
  a) Layout uses <button> elements, not raw text
  b) onClick="KEY:n" or href="path.chtpm" attributes are correct
  c) Module processes the key codes from onClick
  d) Parser is loading the correct layout file

27) APP vs PROJECT TOPOLOGY (ARCHITECTURAL DECISION)
- APP (pieces/apps/<app_id>/): System tools like op-ed, editor, user auth
  - Launched via href buttons from os.chtpm or other layouts
  - Has <app_id>.pdl with app_id and layout_path fields
  - Module reads layout_path from PDL via resolve_my_layout()
- PROJECT (projects/<project_id>/): User content like games, custom experiences
  - Launched via loader menu selection
  - Has project.pdl with project_id and entry_layout fields
  - project_loader.c reads entry_layout and writes to layout_changed.txt
- DO NOT create projects/user/ for system tools - use pieces/apps/user/
- DO NOT create pieces/apps/fuzz-op/ for games - use projects/fuzz-op/
- Confusing these causes loader failures and wrong state file locations.

28) APP TITLE OVERWRITE (RENDER_MAP CONFLICT)
- `render_map` (pieces/apps/playrm/ops/src/render_map.c) has a default `app_title = "PIECEMARK PLAYER"`.
- It writes this title to `pieces/apps/player_app/state.txt` on every render.
- If a system app (like User Auth) writes its own title to state and then calls `trigger_render()`, `render_map` will immediately overwrite it.
- FIX: System apps with their own dynamic layouts (using `${piece_methods}`) should NOT call `trigger_render()` unless they actually need a map rendered.
- FIX: `chtpm_parser.c` should clear `app_title` at the start of `load_vars()` to prevent stale titles from persisting across app/project switches.
- Rule: verify which process "owns" the header title in the current layout context.

29) NAVIGATION BOUND TRAP (digit_accum limit)
- `process_key()` in the parser bounds digit input to enable immediate single-digit jumping.
- Large bounds (like `MAX_ELEMENTS`) break this "strong inside" behavior by forcing multi-digit waits.
- RULE: Use `count_projects()` or similar context-aware bounds to preserve immediate responsiveness. Don't "over-fix" this at the expense of core UI feel.

30) SEA SHELL INTEGRITY (Core Nav/Jump behavior)
- The core logic of `do_jump()` is designed to move focus only, not execute actions.
- Attempting to add immediate execution to `do_jump()` breaks the canonical UI flow.
- RULE: Preserve the strong inside (core nav); execution must remain triggered by Enter or explicit onClick routing.

31) APP LAYOUT project_id DRIFT (RECURRING AGENT MISTAKE)
- When an APP layout (e.g., `pieces/apps/user/layouts/user.chtpm`) is loaded via `href="..."`, the parser does NOT automatically set `project_id`.
- `load_vars()` reads `project_id` from `pieces/apps/player_app/manager/state.txt`, which may contain a stale value from a previously loaded PROJECT (e.g., `project_id=fuzz-op`).
- Consequence: `resolve_dynamic_pdl_path()` looks in wrong locations:
  - WRONG: `projects/fuzz-op/pieces/user_main/piece.pdl` (doesn't exist)
  - CORRECT: `pieces/apps/user/pieces/user_main/piece.pdl` (exists)
- Symptom: App loads but `${piece_methods}` shows nothing, buttons don't work, module may not launch correctly.
- RULE: When loading an APP layout via href, explicitly set `project_id` to match the app:
  ```c
  // In parse_chtm(), after layout file is read:
  if (strstr(current_layout, "pieces/apps/user/")) {
      set_var("project_id", "user");
  }
  ```
- Alternative: Use `LAUNCH:user` command instead of `href`, which can set up the full app context.
- Apps affected: user, op-ed, man-ops, man-pal, man-add, gl-os (any `pieces/apps/*/` layout).
- This mistake has tripped up multiple agents because PROJECT loading (`LOAD_PROJECT:`) sets `project_id` correctly, but APP loading (via `href`) does not.

32) STATE FILE CONTEXT CONFUSION (player_app vs app-specific)
- System apps share `pieces/apps/player_app/` for routed input (`history.txt`) and state (`state.txt`, `manager/state.txt`).
- When switching between apps/projects, stale state in these shared files can cause:
  - Wrong `project_id` (see Pitfall #31)
  - Wrong `active_target_id` (methods load from wrong PDL)
  - Wrong `app_title` (see Pitfall #28)
- RULE: Apps should write their context (`project_id`, `active_target_id`, `app_title`) to shared state files immediately on launch/layout change.
- Parser should clear or overwrite context variables at start of `load_vars()` to prevent stale data persistence.

33) CLI_IO INPUT PERSISTENCE (input_buffer vs layout var)
- When using `<cli_io>` elements for text input (username, password, etc.), the render logic must display `el->input_buffer` content even after ESC deactivates the field.
- BUG: After typing "hi" and pressing ESC, field shows empty `[]` instead of `[hi]`.

34) STALE FRAME RELAUNCH ILLUSION (ACTIVE WRAITH CROSSOVER)
- A runtime can relaunch and still show an older user-visible frame snapshot.
- Symptom:
  - current visible frame shows stale key label / stale clock / stale open window
  - underlying state files may already be newer
- This is not just an input bug. It is usually a frame ownership or render-trigger mismatch.
- Rule:
  - if the visible clock/key do not advance after relaunch, verify the actual visible frame writer first
  - do not trust sidecar state files as proof of a live UI

35) ONE INPUT HISTORY RULE (ACTIVE WRAITH CROSSOVER)
- Working TPMOS app/project managers usually consume one canonical interact history such as:
  - `pieces/apps/player_app/history.txt`
- Do not let a project simultaneously depend on:
  - raw keyboard history
  - app history
  - custom session history
  as equal live input owners.
- Keep raw keyboard upstream, and keep project/app semantics on one canonical routed history.

36) ONE VISIBLE FRAME WRITER RULE (ACTIVE WRAITH CROSSOVER)
- It is acceptable to generate secondary audit/projection files, but only one component should own the user-visible current frame.
- If one system writes:
  - `pieces/display/current_frame.txt`
  and another writes:
  - project/session/frame.txt
  - manager/view.txt
  treat only one as authoritative for the user-visible terminal.
- Otherwise you get the exact failure class seen in Wraith:
  - state files advance
  - visible frame lags
  - debugging becomes misleading
- CAUSE: Render used `scratch_substituted` (from layout variable `${cli_username}`) instead of `el->input_buffer`.
- FIX: In `render_element()`, check if `el->input_buffer` has content and use it for display when not active.
- File is source of truth: parser reads last U/P value from `cli_buffers.txt` on parse, ESC only deactivates (never clears).

34) JOYSTICK THREAD BLOCKING (orchestrator join deadlock)
- Joystick input runs forever waiting on `/dev/input/js0` events.
- BUG: Ctrl+C hangs because main thread waits on `pthread_join(joy_t)` but joystick thread never returns.
- CAUSE: `launch_and_register()` calls `waitpid()` which blocks until child exits (never for joystick).
- FIX: Joystick thread should fork-and-return (don't wait), and main should NOT join the joystick thread.
- Signal handler kills all children via `kill(0, SIGTERM)` + `kill_all.sh`.

35) BACKUP FILE BLOAT (.bak* accumulation)
- Backup files (`*.bak`, `*.bak.*`, `*.bak*`) accumulate during development and can add 400KB+ to project size.
- This is expected and intentional (no-git mode requires manual snapshots).
- Use `./#.tools/remove_baks.sh` to clean up when needed.
- Use `./#.tools/remove_binaries.sh` to remove compiled `.+x` files (also large).
- Use `./#.tools/check_size.sh` to monitor disk usage.

36) GL-OS MULTI-LINE STATE TRAP
- Standard state file parsers read line-by-line (`key=value`).
- Complex UI blocks (like `menu_options`) contain newlines.
- BUG: Parser only captures the first line; items 2+ are lost.
- FIX: Use the "Continuation Pattern" where lines without an `=` are appended to the previous key's value.

37) GL-OS INPUT FOCUS LOCK LEAK
- GL-OS creates `input_focus.lock` to isolate input from ASCII-OS.
- If GL-OS crashes or is killed forcefully, the lock persists.
- Symptom: Joystick/Keyboard input appears "dead" in ASCII-OS after GL-OS usage.
- FIX: `kill_all.sh` MUST explicitly `rm -f pieces/apps/gl_os/session/input_focus.lock`.

38) GL-OS IDLE THROTTLE VS RESPONSIVENESS
- Aggressive throttling (100ms timer) saves CPU but makes input feel "laggy".
- FIX: Input callbacks (`keyboard`, `mouse`, `joystick`) must reset `last_interaction_time`.
- The timer should stay at high-performance (30ms) for at least 5 seconds after any interaction before backing off.

39) GL-OS JOYSTICK AXIS ASSIGNMENT
- Standard CHTPM axis mapping may differ from GL-OS expected behavior.
- In GL-OS desktop, the Y-axis is typically mapped to vertical menu movement.
- Ensure `last_joy_y` is updated outside the focus check to prevent edge-triggering "stickiness".

40) GL-OS SPECIAL KEY REGISTRATION
- Standard `glutKeyboardFunc` does not capture arrow keys.
- BUG: Arrow keys don't navigate terminal menus.
- FIX: Implement and register `glutSpecialFunc` to handle `GLUT_KEY_UP/DOWN` specifically.

41) COMPILE_ALL.SH ENTRY DELETION (SILENT MODULE FAILURE)
- When agents refactor compile_all.sh, they may accidentally remove entries for project-specific managers.
- Symptom: Project loads but shows generic UI (wrong module), hotkeys don't work, actions are missing.
- Example: `fuzz-op_manager.+x` was removed, causing fallback to `playrm_module.+x` (generic actions).
- RULE: After ANY compile_all.sh changes, verify ALL project managers are still listed:
  - `projects/*/manager/+x/*_manager.+x` binaries exist
  - Each has a corresponding gcc line in compile_all.sh
- Quick check: `find projects -name "*_manager.c" | wc -l` vs `find projects -name "*_manager.+x" | wc -l`
- If source exists but binary doesn't, add to compile_all.sh immediately.

50) REGISTRY FULL SILENT FAILURE (DESKTOP)
- Project managers with window registries (like Wraith) often have hardcoded `MAX_WINDOWS` limits.
- When the limit is reached, launch commands are ignored.
- Symptom: UI buttons and Enter keys appear "dead" even though the input pipeline is healthy.
- RULE: Always log "registry full" errors to the manager log and provide UI feedback if possible. Check logs before assuming an input bug.

51) FIXED BUFFER SNAPSHOTS (RENDER STABILITY)
- Using dynamic allocation (`strdup`/`malloc`) for frame snapshots in high-frequency parser loops (e.g., for mouse hit-testing) is risky.
- Race conditions or fragmentation can lead to crashes or corrupted coordinate mapping.
- RULE: Use fixed-size buffers (`char last_rendered_frame[MAX_BUFFER]`) for stable, reliable hit-testing snapshots.

52) MOUSE MOVE REDRAW SPAM
- Do not trigger a full frame redraw (`trigger_render`) on every mouse movement event (SGR 32+).
- Movement-heavy sessions will saturate the CPU and lag the UI.
- RULE: Only trigger renders on mouse click transitions (press/release, SGR < 4). Updates to state-on-disk for debug displays can still happen on movement, but should not dirty the frame.

42) 404 PAGE VS MODULE MISSING (DIAGNOSIS GUIDE)
- 404 page shows when: Project exists in `projects/` but NOT in `pieces/os/project_routes.kvp`
- Module Missing page shows when: Project IS in routes, but module binary doesn't exist
- Generic/wrong UI (OLD BEHAVIOR, pre-fix): Module binary missing but no error shown
- Diagnosis steps:
  1) Check project_routes.kvp: `grep project_name pieces/os/project_routes.kvp`
  2) Check binary exists: `ls -la projects/project_name/manager/+x/`
  3) If binary missing, add to compile_all.sh and recompile
- Do NOT add route to project_routes.kvp if module isn't compiled - it will show Module Missing page.
- NEW FEATURE (post-fix): Parser now detects missing modules and shows helpful error page instead of silent failure.
- Variables set: `module_error=MISSING`, `module_error_path=<path>`

43) MODULE BINARY CHECK (NEW ERROR PAGE)
- Parser now checks `access(module_path, X_OK)` before launching modules.
- If binary is missing, switches to `module_missing.chtpm` layout with:
  - Project ID and missing module path displayed
  - Possible causes (not compiled, compile_all.sh entry removed, binary deleted)
  - Fix steps (check source, add to compile_all.sh, recompile)
- This prevents confusing "generic UI" symptom from silent module failures.
- Variables set: `module_error=MISSING`, `module_error_path=<path>`

44) WINDOWS ARROW KEY TRAP (MSYS2 PSEUDO-TERMINAL)
- Physical arrow keys don't work when running orchestrator via MSYS2/mintty bash.
- Root cause: `_kbhit()` doesn't work in pseudo-terminals, only native Windows console.
- Symptom: Number keys (1-9) work for navigation, but arrow keys do nothing.
- Keyboard injection works (writing to history file), proving parser is functional.
- Fix attempted: Added fgetc() fallback and escape sequence parsing in readKey().
- Status: Still broken - may require native Windows console (not MSYS2 bash).
- Workaround: Use number keys for menu navigation on Windows.
- Files affected: `pieces/chtpm/plugins/orchestrator.c` (readKey function)

45) WINDOWS JOYSTICK (XINPUT - IMPLEMENTED)
- Linux uses `/dev/input/js0` (linux/joystick.h) - doesn't exist on Windows.
- Windows uses XInput API (xinput.h) for Xbox controllers.
- IMPLEMENTATION (April 2, 2026):
  - File: pieces/joystick/plugins/joystick_input_win.c
  - Supports 4 Xbox controllers (XInputGetState 0-3)
  - Maps buttons to codes 2000-2013 (A,B,X,Y, bumpers, D-pad, etc.)
  - Maps axes to codes 2100-2109 (left/right sticks, triggers)
  - Writes to same input files as Linux (keyboard/history.txt, etc.)
  - Compiled by: pieces/buttons/windows/legacy/compile_all.ps1
  - Killed by: pieces/buttons/windows/legacy/kill_all.ps1 (joystick_input)
- Status: Linux joystick working. Windows XInput code complete, needs hardware test.
- Files: pieces/joystick/plugins/joystick_input.c (Linux), joystick_input_win.c (Windows)

46) DESKTOP CHROME DUAL-OWNERSHIP (ACTIVE WRAITH LESSON)
- Do not let one visible desktop action execute from both raw manager-side Enter and parser button activation.
- Symptom:
  - first minimize/restore seems fine
  - later cycles become inconsistent or appear random
  - logs show duplicate or missing action paths
- Correct rule:
  - parser owns activation of titlebar/taskbar buttons
  - manager owns canonical state mutation after receiving the routed command
  - raw keyboard digits/arrows may still update selection/debug state, but raw Enter must not duplicate desktop chrome activation

47) TOOLBAR DUPLICATION LIE (ACTIVE WRAITH LESSON)
- A window must not be both:
  - open in desktop chrome
  - present in the taskbar as a restore target
- Correct desktop truth model:
  - open window => not in restore strip
  - minimized window => restore strip entry visible
- If both appear together, the frame is lying about lifecycle state and later minimize/restore behavior will drift.

48) MARKER-FILE BYPASS DRIFT
- Standard visible refresh remains `pieces/display/frame_changed.txt`.
- Do not add an ad hoc project-specific "reparse now" file just because one desktop shell has a stale-frame bug.
- First fix ownership and projection:
  - manager writes canonical state
  - manager appends `frame_changed.txt`
  - parser reparses or recomposes under the normal pulse
- If more freshness is needed, compare projected-state signatures during compose instead of creating a second visible-frame trigger.

49) SINGLE-PRESENTER / MULTI-INSTANCE CONFUSION
- A desktop shell may have a scalable manager-side registry before it has full multi-window tiling.
- That is acceptable if the contract stays honest:
  - registry can hold multiple instances
  - current presenter shows one active open window
  - other instances appear minimized in taskbar restore slots
- Do not call a single-window special case "the registry" if it still hardcodes one fake window. The registry must remain canonical even when the presenter is simpler than the future end state.

Operational rule:
If a change touches path resolution, state rewriting, loader mapping, or event persistence, treat it as high-risk and run full Step-1 matrix before merge.

104) PROJECT CLONING AND PATH DRIFT
- SYMPTOM: Duplicated projects show incorrect UI/Headers, or paths revert to the original project name (e.g., `op-ed` appearing inside `slop-ed-dev`) even after a rename.
- ROOT CAUSE:
    1. "Double-rename" bug: running `sed` on strings that already contain the replacement pattern (e.g., replacing `op-ed` with `slop-ed-dev` in a file path that already contained `slop-ed-dev` makes it `slslop-ed-dev-dev`).
    2. Hardcoded project paths in `gui_state.txt` or `manager.c`.
    3. Stale `project_id` in `project.pdl` or `gui_state.txt`.
    4. Incorrect `entry_layout` module paths in `.chtpm` files after renaming the directory.
- MANDATE: When duplicating a project:
    1. Use a temporary intermediate name to avoid double-renaming if reusing patterns.
    2. Verify `project.pdl` and `manager.c` for hardcoded strings.
    3. Clear ALL `gui_state.txt` and temporary session files before first run.
    4. Recompile EVERYTHING (`./button.sh compile`) to ensure binaries are built from the new paths.
- TPM PRINCIPLE: "A duplicate is only as good as its rename. If one link in the chain is stale, the whole system drifts."

53) COMMAND QUOTING TRAP (system() calls)
- Quoting the entire command string (e.g. system("'path/to/bin arg'")) causes the shell to treat the command + arguments as a single file name.
- RULE: Quote ONLY the binary path part of the command string.
- CORRECT: asprintf(&cmd, "'%s/%s' %s", root, bin, args);

54) SILENT ACTION FAILURE (popen vs system)
- Using system() for hotkey execution doesn't capture stdout/stderr, leaving the UI with empty response lines.
- RULE: Use popen() to execute actions and capture the first line of output for display in the layout ${method_response} field.
- If an op outputs "Collected Treasure", the user must see it.

55) PIECE ID DECOUPLING (xlector vs selector)
- Shared ops in pieces/apps/playrm/ops/src/ often hardcode "selector" or "fuzzpet".
- Symptom: Hotkeys "work" but target the wrong piece (or nothing).
- RULE: Ops must use argv[1] as the piece ID, defaulting to xlector/selector only if missing.
- Rule: Ops and PDL readers MUST use recursive path resolution (find_piece_state_recursive) to support canonical world/map nesting.

56) WRAITH "TWO RENDER PATHS" MISREAD (AGENT-CONFUSION CLASS, NOT A CODE BUG)
- SYMPTOM: A new Wraith-hosted project's embedded desktop window shows plain,
  unframed text with no nav buttons, while its `layouts/<project>.chtpm` file
  (with `<button>` tags and `${var}` placeholders) looks correct but never
  appears to render.
- WRONG DIAGNOSIS (an agent's own real mistake, corrected same session): concluding
  Wraith has two structurally separate, competing rendering pipelines — an
  "embedded path" (`session/wraith_body.txt`, allegedly plain-text-only) vs. a
  "standalone path" (`layouts/*.chtpm`, real substitution) — and that a project
  must pick one or bridge them with an `href` pop-out button. This framing is
  wrong and wastes a full research cycle.
- CORRECT MODEL: it is the SAME render pipeline as any TPM project (see
  `^.chtpm-faq/CHTPM_TPM_DOX+1.md`'s "Module -> ops -> render op produces
  artifact -> parser composes frame"). `session/wraith_body.txt` **is** that
  artifact — the project's own hand-rendered copy of what its layout shows,
  with `${var}`s already resolved to literal values by the project's own ops
  (not by a generic parser re-opening the `.chtpm` file). `layouts/<project>.chtpm`
  remains the authoritative structural spec (box framing, button placement,
  nav order) that the ops's hand-rendered output should match.
- KEY MECHANIC: a `wraith_body.txt` line whose first character is `<` is
  appended verbatim by `wraith-alpha_manager.c`'s `append_project_probe_body()`
  (confirmed ~line 1645) and parsed downstream as real `<text>`/`<button>`/
  `<cli_io>` markup — genuine nav-indexed, clickable elements. A line NOT
  starting with `<` is wrapped in a fixed plain-text border and can never
  become a button (ties into Pitfall #26).
- RULE: to get a project's real layout structure (box frame, INTERACT
  button, numbered nav) showing inside its embedded desktop window, the
  ops must emit the same `<text>`/`<button>` tags the layout would contain,
  values already substituted, as `<`-prefixed lines directly into
  `session/wraith_body.txt`. Do not treat this as a workaround or a
  second competing system — it is the render step, performed by project
  code instead of the generic parser, same as any TPM render-op artifact.
- REFERENCE: `settings_manager.c`'s own header comment documents this
  explicitly — settings substitutes `${settings_menu_markup}` into
  `settings.chtpm` via the generic href path, AND separately ships a
  hand-written `session/wraith_body.txt` with equivalent `<button>`/
  `<cli_io>` markup so the same menu renders correctly when opened
  embedded from the desktop launcher instead.

57) ASCII KEYBOARD COMMAND RELAY -- HISTORY PATH DRIFT (SILENT DEAD LETTERBOX)
- SYMPTOM: a project's onClick command (any prefix requiring the full
  chtpm_parser.c -> inject_command() -> history-file -> wraith-alpha_manager.c
  poll -> route_command() round trip) works instantly via GL mouse click but
  does nothing -- or only "catches up" seconds later, coinciding with an
  unrelated later mouse event -- when activated via ASCII keyboard Enter.
- ROOT CAUSE: `chtpm_parser.c`'s `resolve_project_history_path()` built
  `projects/%s/history.txt` from `project_id` unconditionally.
  `wraith-alpha_manager.c`'s `main()` loop, however, only ever polls the
  ONE fixed path `projects/wraith-alpha/session/history.txt` for
  "COMMAND: " lines (see `process_history_file()` call sites). For the
  wraith-alpha desktop shell itself, and for every embedded window it
  hosts (whose `project_id` is the nested
  "wraith-alpha/wraith-projects/<id>" form -- see Pitfall #31's own fix),
  the built path was missing the "session/" segment (and, for nested ids,
  had extra segments too), producing a file NOTHING ever polls. Commands
  written there (`inject_command()`/`inject_raw_key()`, e.g. SETTINGS_PAGE:,
  KEY:n nudge buttons, PROJECT_ACTION:) vanish silently -- no error, no log,
  just never routed. GL mouse clicks are unaffected because GL calls
  `route_command()` directly in-process, never touching this file relay,
  which is exactly why "it works with mouse but not keyboard" is the
  tell-tale symptom of this specific pitfall (do not misdiagnose as a
  render-trigger/marker-file timing bug -- the render pipeline itself is
  fine; the command never arrives for it to render).
- FIX: `resolve_project_history_path()` must special-case any project_id
  equal to `wraith-alpha` or starting with `wraith-alpha/` to route to
  `projects/wraith-alpha/session/history.txt` (the one path the shell's
  poll loop actually reads), leaving the legacy `projects/<id>/history.txt`
  convention untouched for standalone top-level projects (op-ed, yahoo,
  mouse-test, etc.) that run outside the wraith-alpha shell with their own
  manager/poll loop.
- RULE: whenever a NEW round-trip command path is introduced (a new onClick
  prefix, a new project feature needing keyboard-driven cross-process
  dispatch), verify with `grep` that the exact file `inject_command()`/
  `inject_raw_key()` writes to is the SAME literal path the consuming
  poll loop stats/reads -- string-build the path from both ends and diff
  them by hand. Do not assume symmetry; this bug shipped invisibly because
  every write appeared to "succeed" (fopen/fprintf never failed) and every
  poll loop iteration also ran without error -- they were just never
  talking about the same file.
- CORRECTION (same day, confirmed live): the fix above is real and worth
  keeping (it's still the correct behavior for `resolve_project_history_path()`
  generically), but for the wraith-alpha desktop shell specifically it
  turned out NOT to be the active code path for this exact symptom --
  `inject_command()`'s PRIORITY 1 branch (`interact_history_path`, set
  from the outer KPI/desktop layout's `<interact>` tag) already pointed
  at `projects/wraith-alpha/session/history.txt` and fired first. Live
  `alpha_manager.log`/`gui_state.txt` mtimes proved the command WAS
  received and route_command() DID run within the same second as the
  Enter press -- the round trip was never broken for this case. See
  Pitfall #58 for the actual root cause of "Enter does nothing, mouse
  catches it up." Lesson: a plausible, evidence-consistent theory (path
  mismatch fits "keyboard silently fails, mouse works") can still be
  wrong about WHICH file/branch is live -- re-verify with fresh log
  timestamps after each fix attempt instead of declaring done once the
  fix compiles and matches the symptom on paper.

58) SETTINGS_PAGE: HANDLER MISSING update_state() (STALE COMPOSED FRAME,
    NOT A ROUTING FAILURE)
- SYMPTOM: identical to Pitfall #57's -- ASCII keyboard Enter on a
  SETTINGS_PAGE:-onClick button appears to do nothing; the transition
  only appears after a later, unrelated mouse move/click. This is the
  ACTUAL root cause of that symptom (Pitfall #57's history-path theory,
  while a real bug in its own right, was not what was happening live).
- ROOT CAUSE: in `wraith-alpha_manager.c`'s `route_command()`, every
  other branch that changes what should appear on screen calls
  `update_state(...)` THEN `trigger_render()` as a pair -- `update_state()`
  (via `write_projection()`) re-projects the active window's current
  `wraith_body.txt`/`gui_state.txt` into `alpha_state.txt` /
  `desktop_ui_state.txt`, the actual files chtpm_parser's ASCII composer
  reads to build the "WRAITH DESKTOP GUI" frame; `trigger_render()` only
  pings the marker files (`frame_changed.txt`, `state_changed.txt`) that
  tell chtpm_parser's main loop to redraw NOW, from whatever is currently
  in those projection files. The `SETTINGS_PAGE:` handler called
  `run_active_project_input_op()` (correctly updating settings' own
  `gui_state.txt`/`wraith_body.txt` -- confirmed via file mtimes landing
  within the same second as the keypress) then jumped straight to
  `trigger_render()`, skipping `update_state()`. So the marker ping fired
  and chtpm_parser dutifully redrew -- the STALE pre-transition
  projection, because it was never regenerated. The very next mouse
  event's handler DOES call `update_state()`, which finally re-projects
  the (already correct, already-written) settings state, producing the
  transition the user sees -- coincidentally, right after the mouse
  event. Confirmed live via `alpha_manager.log` + `gui_state.txt` mtimes:
  state was correct at time T (right after the Enter-triggered
  `Command received: SETTINGS_PAGE:...` log line); the visible frame only
  updated at T+(next mouse event).
- FIX: add `update_state(0);` before `trigger_render();` in the
  `SETTINGS_PAGE:` branch, matching the `KEY:`/`PROJECT_ACTION:` pattern
  immediately above it in the same function.
- RULE: `trigger_render()` alone NEVER refreshes what gets rendered --
  it only asks chtpm_parser to redraw sooner. Any `route_command()`
  branch (or future one) that changes state visible in the ASCII/GL
  frame must call `update_state()` (or whatever recomputes the
  projection files) BEFORE `trigger_render()`, not just after some other
  side-effecting call. When adding a new command branch, diff it against
  a working sibling branch (KEY:/PROJECT_ACTION:) line-by-line for this
  exact pairing rather than assuming "it writes state somewhere, calling
  trigger_render() must be enough."

59) MOUSE CALIBRATION APPLIED LOCALLY, FORWARDED RAW (SPLIT-BRAIN
    COORDINATE BUG)
- SYMPTOM: a click/drag that hits its target when handled LOCALLY (GL's
  own `hit_test_semantic_action()` in `wraith_gl.c`, or `chtpm_parser.c`'s
  own two-step-activation hit-test against `last_rendered_frame`) can
  still miss, or produce a subtly-wrong result, in code DOWNSTREAM of the
  same event -- e.g. `wraith-alpha_manager.c`'s `handle_mouse()`, which
  only ever receives the forwarded `MOUSE_MOVE`/`MOUSE_MOVE_CELL`
  command, not whatever local variable held the calibrated coordinate.
  Concretely: a window-drag (multi-win-j13.txt Phase 4) hit-testing a
  window's title bar by exact row/cell match worked inconsistently with
  real mouse input even though the same math worked fine via injected
  test commands with hand-picked coordinates.
- ROOT CAUSE: both `wraith_gl.c`'s `hit_test_semantic_action()` and
  `chtpm_parser.c`'s `handle_mouse()` apply a known calibration
  correction (a mouse-offset file/constant -- see below) to a LOCAL
  variable (`cell_x`/`cell_y` in the GL case, `cal_x`/`cal_y` in the
  ASCII case) used ONLY for that function's own immediate hit-test. The
  SAME function then forwards the ORIGINAL, uncalibrated `(x, y)` --
  not the corrected copy -- onward via `emit_mouse_event()` /
  the `COMMAND: MOUSE_MOVE ...` line. Every downstream consumer of that
  forwarded command (anything in `wraith-alpha_manager.c`) therefore
  works from a DIFFERENT, uncorrected coordinate space than the
  renderer's own local click handling did -- a split-brain bug: the
  calibration exists and is even applied, just not everywhere the raw
  coordinate flows to.
- FURTHER COMPOUNDING (GL specifically): `wraith_gl.c`'s calibration
  mechanism (`load_mouse_offset()`, reading `#.mouse-offset.txt` from
  the process's own cwd) was ALSO silently inert this whole time -- that
  file never actually existed anywhere in the project tree, so the
  offset was always (0,0) regardless of intent. A calibration NOTE
  (`OFFSET_Y=7`, dated 2026-06-15, "Terminal reported Y=10 for a target
  at Row 3") existed as a standalone text file at the workspace root,
  but was never deployed to the actual path the code reads -- documenting
  a fix and shipping a fix are not the same step; verify the file the
  code ACTUALLY `fopen()`s exists, don't just confirm a note describing
  the right number exists somewhere in the workspace.
- FIX (2026-07-13): calibration is now applied EXACTLY ONCE, at the
  earliest point a raw coordinate is available, BEFORE it is used for
  anything (local hit-test OR forwarding) -- not independently
  re-applied (or, worse, applied in one place and skipped in another)
  by each consumer:
  - `wraith_gl.c`: new `apply_mouse_offset()`, called in `mouse()`
    (both `GLUT_DOWN` and `GLUT_UP`) and `motion()` immediately after
    `map_window_mouse_to_texture()` succeeds, before either
    `hit_test_semantic_action()` or `emit_mouse_event()` see the
    coordinate. `hit_test_semantic_action()` no longer reads the offset
    file itself -- its caller already corrected the input.
  - `chtpm_parser.c`: the existing `cal_x`/`cal_y` calibration (was
    scoped inside the `btn==0` hit-test branch only) is now computed
    unconditionally at the top of `handle_mouse()`, and the forwarded
    command uses `cal_x`/`cal_y` too -- but ONLY for wraith-alpha's own
    `project_id` (`MOUSE_MOVE_CELL`). Every other chtpm project
    (mouse-test, etc.) still gets the plain, uncalibrated `x`/`y` in its
    `MOUSE_MOVE` command deliberately: mouse-test's entire purpose is
    displaying the RAW reported coordinate for calibration probing in
    the first place (this exact offset was originally discovered
    through a "click here to calibrate" probe of that shape) -- silently
    "fixing" its output would defeat that purpose for anyone using it as
    intended.
- RULE: when a codebase has a known input-calibration correction, grep
  for EVERY place the raw (uncorrected) coordinate is still read or
  forwarded after the correction is computed, not just the one place a
  fix happened to land first. A calibration that's "sometimes applied"
  is worse than no calibration at all -- it makes some code paths
  correct and others wrong in a way that looks like a completely
  different, unrelated bug (e.g. "drag works going one direction but not
  the other") depending on which path happens to be exercised.
- RULE: a calibration file referenced by `fopen()` in code is a claim
  that the file exists and is deployed -- verify with `find`/`ls`
  before trusting that a documented calibration value is actually
  active, especially if a workspace-root note or handoff doc describes
  the right number but the code's own expected path doesn't have it.

60) CHROME ICON ROW: ANCHOR TO THE ACTUAL EDGE IT'S SUPPOSED TO SIT AT,
    NOT A FIXED OFFSET FROM SOMETHING VARIABLE-LENGTH
- SYMPTOM: GL's window-chrome icon row (context-menu/geom/minimize/
  close, `write_semantic_projection_files()`'s per-window OBJECT
  emission) rendered with a large, inconsistent dead gap before it for
  short window titles, while ASCII's rendering of the identical row
  never showed this gap at all.
- ROOT CAUSE: the title's own GL object used a flat, hardcoded `w=58`
  (reserving that width regardless of the actual title string length),
  and the icon row's own starting x was a second, independently
  hardcoded literal (`63`) tuned to look right for roughly that width
  assumption. ASCII's rendering of the same row has no equivalent
  problem because it's built by literal string concatenation -- the
  dash-fill after the title button starts immediately after the real
  title text ends, so ASCII's gap naturally shrinks/grows with the
  actual title length. GL's fixed-reservation model had no such
  self-correcting property.
- FIRST FIX (insufficient on its own, corrected same day after live
  user feedback): sized the title's own box to the actual title text
  length and started the icon row right after it plus a small fixed
  gap. This fixed the "huge dead gap for short titles" symptom, but
  produced the WRONG end state: real window chrome (min/max/close
  buttons) is conventionally pinned to the frame's own right edge,
  independent of title length, not glued to wherever the title text
  happens to end.
- CORRECT FIX: right-justify the icon row against the frame's own right
  edge (`frame_content_width`), computed via a pre-pass summing every
  icon's own display width, with a small margin so the last icon's
  right edge lands one character short of the frame's actual border
  (not flush/hanging off it) and a floor clamp at "just past the title
  box" so a very long title can never make the icon row overlap the
  title itself even at the cost of that margin on an unusually narrow
  frame.
- RULE: when anchoring one variable-width UI element relative to
  another variable-width element, ask which one is the ACTUAL semantic
  anchor (here: the frame's own edge, a fixed, well-known reference) vs.
  which one merely happens to sit nearby today. Anchoring to "whatever's
  next to it" only works by coincidence for whatever content length was
  true at the time it was tuned, and silently breaks (or, as with the
  first fix here, gets fixed into a still-wrong shape) the moment that
  content's length changes.
