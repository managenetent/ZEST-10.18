       1 # GL-OS Vision & Architecture: The Virtual Desktop Host
       2 **Date:** 2026-04-22
       3 **Status:** ARCHITECTURAL BLUEPRINT - VERSION 2.0
       4
       5 ---
       6
       7 ## 1. CORE VISION: GL-OS AS A HOST ENVIRONMENT
       8
       9 GL-OS is designed as a **high-fidelity, modular desktop environment** that
         **hosts independent GL-native applications**. It is NOT a direct renderer of
         ASCII processes or their internal memory states. Instead, it functions as a
         sophisticated display manager and input router for projects that adhere to
         TPMOS principles.
      10
      11 ### 1.1 The "Host" vs. "Mirror" Distinction
      12
      13 -   **GL-OS as Host:** The GL-OS desktop environment manages windows, input
         events, and the rendering pipeline for GL-native applications. It provides the
         "theater" for these applications.
      14 -   **File-State Mirroring:** GL-OS applications within this host environment
         interact with **project-specific data files** (e.g., `current_frame.txt`,
         `state.txt`, `view.txt`, tile definitions) located within their own project
         directories (e.g., `projects/gl-os/fuzz-op-gl/`). This ensures data
         sovereignty and decouples GL applications from the internal state of ASCII
         TPMOS processes.
      15 -   **Process Sovereignty:** GL-OS and any ASCII TPMOS applications remain
         entirely **independent processes**. Communication and state synchronization
         between them are mediated exclusively through file I/O, adhering to the TPMOS
         "filesystem as truth" principle.
      16
      17 ---
      18
      19 ## 2. ARCHITECTURE: Modular GL Applications
      20
      21 ### 2.1 Project Structure (`projects/gl-os/`)
      22
      23 Independent GL applications will reside in their own subdirectories under
         `projects/gl-os/`. Each application will follow a standard structure:
      24
      25 ```
      26 projects/gl-os/
      27 └── fuzz-op-gl/              # Example: A GL-native version of Fuzz-Op
      28     ├── fuzz-op-gl.pdl       # Application descriptor (methods, state, etc.)
      29     ├── layouts/             # GLSL shaders, UI definitions
      30     │   └── fuzz-op-gl.glsl
      31     ├── manager/             # Application logic (C/C++)
      32     │   ├── src/
      33     │   │   └── fuzz-op_manager.c
      34     │   └── +x/
      35     │       └── fuzz-op_manager.+x
      36     ├── pieces/              # Project-specific data
      37     │   ├── fuzzball/        # Player character piece
      38     │   │   ├── piece.pdl
      39     │   │   └── state.txt
      40     │   ├── world_01/map_01/ # Map data
      41     │   │   └── ...
      42     └── assets/              # Graphical assets
      43         ├── tiles/           # 8x8x8 RGB cube definitions (.txt)
      44         │   ├── wall.txt
      45         │   └── floor.txt
      46         └── sprites/         # 2D/3D sprites (ASCII art with transparency)
      47             └── player.png # Example sprite file
      48 ```
      49
      50 ### 2.2 Rendering Pipeline
      51 -   **Tile-Based Rendering:** GL-native applications will utilize a renderer
         that interprets `.txt` files containing RGB256 codes for 8x8x8 cubes.
      52     -   **2D View:** Cubes are rendered as flat tiles.
      53     -   **3D View:** Cubes are extruded based on Z-level data, creating a
         sense of depth.
      54 -   **Sprite Support:** Rendering will also accommodate 2D/3D sprites with
         transparency layers for entities and other dynamic objects.
      55 -   **Camera System:** Support for switchable camera perspectives: **Free
         Camera**, **1st Person**, and **3rd Person**.
      56
      57 ---
      58
      59 ## 3. DEVELOPMENT & INTEGRATION
      60
      61 ### 3.1 Launching GL Apps
      62 -   **Project Loader Integration:** The ASCII Project Loader menu
         (`playrm/loader.chtpm`) will include entries for GL projects (e.g.,
         "Fuzz-Op-GL", "Op-Ed-GL").
      63 -   **`gl_desktop.c` Role:** The GL-OS desktop manager will intercept
         selections for GL projects. Instead of just displaying text or mirroring
         ASCII, it will:
      64     1.  Launch the corresponding GL application binary (e.g.,
         `projects/gl-os/fuzz-op-gl/+x/fuzz-op_manager.+x`).
      65     2.  Manage its window, input, and lifecycle.
      66     3.  Ensure GL window state is isolated and controlled.
      67
      68 ### 3.2 Input Handling & Synchronization
      69 -   **Input Bridging:** GL windows will capture input events (keyboard, mouse,
         joystick) and relay them to the appropriate application process. The exact
         mechanism (e.g., piping, shared file) will be defined per application,
         prioritizing TPM's file-based IPC.
      70 -   **State Synchronization:** GL applications will read and write their state
         to files within their project directory, maintaining consistency.
      71
      72 ---
      73
      74 ## 4. FUTURE CONSIDERATIONS
      75
      76 -   **Inter-Theater Communication:** While GL-OS apps are sovereign, future
         enhancements may involve mechanisms for GL apps to "read" specific ASCII state
         files (e.g., global player stats) for display purposes, further enabling
         hybrid experiences.
      77 -   **Asset Pipeline:** Standardizing asset formats (tiles, sprites) for
         seamless integration across projects.
      78
      79 ---
      80
      81 ## 5. DOCUMENTATION & TERMINOLOGY
      82
      83 -   **File-State Mirroring:** GL-OS applications mirror the *state of data
         files* written by their respective projects. This is the primary mechanism for
         rendering content.
      84 -   **Process Sovereignty:** GL-OS and application processes are independent.
      85 -   **Exo-Sovereign Apps:** Applications like `fuzz-op-gl` and `op-ed-gl` are
         external to the core GL-OS manager but hosted by it.
      86
