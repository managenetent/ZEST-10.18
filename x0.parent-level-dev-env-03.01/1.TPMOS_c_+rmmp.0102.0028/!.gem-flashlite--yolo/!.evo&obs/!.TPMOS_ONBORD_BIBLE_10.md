## 1. Core Philosophy: The PMO Hierarchy
The True Piece Method (TPM) decomposes reality into atomic units. The hierarchy is absolute:
**PIECE (Soul) → MODULE (Brain) → OS (Theater)**

- **PIECE:** A directory containing `piece.pdl` (DNA) and `state.txt` (Mirror). It owns its state exclusively.
- **MODULE:** A background daemon (e.g., `_manager.c`) that orchestrates Pieces. It is "Thin" (delegates muscle to Ops).
- **OS / CHTPM:** The theater layer. Substitutes variables (`${var}`), routes input, and composes frames. It must not discover project Piece state or synthesize Piece methods.
- **Mantra:** "If it's not in a file, it's a lie." No hidden memory-only state.

### Projection Contract
Managers own the frame projection for their project. A project manager reads sovereign Piece state, resolves active capabilities through Ops, and writes a flat UI projection to `projects/<id>/manager/gui_state.txt`.

CHTPM consumes that projection. It may load `gui_state.txt`, substitute `${var}` values, manage focus/navigation, and inject keys, but it must stay out of recursive Piece lookup, PDL method extraction, and project-specific state interpretation.

Required manager-published fields for dynamic project screens:
- `module_path`
- `active_layout_id`
- `app_title`
- `active_target`
- `piece_methods`
- any layout-facing display variables such as `pet_name`, `hunger`, `game_time`, or `game_map`

## 3. Engineering & CPU Safety Standards
Failure to follow these patterns causes thermal throttling, orphaned processes, and system crashes.

- **The Fuzzpet Pattern:** NEVER use `system()`. ALWAYS use `fork()/exec()/waitpid()` for child processes.
- **Signal Handling:** Modules MUST register `SIGINT`/`SIGTERM` handlers and call `_exit()` to prevent zombies.
- **Process Groups:** Call `setpgid(0, 0)` in `main()` so the module and its Ops can be killed as a single unit.
- **Active Pulse Throttling:** 
    - **Active:** `usleep(16667)` (~60 FPS) when input is detected or layout is in focus.
    - **Idle:** `usleep(100000)` (10 FPS) when not active.
- **Marker-File Discipline (The "Pulse"):** Prefer marker files (e.g., `frame_changed.txt`) over size-only `stat()` polling. Content can change without size delta.
    - **Usage:** Append a byte/character (e.g., `M\n` or `P\n`) to the relevant marker to wake up the Parser or Renderer.
    - **Efficiency:** Use `stat()` on the *marker* file to detect growth before reading the main state/history files.
- **Path Mandate:** Resolve `project_root` from `pieces/locations/location_kvp` ONLY. Hardcoded paths are lies.
- **Header Convention:** No shared `.h` files. Every `.c` is a self-contained island. Use `asprintf` for all path construction.

## 4. UI/UX Standard Patterns
The CHTPM Parser provides native interactive primitives.

- **The Button Pattern:** Use `<button label="Text" onClick="KEY:1" />`. 
    - The parser AUTOMATICALLY adds `[>]` (focused) and `[ ]` (unfocused).
    - NEVER manually write nav indicators in layout text.
- **The CLI_IO Pattern:** Use `<cli_io id="my_input" label="${val}" />` for text input.
    - **Live Sync:** Parser writes `el->input_buffer` to `gui_state.txt` on EVERY keypress.
    - **Persistence:** Parser reads from `cli_buffers.txt`. ESC deactivates but never clears.
- **Folding ([+]/[-]):** Used for **Directory Traversal** and tree views.
    - Labels containing `[+]`/`[-]` use persistent `fold_ID` state variables.
    - Children stay in the global numbering list (1-99).
- **Activation (Submenus):** Used for **Scoped Task Menus** and localized settings.
    - Use `<button onClick="ACTIVATE" />` to enter a scoped view.
    - Children are hidden unless the root is active; numbering restarts (1., 2., ...) within the scope.
    - **Hierarchical BACK:** ESC or `onClick="BACK"` traverses up the menu tree, preserving focus on the root.
- **Switching (Layouts):** Used for **Major Mode Changes** (e.g., Shop vs. World).
    - Use `<button href="path/to.chtpm" />`.
    - Resets all navigation/focus state to the new layout's entry point.
- **Dynamic Methods:** Layouts render `${piece_methods}`. Managers or manager-called Ops build that value from Piece PDL and publish it through `gui_state.txt`; the parser only substitutes it.

- **Dynamic Colors:** Layouts may also render `${theme_*}` or other color variables.
  - The parser may resolve `fg` / `bg` attributes and apply them at render time.
  - Managers own the color intent and should publish color KVPs through `gui_state.txt`.
  - If multiple projects need the same color config, prefer a reusable Op that writes the KVPs.
  - Keep color logic generic in `chtpm_parser.c`; do not add project-specific color branches there.

## 6. The "Golden" Pitfalls
1. **#31 App Layout project_id Drift:** When loading an APP layout via `href`, explicitly set `project_id` to match the app, otherwise PDL lookups fail.
2. **#87 One Writer Rule:** `current_frame.txt` is owned by the Parser. Daemons MUST trigger renders by writing to `frame_changed.txt` marker, never by setting `dirty=1` or writing the frame directly.
3. **#80 ASCII Injection:** Parser maps `KEY:n` to ASCII `'0'+n` (e.g., 50 for '2'). Managers must subtract `'0'` to get the logical index.
4. **#20 Root/Path Drift:** Always resolve `project_root` at runtime. Never free a path string before its `fopen()` call is finished (Use-After-Free risk).
5. **#94 Live Submission:** For chat/CLI, do NOT deactivate focus on Enter. Sync on every pulse; keep the stack light (use heap for large arrays).
6. **#102 Project Type Pollution:** Never list a GL-only project in the ASCII Project Loader. Keep theaters (ASCII vs. GL) separate to prevent 404/rendering errors.
7. **#103 Parser Sovereignty Drift:** Do not move project-specific state discovery or dynamic method generation into `chtpm_parser.c`. Fix missing values in the manager projection or in a manager-called Op.
8. **#99 Illegal Command Prefix:** The Parser ONLY relays commands starting with `SET_`, `OP:`, or `MP3:`. Commands like `LOAD_FILE:` or `DO_ACTION` will be SILENTLY IGNORED. Always prefix project commands with `SET_` (e.g., `SET_LOAD_FILE:`).

## 9. Preventing Cross-Platform Compatibility Drift (Linux/Windows)
To maintain velocity without breaking the stable Linux foundation, adhere to these "Parity Wraps."

- **Surgical Platform Wraps:** NEVER refactor established POSIX logic to "clean it up" for Windows. Use `#ifndef _WIN32` for Linux/macOS and `#ifdef _WIN32` for Windows blocks.
- **Process Management Parity:**
    - **Linux:** `fork()` + `exec()` + `waitpid()` is the mandatory pattern for thread/process lifecycle.
    - **Windows:** Use `_spawnl()` with `_P_NOWAIT` (for shared console/input access) or `_P_DETACH` (for background daemons).
- **Main Loop Stability:**
    - Linux orchestrators block on `pthread_join(kb_t, NULL)`.
    - Windows orchestrators must use a persistent `while(!should_exit) { Sleep(100); }` loop because detached processes return immediately.
- **Path & IO Hygiene:**
    - **Separators:** Use forward slashes `/` for all internal logic and `fopen`. Only convert to backslashes `\\` for native Windows `_spawn` or `system()` calls.
    - **Functions:** Wrap `mkdir`, `chdir`, and `access` with their Windows equivalents (`_mkdir`, etc.) inside headers.
- **Font & Asset Robustness:** Hardcoded Linux paths (e.g., `/usr/share/fonts/`) will crash Windows renderers. Always provide a Windows fallback path (e.g., `C:/Windows/Fonts/`) or a graceful "no-asset" exit.
- **Signal Integrity:** Windows does not support process groups (`setpgid`). Ensure `handle_sigint` on Windows uses explicit `taskkill` or tracked PID cleanup to prevent orphaned "zombie" renderers.

## 10. Agent Execution Guardrails & Anti-Overengineering Mandate
You are an execution engine, not an architect. Protect the 1-man team velocity at all costs.

* Pattern Compliance: When tasked to build, use existing patterns found in the codebase.
* Invention Restraint: If an example does not exist in the current code, do not introduce it without explicit permission.
* Proactive Warning: If an optimization or implementation explicitly goes against established best practices or threatens to introduce bugs, warn the programmer directly before executing.
* Foundation Sovereignty: The existing legacy codebase is assumed to be near-perfect and TPMOS compliant. It is tried, tested, and true. Do not reinvent the wheel.
* YAGNI Strictness: Apply "You Aren't Gonna Need It" principles relentlessly. Treat tasks as scaffolding to scale organically, never brute-force features for theoretical future bottlenecks.
* Harden Tomorrow: Working code can be hardened tomorrow. We cannot build on a broken or unproven foundation.

## 11. Standalone Operational Architecture
**MANDATE: ALL functional logic MUST be decomposed into standalone, modular executables (Ops).**

- **NEVER** hardcode functional logic (e.g., API requests, complex file manipulations, network communication) directly into manager modules (`manager.+x`).
- **MUST** use `fork()` and `exec()` (or equivalent process management) to invoke functional Ops.
- **DEBUGGABILITY:** Monolithic hardcoded functions are un-testable and un-debuggable. Standalone Ops allow isolated testing, which is a core TPM requirement.
- **TPM PRINCIPLE:** Managers orchestrate; Ops execute. The boundary is absolute.
- **REUSE RULE:** If logic is shared across managers or projects, prefer a reusable Op first. Copy/paste is only a temporary bridge, not the final shape.
- **CONFIG RULE:** Shared projection data such as route registries, theme KVPs, and layout-facing color state should be generated by Ops when practical, not handwritten in multiple managers.

## 12. Op Testability
**MANDATE: Every Op MUST be independently testable via CLI.**

- **TESTABILITY:** Since functionality is extracted to standalone executables (Ops), you MUST verify each Op independently before integrating it into a Manager.
- **HOW TO TEST:**
    1.  **Prepare Input:** Create a sample input file (e.g., `state/test_input.json`).

    4.  **Verify Output:** Inspect the output file/stdout to confirm expected behavior.

## 13. Wraith Projects: Desktop-Embedded Architecture & ASCII/GL Mirroring

**Core Principle:** Wraith projects follow the EXACT SAME manager/ops/gui_state pattern as legacy TPMOS projects. The key difference is execution context: embedded in a desktop environment with two parallel rendering modes (ASCII terminal + GL semantic objects) instead of full-screen standalone.

### 13.1 Project Topology

**Location:** `projects/wraith-alpha/wraith-projects/<project_name>/`

**Structure (identical to legacy projects):**
```
<project>/
├── project.pdl                    (metadata: project_id, entry_layout, manager path)
├── manager/
│   ├── <project>_manager.c        (init-only: forks ops, publishes gui_state.txt)
│   ├── +x/<project>_manager.+x    (compiled binary)
│   └── gui_state.txt              (PUBLISHED by manager: variables for layout substitution)
├── ops/
│   ├── src/<project>_input.c      (hot-path ops: runs on every keypress, mutates state)
│   ├── src/+x/<project>_input.+x  (compiled)
│   └── +x/<project>_input.+x      (symlink/copy to runtime location)
├── layouts/
│   └── <project>.chtpm            (layout template: references ${var} from gui_state.txt)
├── session/
│   ├── wraith_body.txt            (the RENDERED OUTPUT of the layout — see 13.3, same role as a legacy render-op artifact)
│   ├── state.txt                  (wraith-specific: scroll_offset, project_id, etc.)
│   ├── scene.objects.pdl          (GL semantic objects: tag=panel/text/control with x/y/w/h/nav)
│   └── history.txt                (input marker for <interact> tag)
└── pieces/
    └── <data files>               (project-owned content: documents, assets, etc.)
```

### 13.2 Manager/Ops Pattern (Identical to Legacy)

**Manager (`_manager.c`):**
- One-time init only. No persistent polling, no per-keypress state mutation in manager.
- Job: `resolve_paths()`, `load_initial_state()`, fork/exec ops once via `trigger_initial_render()`, then exit/idle.

**Ops (`wraith_project_input.c`):**
- Synchronous hot-path. Wraith invokes this via fork/exec/waitpid on EVERY keypress, BEFORE trigger_render().
- Job: read input from `session/history.txt`, mutate project state, write VALUES to `manager/gui_state.txt`, exit.
- ALL per-frame state mutation happens here, not in the manager.
- **As of 2026-07-11, do NOT hand-render layout structure (box framing, `<button>` tags) directly into `session/wraith_body.txt` in ops code — this applies to every NEW project, full stop, not just ones that happen to meet some condition.** For ASCII mode this is now handled automatically by the shared desktop code (`wraith-alpha_manager.c`'s `render_project_layout_body()`), which reads the project's own `layouts/<name>.chtpm` and substitutes `${var}`s from `gui_state.txt` directly — the same division of labor agy-text-editor's manager uses. Ops only needs to compute and publish values. **The "IF it has `gui_state.txt` and `is_map_control` isn't 1" language in 13.3 describes which EXISTING projects have been migrated so far — it is not a design choice for a new project to make.** Every new project should be built with a `manager/gui_state.txt` and no `is_map_control` gate from day one, so it's automatically on the correct, unified path rather than needing a later migration. GL mode still requires hand-authored `scene.objects.pdl` for now (13.3's GL gap) — that's the one piece not yet unified, not a reason to fall back to hand-rendering ASCII too.

### 13.3 `wraith_body.txt` IS the Rendered Layout — Same Pipeline as Legacy TPM

**Do not think of Wraith rendering as "two separate paths." It is the same pipeline as any TPM project** (`^.chtpm-faq/CHTPM_TPM_DOX+1.md`'s "Runtime pipeline": *input → module → ops update state → render op produces artifact → parser composes final frame*). The only Wraith-specific detail is **who performs the render/substitution step**:

- In a plain (non-Wraith-hosted) TPM project, `chtpm_parser.c` itself opens `layouts/<project>.chtpm` and substitutes `${var}` placeholders from `manager/gui_state.txt` at render time — the parser is both renderer and composer.
- **As of 2026-07-11, ASCII mode has a real, generic version of this for Wraith-hosted embedded windows too**: `wraith-alpha_manager.c`'s `render_project_layout_body()` opens the project's own `layouts/<entry_layout>.chtpm` directly, substitutes `${var}`s from `manager/gui_state.txt` (via `read_gui_state_value()` + `substitute_layout_line()`), and uses that as the window body — no project-side hand-rendering needed.
- **This is opt-in TODAY — that is a transitional state, not the intended permanent architecture. Do not design new projects around "opt in or don't."** The end goal (see `j11.wraith-foundation-fix-fut.txt`, parent dev-env directory) is every Wraith-hosted project working this way unconditionally, the same way every legacy TPM project unconditionally gets parser-driven rendering with no per-project special-casing — no `gui_state.txt`-existence check, no `is_map_control` gate, no per-project hand-authored `scene.objects.pdl`, no hardcoded nav literals (see 13.5's note and Pitfall list). The two conditions below describe where the migration currently stands, not a design a new project should deliberately choose to opt out of:
  1. The project has a `manager/gui_state.txt` file (most don't yet — as of this writing, only `wrai-text-editor` and `settings` do, plus `piececraft-wraith`, which is excluded by condition 2 below).
  2. `session/state.txt`'s `is_map_control` is NOT `1`. Projects with `is_map_control=1` (piececraft-wraith) keep using the older raw-`wraith_body.txt`-passthrough path (`append_project_probe_body()`) untouched — their own layout's `<button>`/`<text>` markup is intentionally not parsed this way (see that project's own `layouts/piececraft-wraith.chtpm` comment).
  If neither condition is met, the OLDER pattern still applies: the project's own ops must hand-write `<`-prefixed `<text>`/`<button>` markup, values already substituted, directly into `session/wraith_body.txt` (still true for `piececraft-wraith`, and for any project without `gui_state.txt`).
- **GL mode was NOT updated to match — this is a known, currently-live gap, not a hypothetical.** `render_project_layout_body()` only feeds the ASCII composer. GL-mode embedded-window rendering (`emit_embedded_line_objects()` and the scene-generation code around it) still reads `session/wraith_body.txt` directly, unchanged. For any project using the new opt-in path (currently `wrai-text-editor`), ASCII and GL can show DIFFERENT content until this is closed — see 13.6.
- Treat `layouts/<project>.chtpm` as the **authoritative structural spec** — box framing, `<button>` placement, nav order — for what the project should look like, regardless of which of the two mechanisms above actually renders it.
- Confirmed via `wraith-alpha_manager.c` (~line 1645): a `wraith_body.txt` line whose first character is `<` is appended **verbatim** into the composed frame and parsed downstream as real `<text>`/`<button>`/`<cli_io>` markup — genuine, nav-indexed, clickable elements. A line NOT starting with `<` is wrapped in a fixed plain-text border instead (inert — never becomes a button, see Pitfall #26). So: **write the same `<text>`/`<button>` tags your layout file would contain, with real values already substituted in, as `<`-prefixed lines in `wraith_body.txt`.**
- Confirmed real example: `settings_manager.c`'s own header comment states settings is "a real, standalone project... [that] needs zero changes to any shared file to get `${settings_menu_markup}` substituted into `settings.chtpm`" via the generic href/`chtpm_parser.c` path — and separately, its `session/wraith_body.txt` contains hand-written `<button onClick="KEY:5" />`/`<cli_io input_mode="numeric" />` lines that mirror that same substituted content, so the identical menu renders correctly whether settings is opened embedded (desktop launcher) or standalone (`href`). Same information, same structure, two renderers (one generic/parser-driven, one project-authored).
- A project can also link from its embedded `wraith_body.txt` to its own full standalone layout via a normal `href="...layout.chtpm"` button (any `route_command()` action containing `.chtpm` triggers `request_layout_change()`, `wraith-alpha_manager.c` ~line 4184) for a richer full-screen view — this is the documented embedded/standalone duality (section 4 of `WRAITH-ARCHITECTURE-AND-ROADMAP.md`), not a competing rendering model.

### 13.4 ASCII/GL Mirroring (The "Theater" Duality)

**Core Concept:** One project state expressed in two rendering modes simultaneously.

**ASCII Mode:**
1. Ops renders the project's layout structure (13.3) into `session/wraith_body.txt`, values already substituted.
2. `wraith-alpha_manager.c` composes `session/wraith_body.txt` into the embedded window's `current_frame.txt` region (raw passthrough for `<`-prefixed lines, bordered plain text otherwise).
3. If reached standalone (`href`/`LOAD_PROJECT:`), `chtpm_parser.c` instead performs the substitution itself directly against `layouts/<project>.chtpm` and composes full-screen — same rendered content, generic renderer.

**GL Mode (OpenGL semantic rendering):**
1. wraith_rgb_daemon.c reads `project/session/scene.objects.pdl` (semantic object definitions).
2. Renders OBJECT tags as 3D/2D GL shapes:
   - `tag=panel` → filled rectangle with border.
   - `tag=text` → text label at (x, y) with colors.
   - `tag=control` → button with nav index and action.
   - `tag=img/video/audio` → media asset preview.
3. Outputs `current_frame.rgba32` (raw RGBA32 framebuffer) + `current_frame.png` (human-inspectable).

**Synchronization Rule:**
- Both modes read the SAME `gui_state.txt` and display the SAME logical state.
- ASCII mode composition happens in the parser (text substitution + character grid).
- GL mode composition happens in wraith_rgb_daemon.c (semantic object rendering).
- Changes to state → both modes must update consistently (managed by ops writing to gui_state.txt + scene.objects.pdl).

**Key Difference from Legacy:**
- Legacy projects: single rendering pipeline (full-screen layout).
- Wraith projects: dual rendering pipelines (ASCII terminal + GL objects), both fed by same state files.

### 13.5 Setting Up a New Wraith Project

**Checklist:**
1. **Create directory:** `projects/wraith-alpha/wraith-projects/<name>/`
2. **Write `project.pdl`:** Include META lines for manager, entry_layout, project_id.
3. **Create manager:** `manager/<name>_manager.c` (init-only pattern) + compile to `manager/+x/<name>_manager.+x`.
4. **Create ops:** `ops/src/wraith_project_input.c` (hot-path pattern) → compile to `ops/src/+x/` → copy to `ops/+x/`.
5. **Write layout:** `layouts/<name>.chtpm` with `<module>`, `<interact>`, and `${var}` placeholders.
6. **Initialize session files:**
   - `session/wraith_body.txt` — the rendered embedded-window body (see 13.3 — NOT a separate plain-text summary; contradicts nothing else in this doc as of the 2026-07-11 fix). If opting into the new `render_project_layout_body()` path (`gui_state.txt` present, `is_map_control` not 1), this file only needs a minimal fallback (e.g. one status line) since it's no longer the primary render source for ASCII. If NOT opting in, this file must contain the full hand-rendered `<`-prefixed markup per 13.3's older pattern.
   - `session/state.txt` — project-specific state (scroll_offset, project_id, `is_map_control`, etc.) — **note `is_map_control` here is what gates 13.3's opt-in behavior; set deliberately, don't leave unset by accident.**
   - `session/scene.objects.pdl` — GL semantic object definitions (OBJECT lines with x/y/w/h/nav/action). Still hand-authored regardless of the ASCII-mode opt-in above (GL was not updated — see 13.3's GL gap note and 13.6).
   - `session/history.txt` — empty marker file for input tracking.
7. **Manager publishes gui_state.txt:** With required fields (module_path, active_layout_id, app_title, project_id) + custom vars for layout. **This alone is enough to opt into the new ASCII render path (13.3) — no `session/wraith_body.txt` hand-rendering needed for ASCII if this file exists and `is_map_control` isn't 1.**
8. **Ops updates on keypress:** Reads session/history.txt (input), mutates state, rewrites manager/gui_state.txt and scene.objects.pdl (GL still needs scene.objects.pdl maintained by hand regardless).
9. **Compile via build system:** `./button.sh compile` automatically finds nested projects under wraith-projects/ and compiles manager/ops.

### 13.6 ASCII/GL Divergence: When They're Different (and Why)

**By Design, Both Modes Should Mirror:**
- ASCII and GL rendering the SAME information from the SAME state files.
- Differences = bugs (usually stale scene.objects.pdl or unsynchronized gui_state.txt updates).

**Known Divergence Sources (watch for these):**
- Manager updates gui_state.txt but forgets to update scene.objects.pdl → GL falls behind.
- scene.objects.pdl has stale x/y/w/h arithmetic → buttons misaligned in GL.
- Parser re-orders nav indices differently than scene.objects.pdl nav= assignments → nav confusion across modes.
- One mode receives input (e.g., keypress routed to ASCII only) while the other doesn't update → frame desync.
- **RESOLVED 2026-07-11**: the `render_project_layout_body()`/GL-parity gap noted below used to be flagged as an open structural divergence needing GL to be "extended." Research that day (`WRAITH_RGB_ARCHITECTURE.md`) found this framing was wrong: ASCII and GL are not meant to become one shared renderer. The proven, working pattern (chrome icons originally, then applied to the launcher row, scene-object nav, and cli_io the same day) is narrower: **one authoritative count or source of truth lives in `wraith-alpha_manager.c`, and both the ASCII markup builder and the GL object builder read that exact same source — not because they're unified into one function, but because there's only one place left that could disagree with itself.** `g_chrome_icons[]`/`CHROME_CONTENT_START`, `pieces/display/chrome_reserved_nav_count.txt`, `gui_state.txt`, and `next_scene_nav` are all live examples. Apply this same pattern to any *new* divergence found — do not treat "GL needs its own copy of the ASCII substitution logic" as the goal.
- **RESOLVED 2026-07-11 (cli_io)**: `emit_embedded_line_objects()`'s live-value display and nav-slot assignment used to be gated on `is_cli_io` specifically. Generalized to `has_target_id` (any tag declaring `target_id`, not just `<cli_io>`) — any future element type wanting a live, persisted value gets this for free now.
- **Believed fixed as of 2026-07-11, unconfirmed root cause**: clicking a nav-indexed button (`onClick="KEY:n"` / GL `action=KEY:n"`) used to sometimes behave differently than pressing the equivalent raw key directly, across multiple projects. User testing after this session's GL `route_input()`/nav-numbering fixes reports this no longer reproduces, but the exact mechanism was never isolated — if it resurfaces, check `action_for_nav_index()`/`hit_test_semantic_action()` in `wraith_gl.c` first (never actually checked). See `x/3d-fix.txt` for the original discovery.

**Prevention:**
- Audit BOTH `gui_state.txt` AND `scene.objects.pdl` changes in the same commit.
- Ops should write BOTH files atomically (temp-file + rename).
- Test both ASCII and GL modes during development; don't assume one mode fixes the other.
- **When adding any new interactive/numbered element type**: find (or create) the ONE place that determines its identity/count/position, and make BOTH the ASCII and GL builders read that same source — never let either side compute its own independent guess. This is the standing rule, not a one-time fix; every future element type is a fresh chance to violate it.

### 13.7 GL Receipts: Headless Auditing Without Pixels

**What are receipts?**
wraith_rgb_daemon.c writes audit-trail files (receipts) after every GL frame, enabling inspection of what was rendered WITHOUT looking at pixel data. These are machine-readable snapshots of the semantic scene.

**Receipt Files (live in `projects/wraith-alpha/session/`):**
- `current_frame.objects.pdl` — list of all OBJECT lines rendered in this frame (tag, x, y, w, h, nav, action, label).
- `current_frame.desktop_state.pdl` — desktop window state (active_window_slot, focused project, window positions).
- `current_frame.focus_state.pdl` — keyboard/mouse focus information.
- `current_frame.cells.pdl` — character cell grid (for ASCII→GL coordinate verification).
- `current_frame.hitmap.pdl` — click/hit detection map (x, y, object_id for click testing).
- `current_frame.audit.txt` — human-readable audit log (frame timestamp, object count, render stats).
- `current_frame.meta.pdl` — metadata (viewport dimensions, camera state, Z-order).

**Why Receipts Matter:**
- **Headless Testing:** Verify GL rendering correctness without a display (CI/CD safe).
- **Debugging:** Read scene.objects.pdl to verify your Wraith project's semantic objects are positioned/labeled/nav-indexed correctly.
- **Divergence Diagnosis:** Compare ASCII `current_frame.txt` (parser output) vs. GL `current_frame.objects.pdl` to spot render mismatches.
- **Frame-by-Frame Audit:** Each receipt is timestamped; inspect frames at specific moments (e.g., right after a keypress).

**How to Use Receipts in Development:**
1. Launch Wraith in GL mode.
2. Interact with your project (click buttons, enter input).
3. Read `projects/wraith-alpha/session/current_frame.objects.pdl` — see exactly what GL rendered.
4. Verify:
   - Objects have correct x/y/w/h (within desktop window bounds).
   - Nav indices (nav=1, nav=2, ...) match button order.
   - Action fields (onClick, href) are correct.
   - Labels match expected text (e.g., document content, button labels).
   - Z-order makes sense (overlays on top, panels behind).
5. If diverged from ASCII mode:
   - Compare `pieces/display/current_frame.txt` (ASCII) vs. receipts (GL).
   - Spot the difference in positioning, nav, or content.
   - Fix gui_state.txt or scene.objects.pdl accordingly.

**Receipts Are NOT Live:**
- Receipts are written AFTER rendering completes (low priority, no frame-time cost).
- They may lag by 1–2 frames; don't rely on them for real-time debugging (use live inspection tools for that).
- A receipt lag of > 10 frames suggests wraith_rgb_daemon.c is stalled or crashing (check process status).

**Template Checklist for New Projects:**
- [ ] Layout uses `<button>` elements (not raw text with `[>]` indicators)
