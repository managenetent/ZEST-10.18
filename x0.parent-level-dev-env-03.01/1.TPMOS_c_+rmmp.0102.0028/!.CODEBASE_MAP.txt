DECISION TREE: "Where does this doc go?"
────────────────────────────────────────
1. Is it a TECHNICAL SPEC for a system component?
   → YES: #.main/
   → NO: Continue

2. Is it for PEOPLE learning/using the system?
   → YES: #.docs/
   → NO: Continue

3. Is it about FUTURE plans/roadmap?
   → YES: #.plans/
   → NO: Continue

4. Is it a SCRIPT for building/testing/maintenance?
   → YES: #.tools/
   → NO: Continue

5. Is it a TEST plan or result?
   → YES: #.testing/
   → NO: Where did this come from?!

================================================================================
QUICK START: First-Time User Walkthrough
================================================================================

STEP 1: Compile Everything
──────────────────────────
./button.sh c

This compiles:
- All projects in pieces/os/compiled_projects.txt
- System apps (man-*, player_app, playrm)
- System components (orchestrator, parser, renderer, keyboard)
- GL-OS (gl_desktop, gl_os_session, gl_os_renderer)

STEP 2: Run the System
──────────────────────
./button.sh r

You'll see the OS menu. Navigate with arrow keys, select with Enter.

STEP 3: Try GL-OS (OpenGL Desktop)
──────────────────────────────────
From main menu:
1. Click "GL-OS" button
2. OpenGL window opens (requires DISPLAY server)
3. ASCII layout shows desktop view in terminal

STEP 4: Try op-ed (Text Editor)
────────────────────────────────
From loader menu:
1. Select "op-ed"
2. Use arrow keys to move cursor
3. Select glyphs with number keys
4. Place tiles on map

STEP 5: Try a Test Project
──────────────────────────
From loader menu:
1. Select "test_fondu"
2. Press 1, 2, 3, 4 to test counter ops
3. Press ESC to exit

STEP 6: Install a Project (Fondu)
─────────────────────────────────
In terminal (after exiting with ESC):
./pieces/system/fondu/fondu.+x --install test_fondu

Now test_fondu is "production" (read-only, in pieces/apps/installed/)

STEP 7: Discover Available Ops
──────────────────────────────
./pieces/system/fondu/fondu.+x --list-ops

Shows all ops from all installed projects.

STEP 8: Use User Ops in PAL Script
──────────────────────────────────
Create a PAL script in your project:

  # my_game/scripts/login.asm
  OP user::create_profile "hero"
  OP user::auth_user "hero"
  OP user::get_session "hero"

Run it with prisc or from your game manager.

================================================================================
KEY CONCEPTS (New to TPMOS?)
================================================================================

WHAT IS A PROJECT?
──────────────────
A project is a self-contained game/app with:
- project.pdl (descriptor with project_id, entry_layout)
- manager/+x/ (orchestrates logic)
- ops/+x/ (optional: reusable "muscle" binaries)
- layouts/ (UI definitions)
- pieces/ (game state, profiles, etc.)

WHAT IS AN APP?
───────────────
An APP is a system tool with:
- pieces/apps/<app_id>/ structure
- LAUNCH:<APP_NAME> command
- Listed in pieces/os/app_list.txt
- Examples: GL-OS, man-ops, man-pal, man-add

WHAT IS AN OP?
──────────────
An op is a standalone binary that does ONE thing well:
- place_tile.+x (places a tile on a map)
- create_profile.+x (creates a user profile)
- auth_user.+x (authenticates a user)

Ops are called from:
1. C code (fork/exec/waitpid pattern)
2. PAL scripts (OP op_name args)

WHAT IS PAL?
────────────
PAL = Prisc Assembly Language. Simple scripting language:

  OP user::create_profile "player1"
  OP user::auth_user "player1"
  JUMP game_start
  LABEL game_start
  OP move_entity "player" 1 0

WHAT IS FONDU?
──────────────
Fondu manages project lifecycle:
- ACTIVE (projects/) → Editable, in compile
- INSTALLED (pieces/apps/installed/) → Read-only, production
- ARCHIVED (projects/trunk/) → Source only, not compiled

Use Fondu to:
- Deploy projects as "apps"
- Archive projects you're not working on
- Discover what ops are available

================================================================================
FILE FLOW: How Input Becomes Action
================================================================================

1. USER PRESSES KEY
   ↓
2. keyboard_input.+x writes to pieces/keyboard/history.txt
   ↓
3. chtpm_parser.+x reads keyboard/history.txt
   ↓
4. Parser injects into active project's buffer (player_app/history.txt)
   ↓
5. Project manager polls buffer, sees key
   ↓
6. Manager calls op (e.g., move_entity.+x)
   ↓
7. Op updates state (projects/your_game/pieces/state.txt)
   ↓
8. Renderer reads state, writes view.txt
   ↓
9. Parser composes frame, writes current_frame.txt
   ↓
10. Display prints to screen

================================================================================
TROUBLESHOOTING
================================================================================

Terminal locked after Ctrl+C?
  → Run: reset
  → Or: ./.tools/fix_terminal.sh

Project not compiling?
  → Check: pieces/os/compiled_projects.txt includes it
  → Run: ./button.sh c and watch for errors

Ops not showing in --list-ops?
  → Check: projects/<id>/ops/ops_manifest.txt exists
  → Run: ./fondu --install <project>
  [fondue is incomplete , references to it are speculative]

GL-OS shows wrong content?
  → Check: pieces/apps/gl_os/session/view.txt has DESKTOP_VIEW
  → Check: parser get_var() reads desktop_view from view.txt
  → See: #.main/gl-os-layout-fix.txt

op-ed xlector moves randomly?
  → Check: op-ed_manager.c has stat() check for history
  → See: #.main/op-ed-history-fix.txt

Joystick not working?
  → Check: #.testing/INVESTIGATION-JOYSTICK-LINUX.txt
  → Check: pieces/joystick/plugins/+x/joystick_input.+x exists
  → Check: pieces/joystick/ledger.txt has events

Can't find where something is?
  → Use this map! Look for the "store" that matches your need.
  → Check PITFALLS first: #.docs/05-PITFALLS/^.pmo.ld-faq+8/PITFALLS_ACTIVE_*.txt
