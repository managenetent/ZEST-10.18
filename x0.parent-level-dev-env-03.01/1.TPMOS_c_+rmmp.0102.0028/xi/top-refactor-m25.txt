# Topology Refactor Plan: M25 (ABCD Topology)

## 1. Analysis of Current `chat-op` Topology
The current `chat-op` implementation relies on a central manager (`chat-op_manager.c`) that delegates network operations to `p2p_socket_daemon.+x`. 

**Strengths:**
- Functional multi-peer (3+) connectivity for real-time messaging.
- Decouples UI/Manager logic from raw network IO.
- Maintains a clean separation between input state (`gui_state.txt`) and message logs (`chat_log.txt`).

**Weaknesses:**
- **Implicit Topology:** The networking model relies on the daemon, but the topology is not explicitly ring-based or auditable in a way that scales predictably.
- **History/Delivery:** There is no mechanism to propagate message history between peers or ensure reliable delivery in a long list, as it's not a structured ring.

## 2. Desired ABCD Topology Goals
- **Ring Structure:** Traffic flows A -> B -> C -> D... and back.
- **Auditability:** Broadcasting is replaced by sequential pass-through, reducing network overhead and increasing message traceability.
- **Dynamic List Handling:** The ring must handle list traversal (passing the list *through* the messages rather than as a separate broadcast).

## 3. Transition Plan
As requested, we will treat `chat-op` as the stable reference/backup and migrate the topology to a new workspace for `p2p-net`.

1.  **Backup/Clone:** Overwrite/initialize `projects/p2p-net` with the functional core of `projects/chat-op`.
2.  **Topology Refactor:**
    - Modify the networking daemon within `p2p-net` to implement the ring-passing logic.
    - Implement the "backwards once, forwards once" traversal.
    - Integrate list-passing within the message structure (blockchain-style).
3.  **Validation:** Test 3+ peer connectivity to ensure the ring closes and audits correctly.
