
# GL-OS Update Preparation: Walkthrough of Current Progress

This document provides an indexed overview of the current state of GL-OS development, focusing on its capabilities as a host environment for independent GL-native applications and the implementation of virtual window launches.

## I. Introduction

*   **Purpose:** To document the progress made in establishing GL-OS as a functional desktop environment capable of hosting and managing separate GL applications.
*   **Context:** This phase represents a significant architectural shift, moving from mirroring ASCII processes to GL-OS acting as a sophisticated display manager and input router for self-contained GL projects.

## II. Core Architectural Principles

This section outlines the foundational design decisions guiding GL-OS development, as detailed in `gl-os_vision.md`.

*   **A. GL-OS as Host Environment:**
    *   GL-OS is designed as a high-fidelity, modular desktop environment.
    *   It manages windows, input events, and the rendering pipeline for GL-native applications.
    *   It does NOT directly render ASCII processes or mirror their internal memory states.
*   **B. File-State Synchronization:**
    *   GL-OS applications within the host environment interact with **project-specific data files**.
    *   Examples of these files include `current_frame.txt`, `state.txt`, `view.txt`, and tile definitions, located within their respective project directories (e.g., `projects/gl-os/fuzz-op-gl/`).
    *   This adheres to the TPMOS principle that **file states are the source of truth**, ensuring data sovereignty.
*   **C. Process Sovereignty:**
    *   GL-OS and any ASCII TPMOS applications remain entirely **independent processes**.
    *   Communication and state synchronization between them are mediated exclusively through file I/O.

## III. Project Launch Mechanism ("Virtual Windows")

This section details how GL-OS launches and manages independent GL applications.

*   **A. The "Virtual Window" Concept:**
    *   As clarified in `#`.gl-2do.prompt.txt, a "virtual window" is a GL-OS window that launches and displays a standalone GL application, reflecting its own project's file state. This is distinct from opening a new, separate GL window or mirroring an ASCII process.
*   **B. Role of `gl_desktop.c`:**
    *   The GL-OS desktop manager, primarily implemented in `gl_desktop.c`, intercepts selections for GL projects from menus (e.g., the ASCII Project Loader menu `playrm/loader.chtpm`).
    *   Instead of simply displaying text or mirroring ASCII, it handles the launch sequence.
*   **C. Launching GL Application Binaries:**
    *   When a GL project is selected, `gl_desktop.c` launches the corresponding GL application binary (e.g., `projects/gl-os/fuzz-op-gl/+x/fuzz-op_manager.+x`).
*   **D. Window Management:**
    *   GL-OS takes responsibility for managing the lifecycle of these launched GL application windows, including their input handling and overall state.
