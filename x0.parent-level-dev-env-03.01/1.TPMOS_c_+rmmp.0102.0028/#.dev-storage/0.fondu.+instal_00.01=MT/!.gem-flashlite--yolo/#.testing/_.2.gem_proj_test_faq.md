# AI Agent Protocol: Project Functionality Testing

## Version
1.0

## Date
2026-04-21

## Objective
To systematically test and report on the functionality state of every project within the TPMOS architecture, ensuring a consistent and comprehensive evaluation.

## Starting Point

1.  **List Available Projects:**
    *   Execute `list_directory projects/` to obtain a list of all project directories.
2.  **Environment Setup:**
    *   Ensure the core CHTPM system is running (`./button.sh run`).
    *   Verify binaries using `./button.sh check`.
    *   If encountering persistent issues or prior to testing a new batch of projects, consider a clean restart via `./button.sh kill` followed by `./button.sh run`.
3.  **Project Iteration:** Process each project directory found in step 1.

## Testing Protocol for Each Project

For each identified project:

1.  **Project Identification & Launch Mechanism:**
    *   Examine the project directory (e.g., `projects/<project_name>/`).
    *   Look for project-specific launch scripts (e.g., `run.sh`, `start.sh`, manager executables within a `manager/` directory).
    *   Inspect `project.pdl` for launch configurations or metadata.
    *   If no direct launch mechanism is found, attempt to launch via GL-OS desktop if accessible (`./button.sh debug`).
    *   *If project has a `project.pdl`, read its content to understand project type and parameters.*

2.  **Attempt Project Launch:**
    *   Execute the identified launch mechanism.
    *   If successful, note the interaction method (CLI menu, GL-OS, etc.).

3.  **Core Functionality Testing (Black-Box Runtime):**
    *   **Menu Navigation (if applicable):**
        *   Use key injection (following the `[TIMESTAMP] KEY_PRESSED: <code>` format) to navigate the project's menu.
        *   Test common navigation keys (e.g., arrows, Enter, Esc) and project-specific menu options.
        *   Monitor frame updates in `session_frame_history.txt` for expected changes.
    *   **Core Feature Interaction:**
        *   Identify and test the primary features of the project based on its name or known purpose (e.g., for `chat-app`, test sending/receiving messages; for `editor`, test basic editing operations).
        *   Use key injection or other identified input methods.
        *   Observe frame updates and any console output for success, errors, or unexpected behavior.
    *   **State Persistence (if applicable):**
        *   If the project supports saving and loading (e.g., `create->save->load` cycle), perform this test.
        *   Verify data integrity after loading.

4.  **Error Detection & Stability:**
    *   Observe for crashes, unhandled exceptions, or significant error messages.
    *   Note any unexpected behavior or deviations from expected functionality.

5.  **Resource Usage (Optional/Advanced):**
    *   Monitor CPU/memory usage if tools are available and relevant to project type.

## Reporting Format

For each project, generate a report with the following structure:

```
Project: <project_name>

--- Status ---
[LAUNCHABLE / UNLAUNCHABLE / LAUNCH_FAILED]

--- Launch Details ---
Launch Method: <e.g., './projects/<project>/run.sh', './button.sh debug'>
Observed Interface: <e.g., CLI Menu, GL-OS, Direct CLI Output>
Initial Errors on Launch: <List any errors, e.g., "Missing cleanup files for editor">

--- Core Functionality ---
[Describe key features tested and their observed state]
- Feature 1: <Status: PASS/FAIL/PARTIAL/NOT TESTED>
  - Notes: <e.g., "Text injection worked, frame updated. Save/Load failed.">
- Feature 2: <Status: PASS/FAIL/PARTIAL/NOT TESTED>
  - Notes: <e.g., "Navigation via Enter key successful.">
...

--- Stability ---
Crashes observed: [YES / NO]
Critical Errors: [List critical errors or "None"]

--- Key Learnings & Issues ---
- [Describe any specific challenges encountered, e.g., "Key injection format for 'Enter' is '13'", "Project 'xyz' failed to launch due to missing dependency."]
- [Note any deviations from expected behavior or documentation.]

--- Stuck Points & Resolution Attempts ---
[Describe any points where the agent got stuck, what actions were taken to resolve, and the outcome.]
```

## Error Handling and Stuck Points

If the agent encounters a situation where it cannot proceed with testing a specific project:

1.  **Log the Failure:** Record the project name, the exact step where the blockage occurred, and all observed error messages or symptoms.
2.  **Attempt Basic Recovery:**
    *   If the CHTPM system is running but unresponsive, try restarting it using `./button.sh kill` followed by `./button.sh run`.
    *   Check the `Known Testing Pitfalls` section in the testing guide for common issues (e.g., pathing, context sensitivity, injection format).
3.  **Document the Blockage:** Clearly state that the project could not be tested beyond a certain point. Detail the error, the recovery attempts made, and the final state. Do **not** enter an infinite loop trying to resolve a single issue.
4.  **Proceed to the Next Project:** Move to the next project in the list to ensure comprehensive coverage.
