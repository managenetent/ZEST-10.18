✦ Yes, that's an excellent point and directly relates to how processes maintain
  their context. The user's button.sh script failing to find its files indicates
  it was running in an unexpected directory.

  The orchestrator.c does perform a chdir("../..") within its
  execute_system_command function. This is done in the child process right
  before execl is called. This ensures that the button.sh script, when executed,
  has the correct project root as its current working directory.

  However, the keyboard_muscle (which is also a child process launched by
  orchestrator.c) was not explicitly changing its directory. It was relying on
  relative paths (../../pieces/keyboard/history.txt) from wherever the
  orchestrator launched it.

  To address this and ensure consistent path resolution:

   1. Modify orchestrator.c: Pass the target_dir (which is the project root) to
      the bot5_keyboard_muscle when launching it.
   2. Modify keyboard_muscle.c:
       * Accept the target_dir as a command-line argument.
       * Remove the hardcoded ../../ paths.
       * Construct all file paths using the provided target_dir as the base.
       * Crucially, chdir to the target_dir at the beginning of its main
         function. This sets its working directory correctly, so relative paths
         like pieces/keyboard/history.txt and xo/bot5/debug/muscle_log.txt will
         resolve properly from the project root.
