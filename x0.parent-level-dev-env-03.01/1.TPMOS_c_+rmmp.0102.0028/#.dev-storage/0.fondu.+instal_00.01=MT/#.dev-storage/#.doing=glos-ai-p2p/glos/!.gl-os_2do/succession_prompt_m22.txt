# SUCCESSION HANDOVER - GL-OS PROJECT (M22/M23)
**Date:** March 24, 2026
**Current Focus:** GL-OS Desktop & Terminal Integration with TPM-Compliant Modularity

## 1. ARCHITECTURE SUMMARY

The project is a C-based Desktop Environment (`gl_desktop.c`) running inside an OpenGL/GLUT window.

**TPM Compliance:**
- GL-OS is a sovereign Piece with isolated session state
- Session files: `pieces/apps/gl_os/session/` (state.txt, view.txt, history.txt, etc.)
- **Ledger:** `pieces/apps/gl_os/ledger/master_ledger.txt` (ISOLATED - NOT global ledger)
- All window events logged to GL-OS ledger only

**Key File:** `pieces/apps/gl_os/plugins/gl_desktop.c`

**Architecture Principle:**
- GL-OS apps are contained within `pieces/apps/gl_os/apps/<app>/`
- Each app has its own module binary (`plugins/+x/<app>_module.+x`)
- Apps communicate via session files (TPM Mirror pattern)
- Similar to CHTPM modularity: `<module>` tag → separate `+x` binary

## 2. COMPLETED FEATURES

- **Window Management:** Draggable, resizable, and focusable (Z-order) windows.
- **Taskbar/Toolbar:** Bottom bar showing "User: guest", a real-time clock, and minimized apps.
- **Window States:** Windows can be minimized to the toolbar and restored by clicking.
- **3D Rendering:** Integrated "Cube App" rendering a rotating 3D cube inside a sub-viewport (scissor-tested).
- **Virtual Terminal:**
    - Text input with backspace and enter.
    - Scrollable command history (12 lines).
    - Basic command interpreter implemented (`process_terminal_command`).
    - `help` command implemented inside the virtual terminal.

## 3. CRITICAL DISTINCTIONS (FOR THE SUCCESSOR)

### 3.1 Virtual Terminal ≠ Linux Terminal

The GL-OS Virtual Terminal is a **window inside GL-OS**, NOT a Linux terminal emulator.

**What it IS:**
- OpenGL-rendered text window inside GL-OS desktop
- Commands interpreted by `process_terminal_command()` in gl_desktop.c
- Uses GL-OS app modules (separate `+x` binaries)
- TPM-compliant: all state in session files

**What it is NOT:**
- NOT xterm, gnome-terminal, or bash
- Does NOT run Linux system commands (`/bin/ls`, `/usr/bin/top`)
- Does NOT write to global ledger

### 3.2 Input Focus Matters

**When terminal window is focused (active_window_id points to terminal):**
- Pressing '1' → types character '1' in input buffer
- Pressing '2' → types character '2' in input buffer
- All keys 32-126 → text input

**When NO window is focused (active_window_id == -1):**
- Pressing '1' → creates new terminal window (shortcut)
- Pressing '2' → creates new cube window (shortcut)

**COMMON BUG TO AVOID:**
Don't make '1' always create a terminal - it should be text input when terminal is focused!

### 3.3 GL-OS Apps vs CHTPM Apps

**CHTPM Apps (Reference: pieces/apps/op-ed/):**
```
pieces/apps/op-ed/
├── op-ed.pdl
├── layouts/op-ed.chtpm
├── plugins/+x/op-ed_module.+x
└── pieces/
```
- Launched via CHTPM parser (`href="..."` button)
- Module reads layout, processes input, writes state
- Renders ASCII via CHTPM parser

**GL-OS Apps (Future Structure):**
```
pieces/apps/gl_os/apps/terminal/
├── terminal.pdl
├── layouts/terminal.gl layout
├── plugins/+x/terminal_module.+x
└── pieces/
```
- Launched by gl_desktop.c (reads app.pdl, forks module)
- Module writes to GL-OS session files
- Renders OpenGL via gl_desktop.c

**Key Insight:** Same TPM pattern, different renderer (ASCII vs OpenGL)

### Priority 3: Modular App Structure
1. **Current State (Built-in):**
   - `process_terminal_command()` has hardcoded commands
   - `create_cube_window()`, `create_terminal_window()` are direct calls
   - Works but requires recompiling gl_desktop.c for new apps

2. **Future State (Modular - Like CHTPM):**
   ```c
   // FUTURE: Dynamic module loading
   void process_terminal_command(DesktopWindow* win) {
       char *module_path = get_module_for_command(cmd);  // Read from app.pdl
       if (module_path) {
           pid_t pid = fork();
           if (pid == 0) {
               execl(module_path, module_path, cmd, NULL);
               _exit(1);
           }
           // Module writes to session files, gl_desktop.c renders
       }
   }
   ```
   - Each app has own folder: `pieces/apps/gl_os/apps/<app>/`
   - Each app has own module: `plugins/+x/<app>_module.+x`
   - Add new app = new folder, no gl_desktop.c recompile

## 5. TECHNICAL POINTERS

- **Compilation:** Use `bash compile_all.sh`. It now includes `-lGL -lglut -lGLU` and handles macOS frameworks.
- **Rendering:** `draw_window()` handles branching logic based on `win->type`.
- **Z-Order:** Clicking a window moves it to the end of the `windows[]` array (highest render priority).
- **TPM Consistency:** Ensure `write_session_state()` is called after any window modification (move, resize, rename).
- **Ledger Isolation:** Use `pieces/apps/gl_os/ledger/master_ledger.txt` (NOT global ledger)
- **Session Files:** All state in `pieces/apps/gl_os/session/` (state.txt, view.txt, etc.)
- **Modularity:** Future apps use separate `+x` modules (like CHTPM)
