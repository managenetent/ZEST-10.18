================================================================================
BEST PRACTICE: CENTRALIZED OPS DIRECTORY
================================================================================

WHERE TO PUT OPS:
─────────────────
All ops for a project should live in ONE central location:

  projects/<project_id>/ops/
  ├── +x/                    # Compiled binaries
  │   ├── op_one.+x
  │   ├── op_two.+x
  │   └── op_three.+x
  ├── src/                   # Source code (optional, can be in root)
  │   ├── op_one.c
  │   ├── op_two.c
  │   └── op_three.c
  └── ops_manifest.txt       # Registry of all ops (REQUIRED for Fondu)

This is the canonical structure. Don't scatter ops in traits/, plugins/, etc.


ops_manifest.txt FORMAT:
────────────────────────
# Comment lines start with #
# Format: op_name|description|args_format

create_profile|Creates a new user profile|project_path:string,username:string
auth_user|Authenticates a user session|project_path:string,username:string
get_session|Get current session data|project_path:string,username:string


PROJECT.PDL REGISTRATION:
─────────────────────────
In your project.pdl, declare that your project exposes ops:

  [META]
  project_id = user
  exposes_ops = true
  entry_layout = projects/user/layouts/user.chtpm
  
  [OPS]
  ops_dir = projects/user/ops/
  manifest = projects/user/ops/ops_manifest.txt


FONDU INSTALLATION:
───────────────────
When you run: ./fondu --install <project>

Fondu will:
  1. Read project.pdl → finds exposes_ops = true
  2. Read ops/ops_manifest.txt → gets list of ops
  3. Copy ops/+x/*.+x to pieces/apps/installed/<project>/ops/+x/
  4. Register ops in pieces/os/ops_registry/<project>.txt
  5. Update pieces/os/ops_catalog.txt

Now other projects can call your ops via PAL:
  OP user::create_profile "player1"


EXTERNAL OPS KVP (OPTIONAL):
────────────────────────────
For advanced setups, you can maintain an external KVP file:

  projects/<project_id>/ops/ops_registry.kvp
  
  Format:
    op_create_profile=projects/user/ops/create_profile.c|username:string
    op_auth_user=projects/user/ops/auth_user.c|username:string
    op_get_session=projects/user/ops/get_session.c|username:string

This is useful for:
  - Dynamic op discovery
  - Ops that span multiple directories
  - External/trunk ops that aren't in projects/


WHY CENTRALIZE OPS?
───────────────────
✓ Fondu can install them as a bundle
✓ Other projects can discover them via --list-ops
✓ Clean separation of "muscle" (ops) from "brain" (manager)
✓ Easy to test ops standalone
✓ Reusable across projects (PAL scripts call by project::op_name)
✓ Single source of truth for what ops exist

DON'T:
  ✗ Scatter ops in traits/, plugins/, manager/, etc.
  ✗ Hardcode op paths in C code (use project::op_name)
  ✗ Forget to update ops_manifest.txt when adding ops

================================================================================
HOW TO READ THIS DICTIONARY
================================================================================

Each op entry contains:
- OP NAME: The command name (e.g., user::create_profile)
- LOCATION: Where the source code lives
- PURPOSE: One-line description
- ARGS: Arguments it takes
- RETURNS: What it outputs/changes
- EXAMPLE: How to call it in PAL script
- STATUS: Active / Legacy / Deprecated

FONDU LIFECYCLE OPS
================================================================================

Fondu is the project lifecycle manager. It provides these commands:

Location: pieces/system/fondu/fondu.+x

Commands:
  --install <project>    - Deploy to production + register ops
  --uninstall <app>      - Remove from production + unregister ops
  --archive <project>    - Move to trunk (source only)
  --restore <project>    - Restore from trunk + recompile
  --list                 - Show all projects and states
  --list-ops             - Show all available ops
  --help                 - Show help
