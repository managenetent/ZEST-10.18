========================================================================
1. WHAT chtpm_rgb_render.c ACTUALLY IS (context if you weren't here for
   the earlier build)
========================================================================
`shared-ops/chtpm_rgb_render.c` is a new, persistent daemon (real
wraith_rgb_daemon.c equivalent) that watches `pieces/display/
frame_changed.txt`, reads `pieces/display/current_frame.txt`, and font-
rasterizes it into `pieces/display/rgb_frame.raw` - the SAME path
`system/gl_mirror.c` already reads. It was built to fix an EARLIER,
different complaint: chtpm's own menu chrome (the "Control Hero"
button, footer options past #5) never appeared in the GL window,
because the OLD renderer for that window (`ops/compose_rgb_frame.c`)
draws real 2D tile graphics straight from game state and never touches
`current_frame.txt` at all. Full background: `!.handoff-doc-j17.txt`
and `project_chtpm_to_pal_layout_bridge.md` (memory).

That fix is real and was live-tested (via PNG dumps - see sec. 4 below
for why that phrase matters). This handoff is about a SEPARATE,
follow-up report after that fix: the RGB/GL output now looks worse in
a different way than before.

3. THE REAL, LIKELY EXPLANATION: flat colors, not real pixel art - a
   deliberate scope decision that may need revisiting
========================================================================
`chtpm_rgb_render.c` does NOT render actual emoji-looking pixel art. It
renders a SOLID, FLAT color block for any recognized multi-byte UTF-8
tile glyph, looked up from `terrain_types.txt`/`furniture_types.txt`'s
own `rgb_top_emoji` field. This was a direct response to an earlier
round of feedback ("i liked it more when it showed colored blocks or
emoji... what changed" - the fix at the time was moving from an
ARBITRARY hashed color to the REAL curated registry color, verified via
PNG dump showing correct wall/floor colors matching the registry
exactly).

But `ops/compose_rgb_frame.c` (regular "run" mode's own GL renderer)
does MORE than flat colors for tracked entities (furniture/items/
monsters/hero/xlector, via `cell_asset[]`): it loads REAL, pre-
rasterized 16x16 pixel-art voxel data from `pieces/registry/
emoji_assets/<asset_id>/voxels_16.csv` (`load_emoji_voxels()`/
`blit_emoji_tile()`) and blits it ON TOP of the flat base color. THAT
is almost certainly what "looks good like wraith" (a loose way of
saying "looks as good as our own regular run-mode rendering already
does" - real wraith-alpha itself is plain-text-only, no emoji tiles at
all, confirmed via direct research, so "like wraith" here means "like
our own family's best existing GL output," not literally wraith-
alpha's own appearance).

`chtpm_rgb_render.c` was deliberately built WITHOUT this voxel-art
layer - it only has the DISPLAYED EMOJI BYTES from `current_frame.txt`
text, not the game state `cell_asset[]` needs (asset IDs, tracked-
entity positions). Terrain/furniture tiles get a flat color; ANYTHING
without a registry `rgb_top_emoji` entry (items, monsters, hero '@',
xlector 'X') falls through to `utf8_hash_color()` - an arbitrary,
non-curated color, which for a map with visible items/monsters would
look exactly like "messed up, not showing the right thing."

**This is very likely the real gap the report is about**, not a new
bug in the strict sense - a real, honest limitation of what was built,
now visible under closer/live use rather than a single PNG-dump check.

6. SECOND FOLLOW-UP, same day: "emojis look bad... gl screen randomly
   disappears when enter is pressed... looks smushed... getting cut
   off". TWO MORE real bugs, both fixed, both confirmed via a direct
   before/after screenshot comparison the user provided.
========================================================================
**Bug A - tiles squished to half width, map cut off / cramped into
half the window.** Root cause: sec. 6's own voxel-art port blitted the
real 16x16 pixel data DOWNSCALED 2:1 to fit this file's then-uniform
GLYPH_W(8)-wide text columns. That's not just visually worse - it was
STRUCTURALLY UNNECESSARY: `VIEWPORT_W(40) * TILE_PX(16) = 640 =
FRAME_W` exactly, confirmed via direct read of `ops/
compose_rgb_frame.c`'s own `#define`s - a full 40-tile map row already
fits the frame perfectly at NATIVE 16px width, no scaling needed at
all. Fixed by making `blit_text()` track a running PIXEL x-position
instead of a fixed column count: matched tile glyphs advance by
VOXEL_PX(16, no scaling, full native resolution), plain ASCII text
still advances by GLYPH_W(8) (chrome/footer lines need that narrower
width's ~80-column budget - confirmed against game.chtpm's own longest
footer line, ~72 chars, which would truncate under a uniform 16px
width). This mixed-width-per-glyph-type approach is real wraith-alpha's
OWN precedent, not invented here - `compose_rgb_frame.c`'s own header
comment already cited it: "[wraith's] daemon targets a variable
emoji_glyph_size."

**Bug B - GL window randomly goes black/disappears on a keypress.**
Root cause: this daemon writes `rgb_frame.raw` via a plain
`fopen(path,"wb")` (truncates to 0 bytes immediately) then `fwrite()` -
and polls/rewrites this file far more often (~60/sec) than `ops/
compose_rgb_frame.c` ever did (only on an actual keypress, gated
behind the module's own `.pal` loop). Confirmed via direct read of
`system/gl_mirror.c`'s own `load_texture()`: it does a plain `fread()`
with no retry, and on a short read (exactly what happens if it opens
the file in the window between this daemon's truncate and the
completion of its write) ZERO-FILLS the rest of the buffer and uploads
that as the frame - a real, textbook torn-read race, not a guess. This
pre-existing write-in-place pattern is shared by `compose_rgb_frame.c`
too, but never manifested visibly there because it writes far less
often - this daemon's much higher write frequency made the same latent
race actually reachable. Fixed by writing to `<path>.tmp` then
`rename()`-ing it into place (POSIX `rename()` on the same filesystem
is atomic - any reader sees either the complete old file or the
complete new one, never a partial one).

Both fixed in `shared-ops/chtpm_rgb_render.c`, rebuilt into mutaclsym
and zoo_0000, live-verified via PNG dump: map now fills the full 640px
width edge-to-edge with correctly-proportioned, unsquished tile icons,
matching the ORIGINAL "how it used to look" screenshot's own quality.
Bug B's fix can't be directly stress-tested without a live GL window
(same sec. 4 caveat as ever), but `rename()`'s atomicity guarantee is
well-established POSIX behavior, not a hopeful guess.

**Real, recurring lesson worth naming explicitly**: this is the SECOND
time in this same thread a fix was shipped, verified via PNG dump, and
still turned out visibly wrong once the user actually looked at the
real GL window. The PNG-dump proxy catches "are the right bytes being
written" but has now missed a scaling/proportion regression (bug A)
and can't catch a timing/race bug AT ALL (bug B, by definition only
shows up under real polling-frequency timing a single static dump
can't reproduce). Do not treat a PNG-dump check as equivalent to "the
user confirmed this looks right" - it isn't, and this thread is the
concrete evidence.

8. FOURTH FOLLOW-UP, same day: THE REAL ROOT CAUSE of the remaining
   flicker/disappearing - a genuine architectural gap vs. real
   wraith-alpha, found only after the user redirected THREE times

   "tpmos fuzz-op wraith all use a frame changed marker file... its a

========================================================================
**The bug**: `system/gl_mirror.c` watched `pieces/display/
frame_changed.txt` for its own "reload the texture" signal - the SAME
file `chtpm_parser_pal.c` grows as its own UPSTREAM signal, which
`shared-ops/chtpm_rgb_render.c` ALSO watches as ITS OWN input trigger.
Confirmed via direct read that real `wraith_gl.c` does NOT do this -
its `WRAITH_FRAME_TRIGGER` is `rgb_frame_changed.txt`, a genuinely
SEPARATE, DOWNSTREAM file grown ONLY by `wraith_rgb_daemon.c`'s own
`pulse_rgb()`, called AFTER its output is fully written - real
wraith's own two-stage pulse: chtpm-parser writes text + grows an
upstream trigger -> rgb daemon renders + grows its OWN, separate,
downstream trigger -> GL reader watches ONLY the downstream one.

**Confirmed as a real, measured bug, not a guess**: built a live
checksum trace (sampled `rgb_frame.receipt.txt`'s own checksum against
`gl_display.receipt.txt`'s own loaded checksum every 30ms through a
rapid Enter/Escape keypress burst). BEFORE this fix: `gl_mirror`
"consumed" the shared upstream trigger before `chtpm_rgb_render` had
even started its own settle-and-render, then had NO further signal to
react to once the real output was ready - not a brief lag, a
PERMANENT stall (21+ consecutive mismatched samples, ~600ms+, that
never self-corrected even after the daemon settled on its final,
stable frame). This is almost certainly the real cause of BOTH "random
disappearing/flickering" AND, very plausibly, part of the earlier
"nav options don't show until enter is pressed" report too.

**Fix**: added `pieces/display/rgb_frame_changed.txt` - a new,
dedicated downstream pulse, matching wraith's own naming closely.
Both `shared-ops/chtpm_rgb_render.c` (chtpm mode) and `ops/
compose_rgb_frame.c` (regular "run" mode, BOTH mutaclsym's and
zoo_0000's own copies, for consistency regardless of which mode is
active) now grow it ONLY after a complete write (rgb_frame.raw +
receipt both done). `system/gl_mirror.c` (both projects) now watches
ONLY this file, no longer `frame_changed.txt` directly.

**Re-verified with the identical stress test post-fix**: 50 samples,
30ms apart, through the same rapid keypress burst - only 2 ISOLATED,
SINGLE-SAMPLE mismatches (each immediately followed by a match on the
very next sample, i.e. normal ~30ms poll latency, not staleness), zero
persistent stalls. A dramatic, measured improvement, not just "seems
better."

**Also cross-referenced real wraith_rgb_daemon.c directly for the
settle-delay question from sec. 7**: it has NEITHER an atomic write
NOR any settle/stability delay at all - confirms sec. 7's own
current_frame.txt-stability-polling fix is an original improvement
beyond what wraith itself does, not something being incorrectly
skipped from a real precedent.
