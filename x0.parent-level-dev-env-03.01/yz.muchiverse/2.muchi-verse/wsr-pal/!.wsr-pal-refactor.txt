correct and load-bearing.

It ALREADY does two things right that mutaclsym had to be corrected on
today - don't "fix" these, they're already the target shape:
  - Real piece.pdl METHOD-table-driven menus (`wsr_menu_input.c`/
    `wsr_compose_frame.c`, `pieces_template/wsr_main_menu/piece.pdl`
    etc.) - this is pal-standards.txt §12's own rule, already followed
    here. mutaclsym's own hero/piece.pdl had to be wired up to this
    pattern for the FIRST time today; wsr-pal's menu system already
    uses it.
  - `system/chtpm_parser_pal` IS ALREADY built from `shared-ops/
    chtpm_parser_pal.c` in `scripts/build.sh` - the base chtpm bridge
    binary compiles and works. What's missing is everything ABOVE
    that: no `.chtpm` layout has been authored, no `button.sh` action
    launches it, no `<module>`/`<interact>` wiring exists.

One real, load-bearing difference in wsr-pal's own piece.pdl
convention worth understanding before wiring `${piece_methods}` up:
its METHOD `VALUE` column holds COMMAND STRINGS wsr_menu_input.c
interprets itself (`STUB`, `CYCLE_CORP`, `GOTO:wsr_trade_menu`,
`TICK_ALL:1`), NOT literal `ops/+x/<name>.+x` executable paths like
mutaclsym's own hero/piece.pdl uses. This should still work correctly
with `load_dynamic_methods()` (see §12.3.5 below) since chtpm's own
dispatch is POSITION-based (digit N = row N in the METHOD table, not
the VALUE string itself) - `wsr_menu_input.c`'s own existing digit
handling doesn't care whether the digit arrived via a real keyboard or
a chtpm-relayed `KEY:n` - but this is a real, untested assumption
(this agent ran out of time to verify it live) - confirm it with a
real test before trusting it.

3. THE INTERACT + MODULE PATTERN
========================================================================

Full account: pal-standards.txt §7, `!.handoff-doc-j17.txt`. Real
mechanism: `<module>${module_path}</module>` launches a SECOND,
persistent `system/prisc+x <some_new_pal_script>.pal` process the
instant the layout is parsed. `<interact src="...">` + a button's
`onClick="INTERACT"` relay EVERY subsequent key into that file while
engaged, until ESC.

Build `pal/main_loop_chtpm.pal` as a variant of wsr-pal's own real
`pal/main_loop.pal` - byte-identical except the `read_history` line
reads from `pieces/apps/player_app/interact_relay.txt` via the
LITERAL-PATH 3-ARG FORM (`read_history <path> xD, xS`) - this REQUIRES
§1.1's shared-ops prisc+x.c migration to be done FIRST, since that
3-arg form doesn't exist in wsr-pal's own current, stale, local
prisc+x.c copy. Two real, already-found-and-fixed bugs to avoid
re-discovering the hard way:
  - Do NOT point `<interact src>` at the SAME file `keyboard_input.c`
    already writes unconditionally (`player_app/history.txt`) - real,
    live, confirmed double-dispatch bug (2 keypresses registered as 4).
    Use a genuinely separate file (`interact_relay.txt`).
  - Do NOT keep a `beq x2, x9, quit` ('q' halts) in the CHTPM variant
    of main_loop.pal - while INTERACT is engaged, chtpm relays EVERY
    key including 'q', so a player quitting normally would silently
    kill the module specifically while chtpm keeps relaying into a
    dead process - "stuck, no input response," a real, live-confirmed
    bug. The module never owns quitting in chtpm mode; that's chtpm's
    job alone (top-level 'q', or ESC then 'q').

4. GL/RGB RENDERING - READ THIS SECTION MOST CAREFULLY
========================================================================

This is where today's real time went. `pal-standards.txt` §11 is the
authoritative, condensed rule list - follow it exactly, in order:
  §11.1 - the two-stage pulse (`rgb_frame_changed.txt`, separate from
          `frame_changed.txt`) is MANDATORY, not optional. Skipping
          this causes a REAL, CONFIRMED PERMANENT STALL (not a brief
          lag) under real play - this was the single most expensive
          bug found today, requiring three separate rounds of "still
          doesn't look right" before the actual root cause (not a
          proxy/guessed fix) was found.
  §11.2 - atomic writes (`<path>.tmp` + `rename()`) for both
          `rgb_frame.raw` and its receipt.
  §11.3 - settle-poll `current_frame.txt`'s own size before reading it
          after a pulse change (a SEPARATE race from §11.1 - both are
          real, both were found live, neither one's fix substitutes
          for the other).
  §11.4 - `gl_mirror.c` reads `frame_w`/`frame_h` from the receipt
          file dynamically, never hardcoded `WIDTH`/`HEIGHT` #defines.
  §11.5 - tile-shaped content (any matched emoji) blits at native
          pixel width (16px), plain text at the narrower glyph width
          (8px) - a running pixel x-position, not a fixed column
          count.
  §11.6 - real registry colors (`rgb_top_emoji` field), never a hashed
          fallback, for anything with a registry row; hardcoded
          entries for hero/xlector-equivalent pieces IF wsr-pal's own
          emoji_mode gives its own pieces (corp tickers? the player
          cursor equivalent?) a real, distinct multi-byte emoji
          representation the same way mutaclsym's hero (🧑)/xlector
          (🎯) do - check whether wsr-pal even HAS an emoji-mode
          concept at all before assuming this applies; it may not.

Practical build order that worked today, recommended here too:
  1. Build `ops/compose_rgb_frame.c` FIRST (the regular "run" mode
     renderer, no chtpm involved yet) - get `system/gl_mirror` showing
     wsr-pal's own real content correctly in regular "run" mode before
     touching chtpm mode at all. This isolates "is the GL pipeline
     itself sound" from "is the chtpm-specific two-stage pulse sound."
  2. THEN build/wire `shared-ops/chtpm_rgb_render.c` into wsr-pal's own
     `scripts/build.sh` and `button.sh chtpm` action (it's a SHARED
     file already - no new C code needed for the daemon itself, just
     wiring: build it, launch it, make sure both it AND
     `compose_rgb_frame.c` grow the SAME `rgb_frame_changed.txt`).
  3. Size `FRAME_W`/`FRAME_H` in `chtpm_rgb_render.c`'s own #defines
     for whatever wsr-pal's own real chtpm chrome + embedded content
     actually needs - do NOT assume mutaclsym's own 640x768 is right;
     count wsr-pal's own real max line count first (same method used
     today: launch a real session, `wc -l pieces/display/
     current_frame.txt`, add real headroom).
  4. Test via `dump_rgb_png.+x` throughout, but READ pal-standards.txt
     §11's own header comment on this file's real limitation: a static
     PNG dump proves bytes are correct, it CANNOT catch either of the
     two real timing races in §11.1/§11.3 - those need a live burst-
     keypress test with a checksum trace (see `!.handoff-july17.txt`
     sec. 8 for the exact trace methodology used to find them - a
     Python `pty.fork()` + real `keyboard_input` binary, sampling
     `rgb_frame.receipt.txt` vs. `gl_display.receipt.txt` checksums
     every ~30ms through a rapid keypress burst).
