2. TEST RESULTS (all real, live-run, not hypothetical)
================================================================================
- **preset/weighted with real ORB seed data (cash=100, price=167.70)**:
  correctly holds every tick - can't afford even one share at real
  starting cash. Not a bug: ORB's real risk_bias=9 (conservative)
  computes a fundamental_value of 116.49 against a 167.70 market price
  - by this model ORB is genuinely overvalued at its real seed values,
  so refusing to buy is the CORRECT call, not a stuck demo.
- **weighted, buy path proven** (book_value bumped to 6000 to force a
  clear undervaluation): `fundamental_value=278.50` vs `price=167.70`
  -> bought repeatedly, cash/shares_held updated correctly each tick.
- **weighted, sell path proven** (shares_held=5, book_value dropped to
  force overvaluation): `fundamental_value=88.34` vs `price=167.70` ->
  sold repeatedly, cash increased/shares_held decreased correctly.
- **llm, same undervalued scenario**: agreed with the weighted tier's
  real signal (`fundamental_value=278.50` -> bought), ~2.3s wall-clock
  for the full 12-tick run including live LAN gemma calls.
- **human tier**: confirmed genuine park-and-wait (no decision queued
  -> "[human] waiting for input", state unchanged), then correctly
  resolved once queued. **One real, meaningful finding**: the human
  tier's own gate is affordability-only (`cash >= price` / `shares_held
  > 0`) - it does NOT check the fundamental-value model at all, unlike
  every automated tier. A human CAN buy something the valuation model
  would call fairly-valued or even overvalued; the automated tiers
  cannot. This is a genuine, not-yet-decided design question worth
  flagging: should human trades be bounded by the same valuation gate
  (more "fair," less interesting as a human override), or stay fully
  discretionary (more realistic - real human traders ignore models
  too)? Left as-is (fully discretionary) for now, named here rather
  than silently decided.

**Two real bugs found and fixed by direct interface testing during this
pass** (not caught by writing the code, only by actually playing it -
see `feedback_test_through_playable_interface.md`):
  1. `wsr_compose_frame.c`'s `get_nth_corp()` used unsorted `readdir()`
     order while the new `scripts/active_corp.sh` (what the new menu
     commands resolve "the active corp" through) used sorted `ls` -
     the two could disagree on which corp index N meant, so the header
     could display "Active: corp_X" while an action actually hit corp_Y.
     Fixed by making both use the exact same sorted-`ls` resolution.
  2. `bank_loan_op.c` allowed a corp to loan to itself if the player's
     own owned/active corp happened to BE the hardcoded bank (e.g.
     electing themselves CEO of corp_AFL, the same corp the financing
     menu's "Request Loan" row targets) - guarded against with an
     explicit borrower==bank check.
  3. `button.sh`'s `new-game` shortcut broke AGAIN the moment this pass
     added 3 new rows to `wsr_main_menu` (pushed New Game from position
     5 to 8) - the SAME hardcoded-digit mistake as §2.6's earlier fix,
     recurring because the fix that time was "use the right digit," not
     "stop hardcoding a digit at all." Fixed for real this time by
     resolving NEW_GAME's row number FROM the piece.pdl itself at
     runtime, matching the whole point of piece.pdl-driven menus.

**A real, if latent, architecture gap found during testing, NOT fixed
this pass** (documented rather than silently left, per this whole
project's own discipline): `main_loop.pal` does `li x1, 0` unconditionally
on every launch - the input-history read-cursor always starts at 0, so
if the interactive process ever restarts mid-session (crash, or a
test/dev tool relaunching it), `history.txt`'s ENTIRE contents replay
from the beginning, not just new keypresses since the restart. This
directly caused a test-methodology bug THIS session (appending new test
keycodes to an un-cleared `history.txt` across separate `sim-key`-style
invocations replayed earlier actions a second time, corrupting a bond
balance during testing) - harmless for a real, continuously-running
play session that never restarts the process, but a genuine risk if it
ever does. Worth fixing (persist the byte-cursor to a small file instead
of resetting it every launch) before this project is trusted for a long
real play session, not urgent enough to block this handoff.

Direct user feedback after trying the new gameplay loop: "i tried

exists (not just a menu label), replicate it exactly - don't shortcut
it to fit the current menu system's capability, extend the menu system

Required a genuinely new UI capability: wsr-pal's menu system had ONLY
ever supported numbered-row selection (digit accumulator + Enter), never
free-text entry. Researched TPMOS's own real `cli_io` element directly
(`chtpm_parser.c`, not guessed) for the right shape: a per-element text
buffer, printable chars append, backspace trims, Enter submits, an
`input_mode=="numeric"` flag filters non-digit keys before they reach
the buffer. Built a small, wsr-pal-native equivalent (NOT a port of
chtpm_parser.c's own code - a fresh implementation in this project's own
style, same "duplicate freely, no shared headers" convention as
everything else here):
  - `wsr_menu/state.txt` gained `prompt_active`/`prompt_step`/
    `prompt_buffer` fields.
  - `wsr_wizard_input.c` (new op) - the actual 5-step flow: step 1
    (industry, numeric list read LIVE from the real `corporations/
    36_industries_wsr.txt`, same file the original game itself reads),
    step 2 (country, same live-read approach against the real
    `governments/generated/gov-list.txt` - re-numbered 1..N by row order
    since that file's own leading index column is confirmed inconsistent,
    never trusted), step 3 (funding amount, numeric text, same real
    $1000M-for-bank-or-insurance/$100M-otherwise threshold
    `incorporation.c` itself enforces), step 4 (corp name, free text,
    5-25 chars, same real bounds - NOT uniqueness-checked against other
    names, a named simplification), step 5 (ticker, free text,
    1-6 chars, auto-uppercased, collision-checked via `corp_ipo.+x`'s
    own existing "already exists" failure, re-prompting on collision
    rather than aborting the whole wizard).
  - `corp_ipo.c` extended (not replaced) to accept the real collected
    name/industry/country as optional trailing args, writing `name`/
    `industry`/`country` fields onto the new corp piece.
  - `wsr_menu_input.c` gained a `prompt_active` check at the very top of
    dispatch that delegates EVERY key to `wsr_wizard_input.+x` instead of
    normal menu logic while a wizard is running (matching real cli_io's
    own "active mode" exclusivity), plus a `START_WIZARD:new_corp`
    special command (replacing plain `NEW_CORP` on the Startup New Corp
    row) that initializes the wizard state.
  - `wsr_compose_frame.c` gained a dedicated wizard render path - shows
    the current step's real prompt text, the real industry/country list
    when applicable, and the typed buffer with a text cursor.
