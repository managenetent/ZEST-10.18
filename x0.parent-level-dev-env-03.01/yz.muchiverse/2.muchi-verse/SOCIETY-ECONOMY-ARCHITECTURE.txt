0.75 CLASSIC / NEW / LEARN - a general mode-selector principle,
     resolved from two clarifying questions into one unifying answer
================================================================================
Two open questions from §0.5 (should production couple directly into
valuation or stay indirect; should the ledger's open loan-request
broadcast stay as designed or narrow to WSR's real shape) both got the
SAME answer, generalized: **every mechanic that has both a "real WSR"
version and a "this project's own extended" version should be a
user-selectable mode, not a single hardcoded choice** - `classic`
(faithfully matches real WSR's actual mechanic, per §0.5's citations -
useful as a purist mode AND as a known-ground-truth baseline), `new`
(this project's own richer/extended design - direct production-
valuation coupling, the open loan-request ledger), and `learn`
(FSM/RL/research-driven - the mechanic's own behavior is tuned from
data rather than fixed by either `classic` or `new`'s hardcoded
formula).

**Direct mapping onto the already-built `decision_mode` chassis** (not
a new selector mechanism, a config axis choosing WHICH formula/tier a
given decision draws from): `classic` and `new` both resolve to
`decision_mode=0`(preset)/`1`(weighted) - same chassis, different
formula/data source per mode; `learn` resolves to `decision_mode=2`
(real `rl`, not yet built - see `GAME-AI-SPEED-DOCTRINE.txt` §4's own
next-priority framing, now with a second concrete reason to build it).

**`learn` mode's stated purpose is real and worth naming explicitly**:
"if they are just trying to run headless to make a more robust sim for
b2b type highly accurate purposes" - this is `MUCHICIVO_BIBLE.txt` §7's
own "sell as both game and product" framing, made concrete for the
economic sim specifically: a `learn`-mode, headless, long-running
instance of this simulation is itself a real potential product
(economic/financial modeling), not just a gameplay mode.

Concretely, per the two resolved questions:
  - Production coupling (§4): `classic` = indirect only (production
    changes Cash/Inventory, the existing fundamental-value formula
    reads those, no formula changes - matches how WSR's own abstracted
    "business assets" number actually works); `new` = direct
    (`fundamental_value()` gets explicit production-capacity/output
    terms, industry-specific formulas); `learn` = the coupling weights
    themselves are tuned from logged outcomes, not fixed by either.
  - Ledger loan-request shape (§6): `classic` = deterministic credit-
    line lending only, no open broadcast, matching real WSR exactly;
    `new` = the open broadcast-or-targeted ledger as originally
    designed; `learn` = which shape a given bank/company prefers (or
    how it weighs broadcast vs. targeted offers) is itself a tuned
    parameter.
This same three-way split should be the DEFAULT way any future "should
this match real WSR or our own extension" question gets resolved
going forward - not re-litigated per mechanic, applied as a standing
pattern.

1. THE VISION, RESTATED (confirming understanding before building)
================================================================================
A population pool of citizens who live real economic lives - eat, pay
rent, earn wages, buy goods - supplies LABOR (hired by companies based
on real, earned expertise from schools) and DEMAND (driving commodity/
product prices via supply and demand, same shape as the stock-price
formula already ported this session). Companies produce: agricultural
companies grow named crops, manufacturing companies (e.g. auto)
transform mined/bought commodities into products, with any commodity
not yet individually modeled falling into a generic "bulk bank"
(industrial_materials) bucket - the SET of tracked commodities is data/
config, not hardcoded, deliberately incremental. Banks issue loans.
ALL of loans, goods, and services trade through a LEDGER - a plain-text
marketplace where an entity posts either a BROADCAST ask (open to any
qualified respondent) or a TARGETED offer (addressed to one specific
counterparty), and other entities' own local decision-making chooses
whether to respond. Companies may have physical brick-and-mortar
locations (real xyz coordinates, eventually rendered in 3D via the
already-proven shared presenter) that gate retail access UNLESS the
civilization tech tree has unlocked "the internet," at which point
e-commerce companies can trade without one. R&D is tiered: rare,
expensive LLM-distillation calls PROPOSE genuinely novel breakthroughs
(companies/colleges doing cutting-edge research), while routine
teaching/training happens constantly via cheap, local, already-trained
IQABOD-style curricula (teachers "doing light training otherwise") -
this is the direct answer to "how does this tie into iqabod and fsm":
it's the SAME §12 curriculum/qualification system and the SAME
decision_mode chassis, just given their real economic role instead of
being described in the abstract.

If any of the above doesn't match the intent, this is the section to
correct before anything below gets built.

5.5 CORPORATE ACTIONS (real WSR precedent, §0.5 - confirmed added to
    the build order, given a real design here rather than just a
    build-order bullet)
================================================================================
Ported honestly from the real manual's own mechanics (§0.5's
citations), not invented: menu-driven, player-OR-AI-initiated
transactions against another corp piece, each resolved by a
probabilistic "does the counterparty accept" check keyed on the offered
premium (higher premium = higher acceptance odds) - closer to a real
negotiation model than anything else in this document, and a genuinely
different SHAPE from the ledger (§6, post-and-wait) or the fundamental-
value formula (§4, no negotiation at all).
  - **Merger**: stock-for-stock exchange at a 5-100% premium over the
    target's current `stock_price`. Resolution: an acceptance
    probability from the premium (real WSR's own shape - target
    shareholder vote, "yours always for, rivals against, Public/neutral
    split by premium," §0.5's citation), gated by a simple government-
    approval check (reuses a `gov_*` piece's own existing state, no new
    government mechanism needed - a natural, cheap use of already-real
    data).
  - **Greenmail/LBO**: forced tender-offer buyback at ≥10%
    (greenmail)/≥25% (LBO) premium over `stock_price` - mechanically a
    variant of the SAME accept-probability-from-premium shape as
    merger, not a separate mechanism.
  - **Spin-off**: split a corp piece's `composed_of`-style asset list
    (once §4's production/inventory fields exist) into a new corp
    piece, taxable or not per a simple flag - reuses `craft.c`'s own
    consume/produce shape in reverse (one piece's assets becoming two
    pieces' assets, still the same underlying mechanism).
  - **IPO/startup**: create a brand-new `corp_<ticker>` piece from
    scratch (minimum seed capital, per real WSR's own $100M/$1000M-for-
    a-bank thresholds) - the FIRST real "player/AI creates a new
    economic entity at runtime" mechanic in this whole family; every
    other entity type so far (creatures, corps, governments) has been
    pre-seeded from real data or hand-authored, never spawned during
    play. Worth building carefully as the reference shape for any
    future runtime-entity-creation need.
`decision_mode` governs WHO initiates these (a `weighted`/`llm`-tier
corp piece deciding "should I try to acquire this rival" is a real,
sensible use of the existing chassis, not a new decision-making system)
- the transaction MECHANICS above are the same regardless of which tier
chose to attempt one.

6. THE LEDGER/AUCTION MARKETPLACE (the piece explicitly called out as
   the most-liked mechanic - designed concretely, not just named)
================================================================================
A plain-text marketplace directory, `pieces/world_01/ledger/` - one
file per posted ask/offer, matching this whole environment's "one
directory entry per real thing" doctrine (the same shape a piece
already has). Two post shapes:
  - **BROADCAST** - open to any qualified respondent. e.g. a loan
    request (`kind=loan_request poster=corp_ORB amount=500
    max_rate=5.0`), a job listing (`kind=job_listing poster=corp_TRF
    required_expertise=college_math101 wage_offer=40`), a sell offer
    with no fixed buyer (`kind=sell_offer poster=corp_AGRI
    commodity=wheat qty=1000 price=2.50`).
  - **TARGETED** - addressed to one specific counterparty, same file
    shape plus a `target=<piece_id>` field (e.g. offering to sell
    specifically to `corp_TRF`, or a company approaching one specific
    bank instead of broadcasting).
Each tick, entities capable of responding to a given `kind` (banks for
`loan_request`, companies for `job_listing`/commodity offers,
population-pool/citizens for retail offers) scan the ledger directory -
**the exact same dynamic-discovery pattern `tick_all.sh` already proves
this session** (scan a directory for matching entries, no hardcoded
list) - filtered to entries with no `target` field, or where
`target` matches their own piece id. Their own `decision_mode`-governed
logic (§0's chassis, local/instant per the doctrine) decides whether to
fulfill. Fulfilling = a real trade execution (mirrors `corp_trade.c`'s
own consume/produce/update-both-parties shape) that then archives or
deletes the ledger entry.

    `GAME-AI-SPEED-DOCTRINE.txt`'s own rule that ongoing weight-tuning

  - **Cutting-edge R&D breakthroughs** ("we WILL use llm and
    distillation... a little bit can go a long way") = the doctrine's
    own explicitly-named exception (rare, special moments) PLUS
    `ROADMAP-models.txt` §5's original R&D-agent worked example

    record_result`, `decision_mode=llm` specifically for the "propose
    something genuinely novel" step) PLUS §12.3's qualification-
    crossing "breakthrough" event framing. A company or college's R&D
    department, staffed by research-tier-expertise citizens (§3),
    occasionally (NOT every tick - rare, exactly per the doctrine) calls
    a real LLM to propose a genuinely novel research direction; that
    proposal then gets tested/validated LOCALLY (no further API calls),
    and if it works, becomes a permanent local asset - a new trained
    curriculum, a new product recipe, a new commodity. This is
    precisely `MUCHICIVO_BIBLE.txt` §5's "R&D that gives companies
    internal corpuses" line, now given its exact mechanical place: rare
    LLM call PROPOSES, local IQABOD-style training/testing VALIDATES,
    the result becomes a real owned/tradeable economic asset.
This is the whole synthesis: population/labor draws on §12's schools,
R&D draws on §11's distillation pipeline and §5's worked example,
routine economic decisions draw on the `decision_mode` chassis governed
by the speed doctrine, commodities/production/tech-tree draw on the
composable chain and `craft.c`'s recipe mechanism - nothing here is a
sixth new system, it's the first five actually meeting each other.

     "zero new code when this day comes" framing; no reason to build it
     before the economic mechanics it displays exist.
  8. Tech-tree gating (§8) - needs Tier 2's tech tree to exist first
     (per `MUCHICIVO_BIBLE.txt`'s own build order), genuinely blocked

  9. R&D tiering (§9) - deliberately LAST, since it's the highest-

  10. `learn` mode (§0.75) for any of the above - deliberately last of
      all, since it needs real `rl` (still a stub per
      `GAME-AI-SPEED-DOCTRINE.txt` §4) to mean anything beyond a

================================================================================
12. WEATHER, BREEDING (HUMAN AND ANIMAL), FACTORIES-ON-COMPANY-BOOKS,
    HOSPITALS, HIGHWAYS - extending §2-§7, not new systems
================================================================================
Each of these maps onto a mechanism this document (or its siblings)
already named - worth being explicit that none of them are a sixth
architecture to invent:
  - **Weather** - `EVO-DESIGN.txt` §2's own "ecosystem/planetary" tier
    (the SimEarth-flavored layer), already specified as mostly
    `decision_mode=preset`/deterministic per `GAME-AI-SPEED-DOCTRINE.txt`
    (a climate system is, appropriately, always the cheapest tier by
    default). Nothing new needed except actually building that tier -
    it was designed, not yet implemented.
  - **Population breeding/change, human AND animal** - `EVO-DESIGN.txt`
    §2's own aggregate population-tier dynamics ("population/biomass
    per species... competing for niches... a selection pressure") IS
    this mechanism. The real unification worth naming explicitly: §2 of
    THIS document (the human population pool) and `EVO-DESIGN.txt` §2
    (ecosystem-tier species population) are not two systems that happen
    to look similar - they are the SAME mechanism, applied to citizens
    in one case and wild/farmed creatures in the other. A farm's
    livestock is a population-pool bucket exactly like a city's
    citizenry, just consumed as a commodity input (§4) instead of
    supplying labor (§3). One aggregate-population mechanism, reused
    across both the evo tier and the economy tier - not duplicated.
  - **Factories/manufacturing on company books** - directly grounded in
    REAL data already read this session: a corporation's real balance
    sheet (per `ORB.txt`, already migrated) has an actual `Working
    Capital (A/R, Inven.)` line - inventory is ALREADY a tracked
    balance-sheet field in the real data, not something to invent
    parallel to it. Manufacturing output should increase THAT real
    field (and the real `Total Assets` roll-up it feeds), sales should
    decrease it and increase `Cash`/`Free Cash and Equivalents` (also
    already-real fields `corp_trade.c` already reads/writes) - §4's
    production mechanic should route through the corporation piece's
    EXISTING real financial fields, never a separate shadow inventory
    system.
  - **Hospitals** - a structure-tier composable (§7) citizens visit;
    the mechanical hook is population health/mortality feeding back
    into the SAME breeding/population-dynamics mechanism named above
    (a healthier population has a different growth-rate input to that
    same formula) - not a separate health-simulation engine.
  - **Highways** - structure-tier composables connecting physical
    locations (§7); their mechanical role is logistics for the ledger
    (§6) - goods moving between a producer's and a buyer's location
    plausibly costs time/money proportional to highway
    connectivity/distance, a real, if simple, extension of ledger
    fulfillment rather than a new transportation-simulation system.
None of these are scoped into the build order (§11) yet - named here so
they're not lost, and so their mechanical homes are clear before they
are scoped, per this document's own stated purpose.

================================================================================
13. ABSTRACT VS. DETAILED, PER SYSTEM, BY CONFIG - the fractal-zoom
    mechanism EXPOSED AS A REAL PLAYER-FACING SETTING (direct
    instruction: "they can be abstract or detailed depending on config
    or gameplay style selected")
================================================================================
This is not a new mechanism - it is `wrai-dust-fable` §14's "fractal
zoom applies to AI compute, not just pixels" idea and
`COMPOSABLE-MATERIALS-ARCHITECTURE.md`'s own granularity/λ axis, made
explicit as something the PLAYER chooses, not just an internal cost
optimization the engine picks silently. Worth stating plainly since
it's the single idea that ties §2's population pool, §12's weather, and
every other system in this whole family together:

**Every system in this document (and its siblings) already has, or
should have, an abstract default and a detailed option on the SAME
granularity axis** - a `sim_granularity` (or per-system override) config
value picks which end a given playthrough runs at:
  - Population: abstract = an aggregate stat bucket (§2's default,
    cheap, `decision_mode=preset` for the whole pool); detailed =
    individual `citizen_<id>` pieces with their own health/relationships
    /expertise, promoted per `wrai-dust-fable` §14's own "detail only
    for what's currently in focus" framing - not everyone, just whoever
    the config says should be detailed (e.g. only citizens the player
    is directly interacting with).
  - Weather: abstract = one global climate number (§12); detailed = a
    real per-region `EVO-DESIGN.txt`-style ecosystem-tier simulation.
  - Companies: abstract = the formula-driven `weighted` tier already
    built and proven this session; detailed = a real per-department
    breakdown (R&D, manufacturing, retail each their own sub-piece).
This is genuinely the SAME switch, not a different one per system - a
single config axis (or a small number of per-domain overrides) that
moves EVERY applicable system along the same cheap<->detailed spectrum
`GAME-AI-SPEED-DOCTRINE.txt` §2 already uses to reason about compute
cost, now framed as a feature (let the player pick their preferred
simulation depth / performance tradeoff) rather than only an
implementation detail. Worth building the CONFIG MECHANISM once,
generically (a `sim_granularity` field read by whichever
tick-loop/population-pool/weather-system op is running, same "one
mechanism, many consumers" pattern `craft.c`'s recipe reuse already
established repeatedly in this document), rather than inventing a
separate abstract/detailed toggle per system as each one gets built.
