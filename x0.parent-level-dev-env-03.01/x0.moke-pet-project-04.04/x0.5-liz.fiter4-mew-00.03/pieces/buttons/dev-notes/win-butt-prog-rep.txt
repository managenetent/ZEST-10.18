### Issue 3: Terminal Display Not Working (FIXED)

**Problem:** When running `.\button.ps1 r`, the GL terminal opens in a separate
window, but the PowerShell terminal only shows launch messages.

**Root Cause:**

On Linux, `fork()` creates a child process that inherits all file descriptors
from the parent, including stdout. The external `renderer.+x` process can call
`printf()` and output appears in the same terminal.

On Windows, `_spawnl()` creates a process that does NOT inherit the parent's
console stdout handle - even with `_P_NOWAIT`. The child's output goes to a
separate buffer that isn't visible in the parent's console.

**Solution: Integrated Render Thread**

The orchestrator's `render_thread_func()` now handles display rendering directly
on Windows:

```c
void* render_thread_func(void* arg) {
#ifndef _WIN32
    /* Linux: Spawn external renderer (fork inherits stdout) */
    launch_and_register("renderer", "...", NULL, false);
#else
    /* Windows: Render directly in orchestrator thread */
    /* Poll renderer_pulse.txt for changes */
    /* Clear screen using Windows Console API */
    /* Read current_frame.txt and printf() content */
    while (!should_exit) {
        if (pulse_changed()) {
            clear_screen_win32();
            print_frame();
        }
        Sleep(17);  /* ~60 FPS */
    }
#endif
}
```

**Files Modified:**
1. `pieces/chtpm/plugins/orchestrator.c` - Added Windows render thread logic
2. `pieces/display/windows_renderer.c` - Reference implementation (still compiles)
3. `#.docs/windows_renderer.md` - Updated architecture documentation

**Architecture Summary:**
| Platform | Renderer Binary | Render Location |
|----------|----------------|-----------------|
| Linux | `renderer.+x` | External process (fork inherits stdout) |
| Windows | `renderer.+x` | Internal thread (orchestrator has console) |

The `windows_renderer.c` still exists and compiles to `renderer.+x` - used for:
- Standalone testing
- Future use if console inheritance is fixed
- Reference implementation for the render loop logic

### Issue 4: Unicode Box Characters Garbled (FIXED)

**Symptom:**
Windows PowerShell shows:
```
ΓòöΓòÉΓòÉΓòÉΓòÉΓòÉΓòÉΓòÉΓòÉΓòÉΓòÉΓòÉΓòÉΓòÉΓòÉΓòÉΓòÉΓòÉΓòÉΓòÉΓòÉΓòÉΓòÉΓòÉΓòÉΓòÉΓòÉΓòÉΓòÉΓòÉΓòÉΓòÉΓòÉΓòÉΓòÉΓòÉΓòÉΓòÉΓòÉΓòÉΓòÉΓòÉΓòÉΓòÉΓòö
```

Expected (Linux):
```
╔═══════════════════════════════════════════════════════════╗
```

**Root Cause:**
1. **PowerShell encoding:** Windows uses different character encoding (Windows-1252
   or OEM codepage 437/850) than Linux (UTF-8).
2. **Unicode box-drawing characters (U+2550-U+256F)** are interpreted incorrectly.
3. **GL renderer font:** FreeType font doesn't contain Unicode box glyphs, shows `?`.

**Solution: ASCII Fallback (RECOMMENDED)**

All `.chtpm` layout files converted to use ASCII box characters:
- `╔` `╗` `╚` `╝` → `+`
- `═` → `=`
- `║` → `|`
- `╠` `╣` `╦` `╩` `╬` → `+`

**Benefit:** Works on both PowerShell and GL renderer.

**Files Modified:**
- All `pieces/chtpm/layouts/*.chtpm` (10 files)
- `pieces/chtpm/plugins/chtpm_parser.c` - BOX_* macros
- `pieces/chtpm/plugins/chtpm_player.c` - BOX_* macros

}

### Issue 6: Arrow Keys Not Working in MSYS2 (FIXED)

**Symptom:**
Arrow keys work via keyboard injection (writing to history file), but physical
arrow key presses don't move the selector when running in MSYS2/mintty bash.

**Root Cause:**
1. `_kbhit()` doesn't work in MSYS2/mintty pseudo-terminals
2. MSYS2 sends escape sequences (`\x1b[A` for up arrow) instead of Windows scan codes
3. The `readKey()` function only handled native Windows console input

**Fix Applied:**

Added MSYS2/mintty fallback to `readKey()`:
```c
int readKey() {
#ifndef _WIN32
    /* Linux: read from tty */
    while ((nread = read(tty_fd, &c, 1)) != 1) { ... }
#else
    /* Windows: Try _kbhit() first, fallback to fgetc() for MSYS2/mintty */
    if (_kbhit()) {
        c = (char)_getch();
    } else {
        Sleep(10);
        if (kbhit_fallback_enabled) {
            int ch = fgetc(stdin);
            if (ch != EOF) c = (char)ch;
            else return -1;
        } else {
            return -1;
        }
    }
    
    if (c == '\x1b') {
        /* MSYS2/mintty sends ESC [ A/B/C/D for arrows */
        if (fgetc(stdin) == '[') {
            int ch = fgetc(stdin);
            switch (ch) {
                case 'A': return ARROW_UP;
                case 'B': return ARROW_DOWN;
                case 'C': return ARROW_RIGHT;
                case 'D': return ARROW_LEFT;
            }
        }
    }
    /* Native Windows: scan codes */
    int scan = _getch();
    switch (scan) {
        case 72: return ARROW_UP;
        case 80: return ARROW_DOWN;
        // ...
    }
#endif
}
```

**Files Modified:**
- `pieces/chtpm/plugins/orchestrator.c` - Added MSYS2 escape sequence handling
- Added `kbhit_fallback_enabled` global variable

### Issue 8: Ctrl+C Doesn't Kill All Processes (FIXED)

**Symptom:**
Pressing Ctrl+C kills orchestrator but GL window and other child processes
remain open.

**Root Cause:**
Windows Console Ctrl+C events weren't being caught by the signal handler.

**Fix Applied:**

Added Windows Console Ctrl handler:
```c
#ifdef _WIN32
/* Windows Console Ctrl+C handler */
BOOL WINAPI console_ctrl_handler(DWORD dwCtrlType) {
    if (dwCtrlType == CTRL_C_EVENT || dwCtrlType == CTRL_BREAK_EVENT) {
        handle_sigint(SIGINT);
        return TRUE;
    }
    return FALSE;
}
#endif

int main() {
    // ...
#ifdef _WIN32
    /* Windows Console Ctrl+C handler */
    SetConsoleCtrlHandler(console_ctrl_handler, TRUE);
#endif
    // ...
}
```

Also updated `handle_sigint()` to kill all processes:
```c
#else
/* Windows: Kill all child processes (matches Linux pkill) */
system("taskkill /F /IM gl_renderer.+x 2>nul");
system("taskkill /F /IM renderer.+x 2>nul");
system("taskkill /F /IM orchestrator.+x 2>nul");
system("taskkill /F /IM chtpm_parser.+x 2>nul");
system("taskkill /F /IM clock_daemon.+x 2>nul");
system("taskkill /F /IM joystick_input.+x 2>nul");
system("taskkill /F /IM response_handler.+x 2>nul");
system("taskkill /F /IM keyboard_input.+x 2>nul");
/* Run the cleanup script as a final sweep */
system("powershell.exe -File ./kill_all.ps1 >$null 2>&1");
#endif
```
