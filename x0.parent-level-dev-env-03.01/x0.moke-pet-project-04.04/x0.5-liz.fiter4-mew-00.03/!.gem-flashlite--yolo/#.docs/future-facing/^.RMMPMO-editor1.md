## Executive Summary

This document defines the complete architecture for the **RMMP-MO Editor** (RMMP Module-OS Editor), a tile-based game editor built on the True Piece Method (TPM) architecture. The editor achieves feature parity with commercial tools like RPG Maker MV while maintaining full auditability, modularity, and CPU safety.

**Key Achievements:**
- ✅ CPU-safe operation (no throttling, clean Ctrl+C handling)
- ✅ Full editor parity with rmmp-editor (placement, selection, clearing)
- ✅ **Optional** PDL-based method dispatch (extensible without recompilation)
- ✅ Z-level support (3 floors: surface, underground, sky)
- ✅ Map association (pieces tied to specific maps, no bleed-through)
- ✅ Shadow xlector sync (cursor follows selected entity)
- ✅ Project renamed: `fuzz_legacy` → `fuzz-op`
- ✅ Removed: `fuzzpet_v2` (deprecated)

## Part 1: Architecture Overview

### 1.1 The PMO Hierarchy

The system follows a strict thought priority: **PIECE → MODULE → OS**

```
┌─────────────────────────────────────────────────────────────┐
│                    OS / CHTPM (Theater)                      │
│  - Layout files (.chtpm)                                     │
│  - Parser (variable substitution, frame composition)         │
│  - Navigation routing                                        │
└─────────────────────────────────────────────────────────────┘
                            ↕
┌─────────────────────────────────────────────────────────────┐
│                  MODULE (Brain / Stage Manager)              │
│  - editor_module.c (input routing, state management)         │
│  - Calls Ops for logic                                       │
│  - Performs Mirror Sync                                      │
└─────────────────────────────────────────────────────────────┘
                            ↕
┌─────────────────────────────────────────────────────────────┐
│                    PIECE (Soul / Atomic Unit)                │
│  - piece.pdl (DNA - source of truth)                         │
│  - state.txt (Mirror - high-speed reads)                     │
│  - plugins/ (sovereign behavior)                             │
└─────────────────────────────────────────────────────────────┘
```

### 1.2 The 12-Step Pipeline (Input to Render)

```
1. INPUT     → keyboard_input.+x writes ASCII to history.txt
2. ROUTING   → chtpm_parser.c reads key, injects into app history
3. RELAY     → Parser injects key into pieces/apps/player_app/history.txt
4. TICK      → editor_module.c polls history, sees key
5. TRAIT/OP  → Module calls Op (move_entity.+x, place_tile.+x)
6. SOVEREIGNTY → Op updates piece state in projects/{id}/pieces/
7. MIRROR    → Op flushes update to state.txt (high-speed mirror)
8. STAGE     → render_map.+x reads all mirrors, writes view.txt
9. SYNC      → render_map hits frame_changed.txt (1-byte pulse)
10. COMPOSE  → Parser substitutes ${game_map} with view.txt
11. RENDER   → Parser writes current_frame.txt
12. DISPLAY  → renderer prints frame to terminal
```

**Key Insight:** Every step is file-based. No in-memory shortcuts. Complete auditability.

## Part 2: CPU Safety Patterns

### 2.1 The legacy_fuzzpet Reference

The `fuzz_legacy_manager.c` implementation is the **CPU-safe reference**. It never causes throttling or crashes because:

1. **Signal Handling:** Registers `SIGINT` handler for graceful shutdown
2. **Process Groups:** Uses `setpgid(0, 0)` for clean process tree
3. **fork/exec/waitpid:** Never uses `system()` - all child processes are waited on
4. **Focus Throttling:** Sleeps 100ms when not in focus (6x CPU reduction)
5. **stat()-First Polling:** Checks file size before opening
6. **Bounded Execution:** All child processes complete before parent continues

### 2.2 The Editor Crash Fix

**Problem:** Original `editor_module.c` used `system()` calls without signal handlers. Ctrl+C orphaned child processes → CPU spike → crash.

**Solution:** 
```c
// CPU-SAFE pattern (from editor_module.c)
static volatile sig_atomic_t g_shutdown = 0;

static void handle_sigint(int sig) {
    (void)sig;
    g_shutdown = 1;
}

static int run_command(const char* cmd) {
    pid_t pid = fork();
    if (pid == 0) {
        freopen("/dev/null", "w", stdout);
        freopen("/dev/null", "w", stderr);
        execl("/bin/sh", "/bin/sh", "-c", cmd, (char *)NULL);
        _exit(127);
    } else if (pid > 0) {
        int status;
        waitpid(pid, &status, 0);  // ← Wait for child!
        return WIFEXITED(status) ? WEXITSTATUS(status) : -1;
    }
    return -1;
}

int main() {
    signal(SIGINT, handle_sigint);
    signal(SIGTERM, handle_sigint);
    setpgid(0, 0);
    
    while (!g_shutdown) {  // ← Check shutdown flag
        // ... main loop ...
    }
}
```

### 2.3 CPU Safety Checklist

Every module MUST have:
- [ ] Signal handler for `SIGINT` and `SIGTERM`
- [ ] `setpgid(0, 0)` in main()
- [ ] Main loop checks shutdown flag
- [ ] All external commands use `fork/exec/waitpid` (NOT `system()`)
- [ ] Focus throttling (`is_active_layout()` check)
- [ ] History polling uses `stat()` before `fopen()`
- [ ] Sleep interval ≥ 16ms (60 FPS max)

## Part 3: The PDL Model

### 3.1 What is PDL?

**PDL (Piece Definition Ledger)** is the DNA of a piece. It defines:
- **STATE:** Position, type, properties
- **METHODS:** What actions this piece can perform
- **EVENTS:** What triggers those actions
- **RESPONSES:** What feedback to show

### 3.2 PDL File Format

```pdl
SECTION      | KEY                | VALUE
----------------------------------------
META         | piece_id           | player_5_5
META         | version            | 1.0
META         | determinism        | strict

STATE        | pos_x              | 5
STATE        | pos_y              | 5
STATE        | pos_z              | 0
STATE        | on_map             | 1
STATE        | type               | player
STATE        | map_id             | map_0001_z0.txt

METHOD       | on_interact        | ./pieces/apps/playrm/ops/+x/interact.+x player_5_5
METHOD       | on_move            | ./pieces/apps/playrm/ops/+x/move_entity.+x player_5_5
METHOD       | feed               | ./pieces/apps/playrm/ops/+x/fuzzpet_action.+x player_5_5 feed
METHOD       | play               | ./pieces/apps/playrm/ops/+x/fuzzpet_action.+x player_5_5 play
METHOD       | sleep              | ./pieces/apps/playrm/ops/+x/fuzzpet_action.+x player_5_5 sleep

EVENT_IN     | on_interact        | void
EVENT_IN     | on_move            | void
EVENT_IN     | feed               | void

RESPONSE     | fed                | Yum yum!
RESPONSE     | played             | Wheee~ Best playtime ever!
RESPONSE     | slept              | *snore*
```

### 3.3 PDL vs state.txt

| Aspect | piece.pdl | state.txt |
|--------|-----------|-----------|
| **Purpose** | Source of truth (DNA) | High-speed mirror |
| **Format** | Structured table | Flat key=value |
| **Write Speed** | Slow (parse & format) | Fast (append) |
| **Read Speed** | Slow (line-by-line) | Fast (grep) |
| **Updated By** | create_piece.+x, ops | ops, modules |
| **Read By** | pdl_reader (optional) | render_map, modules |

**Best Practice:** Write to both. PDL for audit, state.txt for speed.

### 3.4 The pdl_reader Utility (OPTIONAL)

Created: `pieces/system/pdl/+x/pdl_reader.+x`

**IMPORTANT:** PDL integration is **OPTIONAL**. The editor works perfectly with hardcoded ops. PDL is for advanced users who want to customize piece behavior without recompiling.

**Commands:**
```bash
# Get handler for specific event
./pdl_reader.+x player_5_5 get_method on_interact
# Output: ./pieces/apps/playrm/ops/+x/interact.+x player_5_5

# List all methods
./pdl_reader.+x player_5_5 list_methods
# Output: on_interact\non_move\nfeed\nplay\nsleep

# Check if method exists
./pdl_reader.+x player_5_5 has_method feed
# Output: 1 (exists) or 0 (not found)
```

**Usage in Modules (OPTIONAL):**
```c
// Get method handler (only if you want PDL support)
char handler[MAX_PATH];
if (get_method(piece_id, "on_interact", handler, sizeof(handler),
               project_root, current_project) == 0) {
    system(handler);  // Execute the handler from PDL
} else {
    // Fallback to hardcoded default
    system("./pieces/apps/playrm/ops/+x/interact.+x piece_id");
}
```

**Current Status:** op-ed uses **hardcoded ops** by default. PDL integration is available but not enabled. See `22.oped_pdl_integration.txt` for migration guide.

## Part 6: RPG Maker MV Emulation

### 6.1 Design Goals

The RMMP-MO Editor emulates RPG Maker MV's core features:

1. **Tile-Based Editing:** Place tiles on a grid
2. **Event Creation:** Place entities (NPCs, chests, triggers)
3. **Map Management:** Multiple maps per project
4. **Layer Support:** Z-levels (surface, underground, sky)
5. **Real-Time Testing:** Playtest immediately

### 6.2 Feature Mapping

| RPG Maker MV | RMMP-MO Editor |
|--------------|----------------|
| **Map Editor** | op-ed / rmmp-editor |
| **Tileset** | Glyph palette (`#`, `.`, `R`, `T`, etc.) |
| **Events** | Pieces (`@` player, `&` NPC, `Z` zombie, `T` chest) |
| **Maps** | `projects/{id}/maps/map_XXXX.txt` |
| **Layers** | Z-levels (0=surface, 1=underground, 2=sky) |
| **Playtest** | Load project in player module |
| **Database** | `piece.pdl` files |

### 6.3 Workflow Comparison

**RPG Maker MV:**
1. Open Map Editor
2. Select tile from tileset
3. Click on map to place
4. Right-click to erase
5. F12 to playtest

**RMMP-MO Editor:**
1. Launch op-ed
2. Cycle glyph with LEFT/RIGHT
3. ENTER to place
4. Backspace to erase
5. F12 to playtest

**Key Difference:** RMMP-MO is keyboard-driven, not mouse-driven. This is intentional for terminal compatibility.

// Save xlector position
void save_xlector_state() {
    fprintf(f, "pos_x=%d\npos_y=%d\npos_z=%d\nactive=1\nicon=X\ntype=xlector\n",
            cursor_x, cursor_y, cursor_z);

// Run external command (CPU-safe)
static int run_command(const char* cmd) {
    pid_t pid = fork();
    if (pid == 0) {
        freopen("/dev/null", "w", stdout);
        freopen("/dev/null", "w", stderr);

// Hit frame marker (trigger render)
void hit_frame_marker() {
    FILE *f = fopen("pieces/display/frame_changed.txt", "a");

### 7.3 The stage_map_idx Pattern

**Problem:** Map palette has "ADD" at index 0, but real maps start at index 1.

**Solution:** Two indices:
- `map_idx` - UI highlight (which map is selected in palette)
- `stage_map_idx` - Active map for editing (≥1 for real maps)

**Initialization:**
```c
scan_maps();  // Populates project_maps[], map_count
if (map_count > 1) {
    map_idx = 1;
    stage_map_idx = 1;  // First real map
} else {
    map_idx = 0;
    stage_map_idx = 0;  // No maps
}
```

**Usage:**
```c
// ALWAYS use stage_map_idx for operations
asprintf(&cmd, "...place_tile.+x %s %d %d '%s'",
         project_maps[stage_map_idx], cursor_x, cursor_y, glyph);
```

## Part 8: Troubleshooting

### 8.1 Common Issues

**Issue:** Glyph placement shows "Tile placed" but nothing appears.

**Diagnosis:**
```bash
# Check stage_map_idx
cat pieces/apps/player_app/manager/state.txt | grep current_map

# Check map file
cat projects/op-ed/maps/map_0001_z0.txt

# Test op directly
./pieces/apps/editor/ops/+x/place_tile.+x map_0001_z0.txt 5 5 #
cat projects/op-ed/maps/map_0001_z0.txt
```

**Fix:** Ensure `stage_map_idx` is initialized to 1 in main().

---

**Issue:** Xlector doesn't render.

**Diagnosis:**
```bash
cat projects/op-ed/pieces/xlector/state.txt
```

**Expected:**
```
pos_x=5
pos_y=5
pos_z=0
active=1
icon=X
type=xlector
```

**Fix:** Add `type=xlector` to state file. render_map checks this field.

---

**Issue:** Backspace doesn't clear tiles.

**Diagnosis:** Check if using `stage_map_idx` not `map_idx`.

**Fix:** Change `project_maps[map_idx]` to `project_maps[stage_map_idx]`.
