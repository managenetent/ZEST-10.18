# Investigation Report: GLTPM Voxel Rendering & Interact Mode Issues

## 1. Problem Statement
The current implementation of the A23 GLTPM framework fails to render 3D "voxels" (tiles/sprites) correctly when using the `${game_map}` variable. Additionally, the "Interact Mode" visual feedback (`[^] activated`) and responsiveness are not functioning as expected.

## 2. Root Cause Analysis

### 2.1 The "Strtok" Overlap Bug (Architectural)
In `gltpm_parser.c`, the `gltpm_load_scene` function uses `strtok(layout_sub, "\n")` to process the substituted layout line-by-line. When a multi-line ASCII block (like `${game_map}`) is encountered:
1.  The parser calls `gltpm_process_canvas_block` for **each line individually**.
2.  `gltpm_process_canvas_block` is passed a fixed `origin_y` (e.g., `-5.0f`).
3.  Each call starts its own `row = 0` counter, causing all lines of the map to be rendered at the same Y-coordinate, overlapping each other and appearing as a single garbled row (or nothing if the coordinates are off).

### 2.2 Formatting Interference
The `render_map.+x` Op (which populates `pieces/apps/player_app/view.txt`) includes ASCII "theater" formatting such as:
- Headers: `+=========================================================+`
- Sidebars: `|  ` and `                                     |`
- Z-level indicators: `| [Z-LEVEL]: 0`

The current `gltpm_process_canvas_block` treats **every character** as a potential tile glyph. It attempts to look up `+`, `=`, `|`, and `[` in `registry.txt`, which likely fails or produces garbage tiles.

### 2.3 Variable Substitution Scope
`gltpm_parser.c` was not explicitly loading `game_map` from `pieces/apps/player_app/view.txt`. While `chtpm_parser.c` has hardcoded logic to inject this file into the variable pool, `gltpm_parser.c` only loads from `state.txt` and `gui_state.txt`. Since `render_map.c` writes to a standalone `view.txt`, the `${game_map}` placeholder in `.gltpm` files may remain empty or stale.

### 2.4 Interact Mode Logic
The user reported that `[^] activated` does not appear. In `chtpm`, this is a parser-level state that changes the prefix of the active button. In `gl_desktop.c`, we added `win->is_map_control = 1`, but:
1.  The rendering loop in `gl_desktop.c` for `gltpm_app` does not currently check `win->is_map_control` to change the indicator from `[>]` to `[^]`.
2.  The `Nav >` buffer logic in `keyboard()` does not account for the state change in a way that provides visual feedback.

## 3. Proposed Fixes (The "Virgin Agent" Guide)

### Fix 1: Block-Aware Parsing
Modify `gltpm_load_scene` to detect when it has entered a raw text block and pass the **entire block** to `gltpm_process_canvas_block` once, or maintain a global `canvas_row_offset` that increments across multiple `TOKEN_TEXT` calls.

### Fix 2: Legend Filtering & Padding
Update `gltpm_process_canvas_block` to:
- Ignore lines starting with `+`, `=`, or containing `[Z-LEVEL]`.
- Strip the leading `| ` and trailing ` |` from map lines.
- Only process characters that exist in the `registry.txt`.

### Fix 3: Explicit View Loading
Update `gltpm_load_scene` to mirror `chtpm_parser.c`:
1.  Determine the `view_path` (defaulting to `pieces/apps/player_app/view.txt`).
2.  Read the entire file into a `game_map` variable.
3.  Inject this variable into the `vars[]` array before calling `gltpm_substitute_vars`.

### Fix 4: Visual Feedback for Interact
In `gl_desktop.c`, update the `gltpm_app` rendering loop:
```c
const char *indicator = (i == win->selected_index) ? (win->is_map_control ? "^" : ">") : " ";
```
This ensures that when `INTERACT` is triggered, the focused button shows `[^]`, matching the biblical `chtpm` standard.

## 4. Adherence to Standards (No Reinventing the Wheel)

I have verified that the directory and file structures in `projects/fuzz-op-gl` are 100% identical to the biblical `projects/fuzz-op` reference:
- **Registry:** Both use `assets/tiles/registry.txt` with the `glyph=tile_id` format.
- **Metadata:** Both use `.tile.txt` and `.sprite.txt` for 3D/ASCII metadata (colors, extrusion).
- **Layouts:** `fuzz-op-gl` now uses the same `${game_map}` variable logic that allows it to consume maps generated by the standard `render_map.c`.
- **Managers:** The manager logic follows the CPU-safe `stat()` polling and `fork/exec` patterns found in the flagship ASCII manager.

The "broken" state is not due to a lack of standard adherence, but rather the **Theater Translation Layer**. ASCII-OS renders the map as raw text; GL-OS must "translate" that same raw text into 3D geometry. The fix involves refining this translation (ignoring `+===+` borders) rather than changing the data format.
