## SHARED APPS DETAILED ANALYSIS

### 1. FUZZPET_APP (Token Heavy / CRITICAL Decoupling Needed)

**Token Count:** ~1,200+ lines (manager + traits + world)
**Category:** TOKEN HEAVY
**Coupling Level:** VERY HIGH
**Decoupling Status:** NEEDS DECOUPLING - Causes zombie flicker & project switching bugs

**Core Files:**
- `manager/fuzzpet_manager.c` - 434 lines (main game loop)
- `traits/zombie_ai.c` - 169 lines (AI logic)
- `traits/move_entity.c` - movement handling
- `world/plugins/+x/map_render.+x` - rendering

**Why Token Heavy:**
- Full game manager with input threading, mirror sync, method dispatch
- Z-level support, clock integration, emoji mode
- Multiple entity types (fuzzpet, zombie, selector, pets)
- Response logging, debug output, state persistence

**CRITICAL COUPLING BUGS (Per !.zombie_flicker.txt):**

#### Bug 1: Hardcoded Project Paths in zombie_ai.c
```c
// BUG: Prioritizes projects/fuzz-op/ for entity state
if (asprintf(&path, "%s/projects/fuzz-op/pieces/%s/state.txt", ...) == -1) return -1;
if (access(path, F_OK) != 0) {
    // Fallback to world/map_01 for fuzzpet_app
    asprintf(&path, "%s/pieces/world/map_01/%s/state.txt", ...);
}
```
**Impact:** When running fuzzpet_app, zombie chases "ghost" player from previous fuzz-op session.

#### Bug 2: Shared Global State Files
- `pieces/world/map_01/` - Shared entity storage
- `pieces/apps/fuzzpet_app/fuzzpet/` - Shared mirror state
- `pieces/display/frame_changed.txt` - Global render trigger
- `pieces/apps/player_app/history.txt` - Global input history
- `pieces/apps/player_app/manager/state.txt` - Global manager state

**Impact:** Project switching fails because state isn't refreshed. Old project data bleeds into new project.

#### Bug 3: Multi-Pulse Contention (Flicker)
1. `move_entity.+x` updates state → hits `frame_changed.txt`
2. Renderer prints OLD frame (parser hasn't updated)
3. Manager calls `trigger_render()` → `map_render.+x`
4. `map_render` updates `view.txt` → hits `view_changed.txt`
5. Parser composes NEW frame → hits `frame_changed.txt`
6. Renderer prints NEW frame

**Result:** Rapid OLD→NEW frame sequence causes visible flicker.

#### Bug 4: Ghost Frame on Startup
- `view.txt` is shared across projects
- `chtpm_parser` reads existing `view.txt` on first render pulse
- If project context not initialized, stale map from previous session displays

**To Decouple fuzzpet_app (Required Changes):**
1. **Fix zombie_ai.c:** Remove hardcoded `projects/fuzz-op/` path, use project_root from kvp
2. **Fix collision detection:** Consistent path resolution for all entity state reads
3. **Project-aware state:** Read project ID from `pieces/apps/player_app/manager/state.txt` before any operation
4. **Dedicated view files:** Each project should have its own `view.txt` in `projects/{id}/`
5. **Fix render pipeline:** Single pulse after state update complete (not before)
6. **State isolation:** Replace `pieces/world/map_01/` with `projects/{id}/pieces/`

## Token Optimization Tips

1. **Reference by Name:** Instead of including full fuzz-op context, reference `projects/fuzz-op/manager/fuzz-op_manager.c` for CPU-safe patterns

2. **Use Templates:** Start from `template/` and add only needed pieces

3. **Modular Ops:** Import individual ops from `pieces/apps/playrm/ops/` instead of entire framework

4. **PDL Optional:** Remember PDL integration is optional (per architecture docs). Hardcoded ops work fine for simple projects.

5. **Shared Apps:** The real logic lives in `pieces/apps/` - projects are data containers. Reference shared apps when needed.
