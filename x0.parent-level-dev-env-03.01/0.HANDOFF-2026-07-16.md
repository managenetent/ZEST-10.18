## mutaclsym - what it is, current state

A real-time, single-player, CDDA-inspired survival game. No ncurses -
three independent processes (`system/keyboard_input` reads raw
terminal input, `system/prisc+x` runs `pal/main_loop.pal` - a tiny
custom VM/assembly-like script dispatching to individual C "ops" -
`system/renderer` prints frames). `./button.sh compile|check|run|gl` is
the entry point; `./button.sh help` explains all verbs.

**Full detail lives in `mutaclsym/dox/00-HANDOFF.md`** (status
checklist, next steps) and `mutaclsym/dox/01-cdda-architecture.md`
(deep architecture notes) - read those, this section is just a summary
so this document doesn't duplicate and drift from them.

Done: terrain/furniture/items/inventory, hunger/thirst/stamina,
monster chase+melee AI, crafting with a real picker panel, a
message-log HUD, numbered-choice input, a camera/viewport (maps can be
bigger than the 40x16 terminal view), save/load with a title screen,
and - the bulk of the previous session's work - **an optional GL/RGB
mirror** (`./button.sh gl`, separate from plain `run`): `ops/
compose_rgb_frame.c` computes an RGBA32 framebuffer in pure portable C
(zero GL calls - see GOVERNING CONSTRAINT below), `system/gl_mirror.c`
is the ONE file allowed to call GL/GLUT, blitting that buffer as a
single textured quad, plus real keyboard/arrow input forwarding from
that window and a focus-lock file so the terminal and the GL window
don't double-write input when both are running. `ops/dump_rgb_png.c`
(uses `libraries/stb_image_write.h`, vendored, public domain) dumps
the current frame as a real PNG - use this to actually SEE the mirror,
since checksums alone missed a real bug (see "A bug class to
remember" below).

Open: procedural map generation (the user's stated favorite CDDA
feature, not designed at all yet - needs a room/corridor-generator-vs-
noise-terrain decision before any code), multi-Z buildings, vehicles,
professions/weather/day-night, item/monster placement on the newer
`map_02`.

## 2.muchi-verse - what it is, current state

**`GRAND-ARCHITECTURE.md` is the real source of truth here - read it
in full, this is only a summary.** Its `GOVERNING CONSTRAINT` section
(at the very top) is load-bearing and non-negotiable: `prisc+x` is
meant to eventually compile to RISC-V, so NONE of our own code may
call GL/OpenGL primitives directly, anywhere, ever - the only
exception in the entire family is one minimal, already-written GL
reader per project (`system/gl_mirror.c`, ported near-unmodified
across projects), whose only job is to blit a CPU-computed RGBA32
buffer. Every other file, including any future "chrome"/widget/UI
code, must compute pixels in portable C and never touch GL state
directly.

The other core idea: "mirror" is a real, specific term (see
`GRAND-ARCHITECTURE.md` §0 and 1.TPMOS's own README.md) - ONE state
(a map/registry/etc.), MULTIPLE renderers reading it differently
(ASCII text, or a CPU-computed RGB buffer a GL window blits). Not two
separate implementations of a game - one source of truth, several
presentations.

### piececraft-3d-pal

A Minecraft-clone-shaped voxel game foundation, defaulting to real 3D
(per direct user instruction - the reference project defaults to 2D).
Uses real piececraft-wraith map/tile assets (copied in, not invented).

`clip_poly_near`/`fill_poly_px`/`draw_box`, all cited with line numbers
in `compose_rgb_frame.c`'s own header comment).

**Camera model was fixed, not ported** - direct user correction:
"the wraith 3d pov was meant to show 1rst person 3rd person and free
camera but it wasn't working yet, u can fix this local one now, ill fix
the reference later." So this project has a REAL implementation the
reference project itself still lacks: `ops/camera_input.c` drives a
single yaw-relative rig position (W/A/S/D move relative to current
facing, Q/E turn, Z/X vertical, R/F look up/down); `camera_mode`
(keys 1/2/3) only changes camera PLACEMENT relative to that rig -
1=first-person (camera is the rig, eye height), 2=third-person (pulled
back+up, biased to look at the rig), 3=free-camera (the default - rig
itself, started pulled back for an unobstructed view). All three were
visually verified as genuinely distinct via the debug-PNG method, and
turning/walking were each confirmed to produce the correct resulting
frame, not just "it compiles." Key `8` toggles to a flat top-down 2D
fallback view.

## A bug class to remember (found and fixed 3 times this session)

Both real tile-registry formats use a printable character as BOTH the
comment marker (`#`) AND a legitimate glyph (mutaclsym's wall tile is
literally `#`; piececraft's wall registry row is literally `#=wall`).
Any comment-skip check written as `if (line[0] == '#') continue;`
silently swallows that tile's own data row. This is REAL, pre-existing
(found first in mutaclsym's `move_player.c`/`tick_monsters.c`, both
predating this session, harmless there only by coincidence since the
miss-fallback "not walkable" happens to be the correct answer for
walls specifically), and was reintroduced fresh at least twice more in
new code written this same session, in different files, within
minutes of the first fix - because it's easy to copy the wrong
comment-check pattern without noticing the collision. The fix
everywhere: check whether the SEPARATOR character (`|` or `=`) appears
immediately after position 0, not just whether position 0 is `#`. If
you're writing ANY new code that reads one of these registries, check
this first.

## Why both receipt-checksums AND real PNGs matter for the GL/RGB work

The GL/RGB mirror pipeline (mutaclsym and piececraft-3d-pal both) was
originally verified only via FNV-1a-64 checksums matched across three
independent computations (the writer op's own receipt, an independent
Python reimplementation, and the GL reader's own receipt) - a genuinely
strong method, EXCEPT that an independent reimplementation which
happens to copy the same bug as the code it's checking will agree with
it perfectly while both are wrong. That's exactly what happened with
the `#`-glyph bug above: checksums matched, and it was still broken
(every wall rendered bright magenta instead of gray). Only actually
LOOKING at a dumped PNG (`ops/dump_rgb_png.c` in both projects) caught
it. Lesson, stated directly by the user and worth repeating here: after
any change to a `compose_rgb_frame.c` or the registries/assets it
reads, run `dump_rgb_png.+x` and actually look at the image, don't just
trust that checksums still match.

## Working conventions worth preserving

- One op = one binary, no shared headers between ops (even within the
  same project) - each op duplicates its own copy of tiny helpers like
  `read_kv_str`/`resolve_root`. Deliberate, not an oversight - matches
  a stated cdda-tpm-std-fast.txt convention.
- Cooperating ops that read the same keypress each tick (e.g.
  `project_browser.c` + `map_edit_input.c` in muchipal-editor,
  `move_player.c` + `choice.c` in mutaclsym) both run UNCONDITIONALLY
  every tick and self-filter by checking shared state (which screen is
  active, etc.) - never a branch in the `.pal` script choosing which op
  to call.
- `system/*.c` files (`prisc+x.c`, `keyboard_input.c`, `renderer.c`,
  `gl_mirror.c`) are copied verbatim across every pal-family project -
  confirmed project-agnostic (root resolution via `PRISC_PROJECT_ROOT`
  env var only, no hardcoded paths).
- The harness's shell resets `pwd` between separate tool calls with no
  warning - any script that writes relative paths must `cd` (or use
  absolute paths) WITHIN the same command block that does the writing.
  A previous mistake this session (documented in
  `mutaclsym/dox/00-HANDOFF.md`) came from missing this.
- Before running anything that touches a REGISTERED real project's live
  state (not a scratch copy), check for a live session first:
  `ps aux | grep -E "prisc\+x|keyboard_input|system/renderer|gl_mirror"`.
