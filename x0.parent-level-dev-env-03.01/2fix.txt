## 2026-07-05: piececraft-wraith box-drawing header showed a corrupted
## replacement character (user-reported, live)

Observed (live, user-reported): piececraft-wraith's map-view header
("╔════...════╗") rendered as `╔══════════════════════════�` — visibly
cut off mid-glyph, ending in the Unicode replacement character.
Pre-existing bug, not introduced this session -- this format string
(`append_project_probe_body()`'s `"| |  %-83.83s |<br .../>"`) predates
today's work entirely.

Root cause, confirmed not guessed: printf's `.83s` precision truncates
by BYTES, not displayed characters. `╔`/`═`/`╗`/`╚`/`╝` are each 3-byte
UTF-8 sequences. Measured the real line: 46 visible characters, 138
UTF-8 bytes. An 83-byte cut lands mid-character (83 is not a multiple
of 3 relative to the string's character boundaries), producing an
invalid, truncated byte sequence -- which terminals render as the
replacement character.

Fix: new `utf8_safe_truncate(src, dst, dst_sz)` -- truncates to at most
`dst_sz - 1` bytes same as before, but backs up to the last complete
UTF-8 character boundary first (checks for continuation-byte pattern
`(byte & 0xC0) == 0x80` and steps back while true). Applied at the one
confirmed call site: the body-line pre-truncates to 84 bytes (83 +
null) via this helper BEFORE the existing `%-83.83s` formatting, so
that formatting no longer has anything left to cut mid-character (the
string is already <= 83 bytes). A too-long UTF-8 line now shows fewer
complete glyphs (a clean, shorter border) instead of a corrupted one --
this does not make long lines "fit," it only removes the corruption
when they don't.

Scope note: this specific `.83s` pattern may exist at other call sites
in this same file (not audited exhaustively) -- only the one confirmed
by the user's live report was fixed. If another garbled-multi-byte
report surfaces elsewhere, the same `utf8_safe_truncate()` helper
already exists and can be applied the same way.

Verified:
- Standalone test (`/tmp/.../utf8_truncate_test.c`) against the REAL
  box-drawing line (byte-for-byte, extracted from the source): output
  cut cleanly at 81 bytes (an exact multiple of 3 -- zero partial
  characters), no corruption, matches expectation.
- Confirmed a plain ASCII line passes through completely unchanged
  (byte-for-byte identical) -- zero regression for the common case.
- `gcc -Wall`: exit 0, 0 errors, same 77 warnings as the prior build
  (the href-in-GL-mode fix build), nothing new.
- Real `+x/wraith-alpha_manager.+x` rebuilt in place.
- NOT yet re-confirmed live (need you to reopen piececraft-wraith and
  check the header renders cleanly now).

## 2026-07-05: End-of-session code review — two more real bugs found and fixed

Found while reviewing today's work before handoff (see
`j5-summary&handoff.txt` for full context), not reported live by the
user — caught by tracing the actual dispatch/parsing code, not
guessing.

**Bug 1 — `href` actions were silently inert in GL mode.**
Observed: `emit_embedded_line_objects()`'s buttons render correctly and
are now nav-indexed (see the nav-numbering fix above), but tracing what
actually happens when one is clicked/activated in GL mode revealed a
real gap. In ASCII mode, `href` navigation is handled entirely inside
`wraith_parser_alpha.c`'s own element tree (`strncpy` into its own
in-process `current_layout` global) — `route_command()` is never
involved. GL mode's click/Enter path is completely different
(`wraith_gl.c`'s `hit_test_semantic_action()` /
`action_for_nav_index()` → `route_command()`), and flattens everything
to a bare `action=` string with no memory of "this came from href."
`route_command()` had no recognized prefix for a raw layout path, so it
silently dropped it — the button looked and nav-indexed correctly but
did nothing when activated in GL mode specifically.
Fix: found `request_layout_change()` already existed in
`wraith-alpha_manager.c`, fully correct, and completely unused (flagged
by the compiler as dead code) — it already writes to
`pieces/display/layout_changed.txt`, which `wraith_parser_alpha.c`
already polls and applies (confirmed by reading its own poll loop:
reads the last non-empty line, sets `current_layout`, re-parses). Added
one fallback branch at the end of `route_command()`: any command
containing `.chtpm` that matched no other prefix now calls
`request_layout_change(cmd)`. This is wiring up dead, correct,
pre-existing infrastructure, not new plumbing.
Verified: `gcc -Wall` diffed line-for-line against the prior build —
every warning identical (just shifted by the new comment's line count)
except `request_layout_change` no longer flagged unused (now called;
replaced by a pre-existing, unrelated `execute_action` unused warning).
Not yet live-tested (needs a real GL-mode click on an href-based
button).

**Bug 2 — `project_probe_body_lines()`'s storage silently truncated
below its own read buffer.**
Observed: its `static char body_lines[16][160]` storage was smaller
than the `char line[256]` buffer it copies from, adding an unnecessary
second, smaller truncation ceiling. Confirmed live against the actual
settings body line (231 chars): got cut at 159, mid-attribute on the
trailing border `<text>` tag. Landed harmlessly this time only by luck
— `emit_embedded_line_objects()`'s malformed-tag guard silently dropped
the cut-off trailing decorative tag rather than misrendering, and the
button itself (earlier in the line) was unaffected. A longer href path
or project name would have cut into the button itself.
Fix: widened `body_lines` to `[16][256]` (matching the read buffer, so
this storage step adds no truncation beyond what `fgets()` already
imposes) and `%.159s` → `%.255s` to match.
Scope note: the 256-byte `fgets()` read limit itself (shared with ASCII
mode's `append_project_probe_body()`) is a deeper, pre-existing ceiling
on body-line length in general — not touched tonight, out of scope, but
worth knowing about for any future project embedding a longer line.
Verified: `gcc -Wall` diffed against the prior build, identical except
the widened buffer — 0 errors, same warning set.

## 2026-07-05: GL-Mode Never Re-Parsed Embedded `<text>`/`<button>` Tags In Project Body Lines

Observed (found while checking why the settings menu button "shows in
ASCII but not GL," per the user's own receipt-check request):
- ASCII mode reads `pieces/display/current_frame.txt`, built by
  `wraith_parser_alpha.c`'s `render_element()` — which genuinely parses
  markup, so raw `<text>`/`<button>` tags embedded in a project's
  `session/wraith_body.txt` (the established trick
  `append_project_probe_body()`'s passthrough relies on) render as real
  interactive elements.
- GL mode reads `current_frame.objects.pdl` instead, built by
  `project_probe_body_lines()` inside `write_semantic_projection_files()`
  — which copied each `wraith_body.txt` line **verbatim** into one
  OBJECT's `label=` field, with **no** tag-parsing at all.
- Confirmed via the actual GL receipt (`current_frame.receipt.pdl`):
  `chtmgl-video-isolate`'s own Play/Pause/Resume/Stop buttons (which use
  the identical embedding trick) showed up with
  `label=<text label="| |  " /><button compact="true" label="Play" onClick="PROJECT_ACTION:VIDEO_PLAY" />...`
  — the literal raw tag text, not a real button. **Pre-existing gap,
  not unique to the new settings project** — it affects every project
  using this embedding convention.

Applied:
- New `extract_attr()` (bounded `memmem`-based `name="value"` scan
  within one tag) and `emit_embedded_line_objects()` in
  `wraith-alpha_manager.c`, right before `append_project_scene_objects()`.
- `write_semantic_projection_files()`'s project-body-line loop now
  branches: a line starting with `<` gets parsed into one real OBJECT
  per tag/text-run (each with its own x/y/w/label/action, x advancing
  by each chunk's rendered width) instead of one OBJECT holding the
  whole raw line as text. Lines that don't start with `<` (the vast
  majority, plain project descriptions etc.) are completely unchanged —
  zero behavior change for the common case.
- **Scope decision, stated not hidden**: these new per-tag objects get
  `nav=0` — not keyboard-Tab-navigable in GL mode specifically. Giving
  them a correct `nav=` would require matching
  `wraith_parser_alpha.c`'s own auto-assigned ordinal for this exact
  line, the same two-independent-numbering-systems fragility already
  documented in `window-geometry-render-plan-j5.md`'s Path B writeup.
  Mouse clicks work correctly (hit-tested by real pixel bounds,
  independent of nav index) — keyboard-Tab parity for embedded body
  buttons in GL mode is a known, separate, deferred gap.

Verified:
- `gcc -Wall` on the full file: exit 0, 0 errors, same 78 pre-existing
  warnings as every prior change this session — nothing new.
- Verified the parser logic standalone first (extracted copy in
  `/tmp/.../embedded_line_test.c`) against both real lines it needs to
  handle: the new settings button line (correctly extracted `href` as
  `action=`) and the pre-existing `chtmgl-video-isolate` Play/Pause
  line (correctly extracted `onClick` as `action=`, correct x-advance
  per chunk for all 5 pieces on that line).
- Real `+x/wraith-alpha_manager.+x` rebuilt in place.

## 2026-07-05: Follow-up — Embedded Body Buttons Had No Nav Number In GL Mode

Observed (user noticed live): the settings menu's "Window Geometry"
button rendered correctly in GL mode after the fix above, but showed no
`[ ] N.` nav number the way every other selectable item does — because
`emit_embedded_line_objects()` deliberately assigned `nav=0` to these
new objects, since `g_max_index`'s formula
(`recompute_nav_bounds()`/`dispatch_menu_index()`) never accounted for
this category of control at all.

Investigated before touching the shared formula (traced, not guessed):
- `count_project_nav_controls()` already existed and already fed
  `extra_project_slots`/`launcher_start` — but only counted nav-bearing
  controls declared in a project's own `session/scene.objects.pdl`
  (confirmed real example: `piececraft-wraith`'s controls are
  `nav=5..22`, contiguous, author-declared directly in that file).
- `session/wraith_body.txt`'s embedded tags were never counted at all.

Applied:
- Split `count_project_nav_controls()` into `count_scene_nav_controls()`
  (the original scene.objects.pdl scan, unchanged logic) +
  `count_embedded_body_nav_slots()` (new: counts labeled chunks across
  `wraith_body.txt` lines starting with `<`, same criterion
  `emit_embedded_line_objects()` uses) — `count_project_nav_controls()`
  now returns their sum, additive to every existing call site
  (`dispatch_menu_index()`, `recompute_nav_bounds()`) with no other
  changes needed there.
- `write_semantic_projection_files()` now computes
  `next_body_nav = 5 + count_scene_nav_controls(project_dir)` before the
  body-line loop, and passes it through to
  `emit_embedded_line_objects()` as an in-out counter — only chunks with
  a real `action` (actual buttons, not plain `<text>` border/padding
  pieces) consume a nav slot, incrementing the shared counter.

Verified (not guessed) before integrating:
- Standalone test (`/tmp/.../nav_count_test.c`) run against the REAL
  `piececraft-wraith/session/scene.objects.pdl` (18 real controls,
  nav=5..22) combined with settings' real `wraith_body.txt`: confirmed
  `next_body_nav` correctly starts at 23 (right after scene's range),
  `launcher_start` correctly lands at 25 — no collision.
- Same test against settings alone (no scene.objects.pdl): confirmed
  `next_body_nav` starts at 5 — matching what ASCII mode already
  independently shows for the same button (`[ ] 5. [Window Geometry]`),
  a nice consistency bonus though not a requirement.
- `gcc -Wall`: exit 0, 0 errors, same 78 baseline warnings, nothing new.
- Real `+x/wraith-alpha_manager.+x` rebuilt in place.
- Not yet re-confirmed live (need you to check the settings menu in GL
  mode again — should now show a real `[ ] 5.`-style nav number on
  "Window Geometry").

the semantic scene is MISSING or EMPTY (`semantic_source_is_allowed()`/

screen, which is exactly this case, so GL rendered the wrong frozen
content indefinitely.

User's direction, explicit: keep `href` navigation (do not force
window-geom to become a registered Window / project-launcher entry

"if they disagree render the fresh one" per wraith-architecture-j25.md):
