1. TOP SUSPECT
========================================================================

**The chtpm-mode overlay render path, combined with the new ray marcher,
has never been run even once.** Everything I verified today (both the
compose_frame.c race fix in item #3 AND the ray marcher's own basic
sanity in item #1) was tested SEPARATELY, at DIFFERENT points in the
session, never together in the same live process. If the user is
playing in chtpm mode (the GL window, via `button.sh chtpm` or
whatever their real launch path is) with `render_mode=1` (3D toggled
on), EVERY tick now runs:
  compose_frame.c's open_memstream fix
    -> writes current_frame.txt + view.txt directly
  compose_rgb_frame.c's chtpm-mode branch
    -> re-derives grid_early
    -> calls render_3d_view() -> NEW raymarch_walls_3d()
    -> writes rgb_frame_3d_overlay.raw + receipt
  shared-ops/chtpm_rgb_render.c
    -> detects the 0x01 marker, blits the overlay, skips ahead

If ANY of these three specifically MISBEHAVE when chained together (as
opposed to how I tested each one alone), the visible symptom could
plausibly be exactly what got reported. A hang or crash in
`compose_rgb_frame.+x` specifically would stall the whole pal script per
item #5 above.

2. HOW TO CHECK, IN ORDER, NEXT SESSION
========================================================================

1. Find out whether the user was testing "run" mode or "chtpm" mode,
   and whether `render_mode` was 0 or 1 (3D on/off) at the time. If
   render_mode was 0, the entire ray-march code path never even
   executes (it's only reached inside `if (!walkable)` territory that
   `raymarch_walls_3d()` covers, called unconditionally from
   render_3d_view() itself - but render_3d_view() is ONLY called at all
   when render_mode==1, both in "run" mode's main() branch and in the
   chtpm-mode overlay block. So render_mode=0 should rule out
   EVERYTHING in items #1/#2 above as a cause). This is the fastest way
   to cut the suspect list roughly in half.

2. Check for a hung or crashed process FIRST, before touching any code:
     ps aux | grep -E "prisc\+x|chtpm_parser_pal|chtpm_rgb_render|compose_rgb_frame|compose_frame"
   A `compose_frame.+x`/`compose_rgb_frame.+x` process still alive and
   consuming CPU (or a defunct/zombie entry) would be direct, immediate
   confirmation of a hang or crash-without-cleanup.

3. Time a direct, read-only invocation against whatever the user's LIVE
   state files actually contain (safe - these ops only read state and
   write display/*, never mutate hero/state.txt):
     cd <live mutaclsym dir>
     time ./ops/+x/compose_frame.+x
     time ./ops/+x/compose_rgb_frame.+x
   If either takes more than a second or two, or never returns, that's
   the bug, confirmed - not a theory anymore. cross-check by then
   running `./ops/+x/dump_rgb_png.+x` and actually looking at the PNG,
   same as every other verification pass this session used.

4. If compose_rgb_frame.+x is the culprit, bisect by temporarily
   commenting out just the `raymarch_walls_3d(...)` call at the bottom
   of `render_3d_view()` (leaving the floor pass and everything else
   intact) and re-testing - this isolates "the ray marcher itself" from
   "the chtpm-mode overlay plumbing it now runs inside of," which are
   two genuinely different suspects bundled under the same top-level
   theory in §1.

5. If it's NOT a hang/crash in either op (both return fast, correct-
   looking output), the bug is likely somewhere I haven't thought of -
   re-read `pieces/keyboard/history.txt` / `pieces/apps/player_app/
   history.txt` / `interact_relay.txt` directly to confirm real
   keystrokes ARE landing in the expected file at all (rules out a
   completely separate input-capture regression, e.g. in gl_mirror.c or
   keyboard_input.c - neither of which I touched this session, but
   worth ruling out explicitly rather than assuming).

5. DECISION MADE AFTER THIS HANDOFF WAS FIRST WRITTEN: DROP CHTPM MODE
========================================================================

Direct instruction, same session: "i wish we could get rid of the chtpm
flag and just run it as normal." Reasoning (mine, not yet independently
confirmed with the user): chtpm mode is what turned a single-file
rendering bug into a whole extra layer of plumbing (the CHTPM VIEW
BRIDGE, the MAP3D_MARKER compositing protocol, the chtpm-mode overlay
branch in compose_rgb_frame.c, the shared chtpm_rgb_render.c daemon) -
none of which regular "run" mode needs at all. Regular "run" mode was
ALSO the thing 3D was originally built and fully verified against
(§4 of `dox/06-3d-view-handoff.txt`, done BEFORE chtpm mode ever got
wired in) - so reverting to run-mode-only isn't a downgrade, it's going
back to the last fully-verified configuration.

**IMPORTANT CAVEAT - do not assume this alone fixes the input
regression above.** The new ray marcher (`raymarch_walls_3d()` and
everything in §0 item #1) runs in BOTH modes - `render_3d_view()` is
called from "run" mode's own main() branch too, not just the chtpm
overlay block. If the actual bug turns out to be in the ray marcher
ITSELF (not the chtpm-specific plumbing around it), dropping chtpm mode
will NOT fix it - it would just remove one of the two places it can
manifest. Test run-mode-only with 3D toggled on BEFORE assuming this
regression is resolved, using the same direct-invocation-and-time
approach in §2.3 above.
