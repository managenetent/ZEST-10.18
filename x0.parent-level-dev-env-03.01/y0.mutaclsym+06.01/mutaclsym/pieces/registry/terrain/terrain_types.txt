# rgb_top matches real 1.TPMOS piececraft-wraith's own
# assets/tiles/*.tile.txt convention (same field name, same meaning -
# solid top-face color for the GL/RGB mirror's software rasterizer, see
# 2.muchi-verse/GRAND-ARCHITECTURE.md's GOVERNING CONSTRAINT section:
# this is portable CPU-computed color data, never a GL primitive call

# unicode/rgb_top_emoji added for the ASCII<->emoji display toggle (see
# GRAND-ARCHITECTURE.md §0a - the real op-ed/fuzz-op toggle mechanism,
# ported the doc-recommended way: one registry field per row, not a
# second hardcoded table that can drift out of sync with this one, the
# exact mistake §0a documents op-ed/render_map.c already made). Every
# row gets a real value (not left to fall back to the ascii glyph) since
# this project's whole glyph set is small enough to cover completely.
# rgb_top_emoji is the GL mirror's flat-color analogue of the emoji
# swap (mutaclsym's GL renderer has no font/glyph rasterization for map
# tiles at all - see ops/compose_rgb_frame.c's own header comment - so
# there is nothing to "draw the emoji" with there; a themed alternate
# top color is the real, doc-consistent parity: real fuzz-op-gl's own
# emoji-tile assets are ALSO just a differently-colored tile variant,
# not a rasterized emoji glyph, confirmed by direct research).
