# 2fix-july-11.txt — Why wrai-text-editor renders from hand-duplicated
# markup instead of real layout+substitution, and whether that's a local
# mistake or a genuine Wraith architecture gap

Date: 2026-07-11 (supersedes the earlier version of this file — that
version's "two competing render paths" framing was itself wrong, per the
user's correction; this version is the researched, reconciled account)

Trigger: user observed wrai-text-editor's embedded window content is
generated by the project's own ops hand-writing pre-substituted markup
(effectively acting like a private, duplicated "gui_state" renderer),
NOT by the real CHTPM parser genuinely parsing `layouts/editor.chtpm`
and substituting `${document_view}` into it — and that this is not how
agy-text-editor works. Asked for extensive research treating
agy-text-editor as the gold-standard reference, checking whether
piececraft-wraith (a working reference) has itself deviated from
agy's principles, and to determine whether this is a Wraith
architecture problem or a local implementation mistake.

---

## Verdict up front

**This is a real, confirmed architecture gap in Wraith's embedded-window
system, not a local mistake in how wrai-text-editor was built.** No
Wraith-hosted project's embedded desktop window currently gets genuine
layout-file-parse-plus-`${var}`-substitution rendering. Every embedded
project that shows real interactive markup (buttons, `<cli_io>`, etc.)
in its desktop window — `settings`, and now `wrai-text-editor` — does so
by hand-duplicating already-substituted markup into `session/wraith_body.txt`
inside its own C code, not by having its `layouts/*.chtpm` file
genuinely parsed. `piececraft-wraith` avoids the duplication problem
only by not attempting real markup/substitution in its embedded body at
all (plain text only, no buttons). Confirmed by exhaustive search: no
shared library anywhere in this codebase lets a project's own process
parse a `.chtpm` file and substitute `${var}`s outside the two
long-lived parser binaries (`chtpm_parser.c`, `wraith_parser_alpha.c`),
each of which tracks exactly one whole-screen `current_layout` — there
is no per-embedded-window instance of this capability.

---

## The gospel: how agy-text-editor actually works

agy-text-editor is architecturally simple and has no Wraith-specific
machinery at all:

- `projects/agy-text-editor/` — no `session/`, no `ops/`, no
  `scene.objects.pdl`. Just `project.pdl`, `manager/`, `layouts/`,
  `pieces/`.
- `layouts/editor.chtpm` is real XML: `<panel>`, `<module>`, `<button>`
  elements, and one `${editor_map}` placeholder (`layouts/editor.chtpm:8`).
- `manager/agy-text-editor_manager.c`'s `build_editor_map()`
  (line 739-793) computes the visible document window (cursor position,
  horizontal/vertical scroll) and formats it **as CHTPM markup**
  (`<text label="║  %-40.40s ║" /><br/>` per line, line 790) into a
  string.
- `update_gui_state()` (line 795) writes that string, plus other
  variables, to `manager/gui_state.txt` under the key `editor_map=`.
- **This is the entire job of agy's manager: compute values, write
  `gui_state.txt`. It never touches `layouts/editor.chtpm` at all.**
- Separately, and entirely outside agy's own code: whichever generic
  parser has `layouts/editor.chtpm` as its `current_layout` (reached via
  `href` or `LOAD_PROJECT:agy-text-editor`) opens that file, walks its
  real `<panel>`/`<button>`/`<text>` tags, and when it hits
  `${editor_map}` inside a `<text label="...">`, substitutes in the
  value it read from `gui_state.txt`. **The parser is the one and only
  thing that ever merges structure (layout file) with values
  (gui_state.txt) into what the user sees.**

This is exactly the CHTPM canonical model
(`^.chtpm-faq/CHTPM_TPM_DOX+1.md`): "1. CHTPM (Theater/View) — Layout
composition, Variable substitution... 6. Parser composes final frame."
Structure lives in the layout file. Values live in gui_state.txt. The
**parser**, and only the parser, merges them. agy's manager never
duplicates the box-drawing/button structure — it only ever emits the
*value* of one substitution variable.

## What wrai-text-editor does today (after this session's fix)

- `layouts/editor.chtpm` exists, has the same shape as agy's (`<panel>`,
  `<module>`, would-be `${document_view}` placeholder) — **but nothing
  in the live system ever opens this file for wrai-text-editor's
  embedded desktop window.** Confirmed by direct code trace of
  `wraith-alpha_manager.c`'s `append_project_probe_body()`
  (the function that composes an embedded window's visible body): it
  does exactly one `fopen("session/wraith_body.txt")` + `fgets()` loop,
  full stop. No `entry_layout` lookup, no `.chtpm` open, no
  `${var}` substitution logic of any kind, anywhere in that function or
  its call chain.
- Because of that, wrai-text-editor's ops (`write_wraith_body()`,
  this session's fix) has to **hand-write the box-drawing frame and
  `<button>` tags directly as C `fprintf()` calls**, with the document
  content already substituted in, straight into `session/wraith_body.txt`.
  This is structurally identical to what agy's manager does for
  `editor_map` **except** it also duplicates the box frame and buttons
  that, in agy's model, belong in the layout file and get added by the
  parser — because there is no parser step available to do that job for
  an embedded window.
- Net effect: `layouts/editor.chtpm` is now dead weight for the embedded
  path — a second, unused copy of the same structure the ops's C code
  also contains. Two sources of truth for one piece of UI, only one of
  which is ever live. This is exactly what the user flagged as wrong,
  and they're right — it does not match agy's model, where there is
  exactly one source of truth (the layout file) and one merge point (the
  parser).

## Does piececraft-wraith do any better? (user's specific question)

**No — it sidesteps the problem rather than solving it, and its own
`layouts/piececraft-wraith.chtpm` explicitly documents why:**

```
<!-- NOTE: this layout is is_map_control=1, so Wraith renders the window body from
     state.txt's game_map= field and this file's own <button>/<text> markup is NOT
     parsed. Clickable nav lives in session/scene.objects.pdl (tag=control objects) -->
```

Read literally: piececraft-wraith's own layout file's `<button>`/`<text>`
tags are **never parsed, even when the layout is reached directly** —
confirmed by inspecting its live `session/wraith_body.txt`, which
contains zero `<`-prefixed lines (pure plain text: literal box-drawing
characters for the header, a raw ASCII grid for the map, plain
"CONTROLS:" documentation text for hotkeys — no `<button>` anywhere).
piececraft-wraith's actual interactivity comes from a third mechanism
entirely: raw single-keystroke hotkeys (`w`/`a`/`s`/`d`, `8`, `9`, etc.)
consumed via `<interact src="session/history.txt" />`, with GL-mode
mouse-click nav coming from `session/scene.objects.pdl`'s
`OBJECT tag=control ... nav=N action=...` rows — a UX paradigm that
never needed button markup or `${var}` substitution in the first place,
so it never hit this gap. It's a working, complete, honest project
(hence a fine architectural reference for the manager/ops/session
pattern) — but it does not exemplify "layout is the real template,
parser does real substitution," because its own layout's markup is
explicitly never parsed even in principle. It deviates from agy's
principles by avoiding the whole problem, not by solving it differently.

`settings` is the closest thing to a hybrid: when reached via `href`
(standalone), its `settings.chtpm` genuinely gets parsed with real
`${settings_menu_markup}` substitution — matching agy exactly for that
surface. But its embedded `session/wraith_body.txt` (for when it's
opened from the desktop launcher instead) is a **second, hand-written,
parallel copy** of equivalent `<button>`/`<cli_io>` markup, maintained
separately in `settings_manager.c`'s C code — the same duplication
problem wrai-text-editor now has, just written by an earlier agent and
not previously flagged.

---

## Root cause, stated plainly

`wraith-alpha_manager.c`'s embedded-window body composer
(`append_project_probe_body()`) was built as a simplified,
parser-bypassing shortcut — plain-text-file passthrough with one crude
`<`-prefix escape hatch for markup — rather than as a genuine
per-window instance of the real CHTPM parse-and-substitute pipeline
`chtpm_parser.c`/`wraith_parser_alpha.c` already implement for
whole-screen layouts. That real pipeline is never invoked for embedded
windows at all. This is a real, confirmed gap relative to the
agy/legacy-TPM model — not a misunderstanding on this session's part
(the earlier version of this document's "two competing paths, pick one"
framing was the actual mistake, now corrected) and not something wired
incorrectly specifically in wrai-text-editor (its current implementation
matches the existing, if flawed, precedent set by `settings`).

---

## Open question for the project owner (not resolved here, no unilateral action taken)

Three ways forward, genuinely different in scope and consequence:

**A. Accept the existing precedent (status quo, what wrai-text-editor
now does).** Every embedded-window project that wants real interactive
markup hand-duplicates its structure into `wraith_body.txt`, matching
`settings`. Cheap, no shared-code changes, but every such project
carries two copies of its own UI structure that can drift out of sync
(the layout file becomes decorative/misleading for anyone who assumes
it's live for the embedded window).

**B. Fix the actual gap: give `wraith-alpha_manager.c` a real
per-window layout-parse-and-substitute step**, so `append_project_probe_body()`
(or a new function alongside it) opens a project's own `layouts/<entry_layout>`
file, does real `${var}` substitution from `manager/gui_state.txt` (same
as agy), and composes that into the embedded window body — bringing
embedded windows fully in line with agy's model, with one source of
truth. This is a real Wraith architecture change affecting the shared
manager, not a per-project fix — needs explicit buy-in given its blast
radius (every existing embedded project, including piececraft-wraith,
web-cam, settings, would be touched or at least need re-verification).

**C. Stop treating wrai-text-editor as an embedded-window project at
all.** Drop `session/`, `scene.objects.pdl`, the desktop-launcher
registration; make it a pure standalone project exactly like
agy-text-editor (only `project.pdl` + `manager/` + `layouts/` +
`pieces/`), reached via `href`/`LOAD_PROJECT:`. This gets a 1:1 match
to the gospel reference with zero duplication, at the cost of it no
longer appearing in the Wraith desktop's launcher row/taskbar — which
was the original ask that started this whole thread ("project not yet
added to wraith projects").

No option was implemented as part of this document — flagging for a
decision before further code changes, since B and C both have real
scope beyond this one project, and A (status quo) is arguably shipping
a known architectural inconsistency without saying so.
